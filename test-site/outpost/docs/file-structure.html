File Structure

A complete map of every file and directory in an Outpost installation — what each one does, what you can edit, and what you must never delete when deploying updates.

Complete Directory Tree

A full Outpost installation looks like this. Everything relevant to the CMS lives inside outpost/. The only file outside that directory is index.php at the web root, which acts as the front controller for Apache and Nginx.

your-site/
  outpost/
    content/              ← everything the user owns
      data/
        cms.db
      uploads/
      themes/
        my-theme/
          index.html
          about.html
          blog.html
          post.html
          contact.html
          404.html
          partials/
            head.html
            nav.html
            footer.html
            [other].html
          assets/
            style.css
            main.js
          theme.json
      backups/
    uploads → content/uploads  (symlink)
    themes → content/themes    (symlink)
    admin/
    cache/
      templates/
    api.php
    member-api.php
    front-router.php
    engine.php
    template-engine.php
    mailer.php
    members.php
    form.php
    sync-api.php
    sitemap.php
    track.php
    docs/
    member-pages/
      login.php
      register.php
      forgot-password.php
      reset-password.php
      verify-email.php
      logout.php
      profile.php
  index.php
Tip — The content/ directory contains everything you own: database, uploads, themes, and backups. Back up this single folder and you have your entire site. The uploads and themes symlinks at the outpost root preserve existing URLs like /outpost/uploads/photo.jpg and /outpost/themes/my-theme/assets/style.css.

Deploy Safety Reference

Before diving into individual files, here is the most important information for anyone deploying updates to a live site:

Path Action on deploy Reason
outpost/content/ PRESERVE — never overwrite Contains all user data: database, uploads, themes, backups
outpost/cache/ Safe to delete entirely — rebuilds automatically Compiled template cache and page HTML cache; stale after a deploy
outpost/admin/ Replace on each deploy if admin source changed Pre-built SPA; must include index.html when updating
All outpost/*.php files Replace on each deploy Core engine files; always deploy the latest version
index.php Replace on each deploy Apache/Nginx front controller
Warning — Never delete or overwrite outpost/content/ when deploying. This directory contains your database, uploads, themes, and backups. Everything else can be replaced safely. The auto-updater already skips content/ automatically.

outpost/admin/

The pre-built Svelte 5 admin single-page application. This directory contains:

  • index.html — the SPA shell; references hashed JS and CSS filenames
  • assets/app-[hash].js — the compiled JavaScript bundle
  • assets/index-[hash].css — the compiled CSS
Warning — Never edit files in admin/ directly. This directory is a build artifact. Any manual changes will be overwritten the next time the admin SPA is built and deployed. To modify the admin UI, edit the Svelte source in cms/src/ and run npm run build.

The hashed filenames in admin/assets/ change on every build. This is intentional — it busts browser caches automatically. When deploying a new admin build, you must copy index.html along with the new asset files, because index.html contains the references to the current hash. If you copy only the JS file and leave the old index.html, browsers will continue loading the old bundle.

outpost/themes/

All themes live here. Each theme is a subdirectory named by its slug. The active theme is tracked in the database — you can switch themes from Admin → Themes without touching any files.

You can have multiple themes installed simultaneously. Only the active theme is used to serve the public site.

Theme template files (.html)

Template files in the theme root map directly to URL paths. The filename (without extension) becomes the URL path component:

File Handles URL Notes
index.html / Required — homepage
about.html /about Any static page slug
blog.html /blog Collection archive page
post.html /post/{slug} Collection single item — slug passed via ?slug=
contact.html /contact Any custom page
404.html No matching route Optional — rendered when no template matches

The URL-to-template mapping is direct: a request for /portfolio will render portfolio.html if it exists. Collection single-item URLs (e.g. /post/my-title) are detected by comparing the URL prefix to registered collection URL patterns and routing to the appropriate template.

partials/

Partials are reusable template fragments included from other templates using {% include 'partial-name' %}. They live in partials/ inside the theme directory.

Three partials are standard across all Outpost themes:

File Contains Included in
partials/head.html <!DOCTYPE html> through the opening <body> tag; all <meta>, CSS, and font links Every page template
partials/nav.html Site navigation markup Every page template
partials/footer.html Footer markup, closing </body></html> Every page template

You can create as many additional partials as you need. Common examples: partials/hero.html, partials/post-card.html, partials/sidebar.html. Reference them in templates with:

{% include 'hero' %}
{% include 'post-card' %}
Note — Partials are compiled and cached independently. Modifying a partial file invalidates its cached version and triggers recompilation on the next request that uses it.

assets/

Static assets for the theme: CSS, JavaScript, fonts, and images used by the theme itself (not uploaded media). These files are served directly by the web server without going through PHP.

Reference assets in templates using a root-relative path or a path relative to the theme's assets directory. A common pattern is to reference them from head.html:

<link rel="stylesheet" href="/outpost/themes/my-theme/assets/style.css">
<script src="/outpost/themes/my-theme/assets/main.js" defer></script>
Tip — You can use any CSS or JavaScript in theme assets — vanilla, bundled, or from a CDN. Outpost imposes no build requirements on theme assets. Just write CSS and link it.

theme.json

A JSON metadata file for the theme. At minimum it contains:

{
  "name": "My Theme",
  "version": "1.0.0",
  "description": "A clean personal blog theme.",
  "author": "Your Name"
}

This file is read by the admin's Themes page to display theme information. See theme.json reference for all available fields.

outpost/content/

The content/ directory holds everything you own. It contains four subdirectories:

  • content/data/ — SQLite database (cms.db) with all your content, settings, users, and configuration
  • content/uploads/ — all uploaded media files, organized by year and month
  • content/themes/ — your theme templates, CSS, and assets
  • content/backups/ — backup zip files created from Admin → Settings → Backups
Warning — Back up the content/ directory regularly. This single folder is your entire site. Losing it means losing all pages, posts, media, themes, settings, and user accounts.

The database is created automatically on first boot. Outpost runs all schema migrations on startup, so updates to the CMS engine that require new columns or tables are applied automatically without any manual SQL.

URL Symlinks

Outpost creates symlinks at the outpost/ root so that existing URLs continue to work:

  • outpost/uploads → outpost/content/uploads
  • outpost/themes → outpost/content/themes

URLs like /outpost/uploads/photo.jpg and /outpost/themes/my-theme/assets/style.css work exactly as before. The symlinks are created automatically during installation or migration.

outpost/cache/

Outpost uses two levels of caching:

cache/templates/

Compiled PHP versions of theme template files. When Outpost renders a template for the first time, it compiles the Liquid-syntax .html file to a plain .php file and saves it here. On subsequent requests, the compiled PHP is loaded directly, skipping the compilation step.

Outpost checks the modification time of the source .html file against the cached .php file. If the source is newer, the cache is automatically invalidated and recompiled. You never need to manually clear this during development.

cache/*.html

Optional full-page HTML cache. When enabled, Outpost saves the complete rendered output of each page as an HTML file here. On the next request for the same URL, the cached HTML is served directly, skipping all PHP execution and database queries entirely.

The page cache is cleared automatically when content is updated through the admin. It can also be cleared manually via Admin → Settings → Clear Cache, or by deleting the files directly.

Tip — It is safe to delete the entire cache/ directory at any time. Outpost will recreate it on the next request. Clearing the cache after a deploy is good practice to ensure visitors see fresh content immediately.

Core PHP Files

These files are the engine of Outpost. They are replaced on every deploy when you update the CMS. Do not modify them — all customisation happens in theme files.

engine.php

The core CMS functions library. Contains all the functions that theme templates compile down to: cms_text(), cms_richtext(), cms_image(), cms_global(), cms_collection_list(), cms_collection_single(), cms_meta_title(), outpost_init(), and many more. This file is included at the top of every compiled template.

template-engine.php

The Liquid-to-PHP compiler. Contains the OutpostTemplate class with its render() and compile() methods. When a request comes in, the router calls OutpostTemplate::render($templateFile, $themeDir), which handles cache checking, compilation, and execution.

front-router.php

The PHP built-in server router. When you run php -S localhost:8080 -t . outpost/front-router.php, this file handles every request. It serves static files directly, routes admin requests to the admin SPA, and calls the template engine for all other requests.

This file is only used with the PHP built-in server. In production with Apache or Nginx, index.php at the web root handles routing instead.

api.php

The admin REST API. Routes requests by the action query parameter to handler functions. Requires session authentication and a CSRF token on all write operations. This is the only backend the admin SPA communicates with.

member-api.php

The member REST API. Handles front-end member operations: login, logout, registration, profile update, and password reset. Does not require admin authentication — it uses its own member session system.

engine.php vs. api.php

engine.php is for rendering the public site. api.php is for the admin panel. They are entirely separate — engine.php does not make API calls, and api.php does not render pages.

mailer.php

Email sending helpers. Supports both PHP's built-in mail() function and SMTP via configured credentials in Admin → Settings. Used by the member password reset flow and contact form submissions.

members.php

Member session helper functions. Provides cms_require_member() and cms_require_paid_member() — called by the router when a page has restricted visibility. Also exposes member data to templates.

form.php

Form submission handler. Processes POST requests from front-end forms, validates fields, optionally emails the submission, and saves it to the database. Referenced by the {% form %} template tag.

sync-api.php

The builder sync API. Used by the Outpost desktop builder application to push and pull theme files over HTTP. Not relevant for typical web server deployments.

sitemap.php

Generates the XML sitemap at /sitemap.xml. Automatically includes all published pages and collection items. Respects visibility settings — members-only and paid content is excluded.

track.php

The analytics tracking endpoint. Receives beacon requests from navigator.sendBeacon() in the theme, records page views to the database, and powers the built-in analytics dashboard.

outpost/member-pages/

Pre-built PHP pages for the member authentication flow. These are served at standard URLs and can be styled by including theme partials.

File URL Purpose
login.php /outpost/member-pages/login.php Member login form
register.php /outpost/member-pages/register.php New member registration
forgot-password.php /outpost/member-pages/forgot-password.php Password reset request form
reset-password.php /outpost/member-pages/reset-password.php Password reset with token
verify-email.php /outpost/member-pages/verify-email.php Email verification landing page
logout.php /outpost/member-pages/logout.php Clears session and redirects
profile.php /outpost/member-pages/profile.php Member profile edit form

These pages include your theme's head.html and footer.html partials automatically, so they inherit your site's styles without any extra configuration.

outpost/docs/

This documentation site. A set of static HTML files served directly by the web server — no PHP required. Accessible at /outpost/docs/.

index.php (web root)

The Apache and Nginx front controller. This file sits at your web root (one level above outpost/) and handles all requests that reach PHP. It performs the same routing logic as front-router.php but is designed for Apache/Nginx rather than the PHP built-in server.

Specifically, index.php:

  • Passes static file requests through to the file system
  • Routes /outpost/ requests to the admin SPA or API
  • Routes all other requests to the template engine
  • Enforces page visibility (members-only, paid) before rendering
Note — index.php and front-router.php do the same job in different environments. front-router.php is only used with php -S. In production with Apache or Nginx, only index.php is used, and your web server must be configured to route all requests to it. See the Deploy guide.

Admin SPA Source (Development Only)

If you are modifying the admin UI, the source code lives outside the deployment in a separate directory structure:

cms/
  src/
    App.svelte           ← root component
    lib/
      components/        ← shared UI components
      pages/             ← admin page components (Dashboard, Posts, etc.)
      stores/            ← Svelte stores
  php/
    admin/               ← build output (deployed to outpost/admin/)
  vite.config.js
  package.json

This source is only needed on a developer machine where you are building custom admin modifications. It is not present on a production server and is not required for normal theme or content work.
