Password Reset

Outpost provides secure self-service password reset flows for both front-end members and admin users. Tokens are cryptographically generated, single-use, and expire after one hour.

Member password reset flow

The front-end member password reset is a self-contained flow handled by the Outpost engine at the /outpost/ routes.

  1. Member visits /outpost/login and clicks Forgot password?, which goes to /outpost/forgot-password.
  2. Member enters their email address and submits the form.
  3. Outpost looks up the email. If a matching account exists, a 64-character hex token is generated via bin2hex(random_bytes(32)), stored in the members table along with a 1-hour expiry timestamp.
  4. A password reset email is sent to the address. The email contains a link to:
    https://yoursite.com/outpost/reset-password?token=TOKEN
  5. Member clicks the link. Outpost validates the token on page load — if the token is invalid or expired, an error is shown immediately and no password form is displayed.
  6. Member enters and confirms a new password (minimum 8 characters).
  7. On successful submission, the token is set to NULL in the database (preventing reuse) and the password is updated using bcrypt at cost 12.
  8. Member is redirected to the login page with a success message.

Admin user password reset flow

The admin password reset flow is handled within the admin SPA. No separate page is required.

  1. Admin user clicks Forgot password? on the admin login screen at /outpost/admin/.
  2. They enter their email address. The SPA sends POST /outpost/api.php?action=auth/forgot.
  3. If the email matches an admin account, a token is generated and stored in users.reset_token with a 1-hour expiry. A reset email is sent with a link to:
    https://yoursite.com/outpost/admin/?reset_token=TOKEN
  4. The admin clicks the link. The SPA detects the ?reset_token= parameter on mount and switches to the password reset view. The token is immediately removed from the browser URL via history.replaceState() so it does not linger in history or browser autocomplete.
  5. The admin enters and confirms a new password. The SPA submits POST /outpost/api.php?action=auth/reset with { token, password }.
  6. On success, the password is updated with bcrypt at cost 12, and the token is set to NULL.
  7. The admin is redirected to the login screen.

Security properties

PropertyDetail
Token format 64-character lowercase hex string — 32 bytes of output from PHP's random_bytes(), which uses the OS CSPRNG (cryptographically secure)
Token expiry 1 hour from generation. Expired tokens are rejected before the password form is shown.
Single use Token is set to NULL immediately after a successful reset. Attempting to reuse a token shows an "invalid or expired" error.
Email enumeration protection Both the member and admin endpoints always return HTTP 200 with a generic success message regardless of whether the email exists. An attacker cannot use the endpoint to discover valid accounts.
Token in URL — admin The admin SPA removes the token from the browser URL via history.replaceState() immediately on mount. The token never persists in browser history or the address bar.
Token in URL — member After a successful reset, the member is redirected to the login page, removing the token URL from the current page's history entry.
Password hashing bcrypt with cost factor 12 via PHP's password_hash(PASSWORD_BCRYPT) and verified with password_verify().
Minimum password length 8 characters — validated on both the client (form) and server (API).
Rate limiting Reset request endpoints are rate-limited to prevent brute-force token discovery and abuse.

SMTP configuration

Password reset emails are only useful if they are actually delivered. Outpost supports two email delivery methods:

PHP mail() — default, not recommended for production

Without any SMTP configuration, Outpost uses PHP's built-in mail() function. This works on many shared hosting environments but is unreliable on VPS servers and cloud hosts, where outbound SMTP on port 25 is often blocked by the provider. Do not rely on it for production without testing first.

SMTP — recommended for all production sites

Configure SMTP via Admin → Settings → SMTP. You can also set the constants directly in config.php:

// config.php — SMTP configuration
define('OUTPOST_SMTP_HOST',      'smtp.mailgun.org');
define('OUTPOST_SMTP_PORT',      587);
define('OUTPOST_SMTP_USER',      'postmaster@mg.yourdomain.com');
define('OUTPOST_SMTP_PASS',      'your-smtp-password');
define('OUTPOST_SMTP_FROM',      'noreply@yourdomain.com');
define('OUTPOST_SMTP_FROM_NAME', 'Your Site Name');
define('OUTPOST_SMTP_TLS',       true);  // enable STARTTLS on port 587
ConstantDescription
OUTPOST_SMTP_HOSTSMTP server hostname
OUTPOST_SMTP_PORTPort — typically 587 (STARTTLS) or 465 (SSL)
OUTPOST_SMTP_USERSMTP account username / login
OUTPOST_SMTP_PASSSMTP account password
OUTPOST_SMTP_FROMThe address in the email From header
OUTPOST_SMTP_FROM_NAMEThe display name in the email From header
OUTPOST_SMTP_TLSEnable STARTTLS — set to true for port 587
OUTPOST_SMTP_SSLEnable SSL — set to true for port 465

After saving SMTP settings in the admin, use the Test SMTP button to send a test message to your admin email address and confirm delivery before relying on it for real users.

Tip — For production sites, use a dedicated transactional email service: Mailgun, Postmark, SendGrid, or AWS SES. These services provide reliable delivery, bounce/complaint tracking, and are never blocked by cloud providers the way direct SMTP often is.
Warning — If SMTP is not configured and mail() does not work on your server, password reset emails will fail silently. Members will not receive the reset email and have no self-service way to recover their account. Admins can manually set a member's password from Admin → Members as a workaround, but configuring SMTP is strongly recommended.

Admin-initiated password reset

Admin users with the admin or super_admin role can reset any member's or admin user's password directly from the admin without going through the email flow:

  • Member password: Admin → Members → click member → Change Password — enter and confirm the new password. No email sent.
  • Admin user password: Admin → Users → click user → Change Password. Only super_admin accounts can change the password of another super_admin.

Token storage in database

Reset tokens are stored in the database alongside the account record:

Account typeTableColumns
Admin user users reset_token, reset_token_expires
Front-end member members reset_token, reset_token_expires

When a token is consumed (successful reset) or expires, reset_token is set to NULL. Only one active token per account exists at a time — requesting a new reset while a previous token is still valid replaces the old token.

Common issues

IssueLikely cause and fix
Reset email never arrives SMTP not configured, or mail() blocked by server. Configure SMTP and test it.
"Token expired or invalid" on a fresh token Server clock is wrong. Ensure the server's system time is correct and synchronized via NTP.
Reset email arrives but link does not work The site URL in the email is incorrect. Check OUTPOST_SITE_URL in config.php — it must match the actual domain where the site is accessible.
Reset email goes to spam Using mail() or a shared SMTP relay with no SPF/DKIM records. Use a dedicated transactional email service and set up SPF and DKIM DNS records.
Member cannot log in after reset Possible copy-paste issue in the new password. Have the member try again, or reset it manually via Admin → Members.
