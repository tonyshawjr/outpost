Members & Access

Outpost has a built-in member system for front-end user accounts — letting you build newsletter communities, paid content platforms, course sites, and subscription-gated experiences without any additional plugins.

What are members?

Members are people who register and sign in on the public-facing site. They are entirely separate from admin users. An admin account does not grant member access, and a member account does not grant access to the admin panel — they are two independent identity systems.

Members register through built-in PHP pages at /outpost/member-pages/. Once registered, they can access protected pages, manage their profile, and — if upgraded to a paid tier — access paid content.

Use cases

  • Newsletter community — readers create free accounts to access member-only posts or archives
  • Paid content platform — gate premium articles, guides, or downloads behind a paid membership
  • Course site — enrolled students log in to access lessons and course materials
  • Member directory — registered members have public or semi-public profiles
  • Subscription content — time-limited paid access with expiry dates

Member roles

Every member has one of two roles:

Role Description
free_member Registered but not paying. Can access members-only content. Assigned automatically at registration.
paid_member Paying customer. Can access paid content in addition to all members-only content. Upgraded manually by an admin or via payment integration.

Member status

Status Description
active Default state. Member can log in and access content appropriate to their role.
suspended Account is blocked. Member cannot log in. Use this to disable access without deleting the account.

Member fields

The members database table stores the following fields:

Field Description
username Unique login handle. Chosen at registration, cannot be duplicated.
email Email address. Used for password reset emails. Must be unique.
display_name Public display name. Defaults to username if not set. Editable on the profile page.
avatar Profile image. Stored as a path to an uploaded file.
password bcrypt-hashed password. Never stored in plain text.
member_since Timestamp of account creation. Set automatically at registration.
member_expires Optional expiry date for time-limited paid memberships. Null means no expiry.
member_status active or suspended. Defaults to active.
email_verified Datetime of when the email was verified. NULL if unverified. Auto-set when verification is disabled.
verify_token 64-character hex token for email verification. Cleared after successful verification.
verify_token_expires Datetime when the verification token expires (24 hours after generation).
reset_token 64-character hex token for password resets. Cleared after use.
reset_token_expires Datetime when the reset token expires (1 hour after generation).

Registration flow

New members register at /outpost/member-pages/register.php. The form collects username, email, and password. On success:

  1. Account is created with role free_member and status active
  2. If email verification is disabled (default): member is automatically logged in and redirected
  3. If email verification is enabled: a verification email is sent and the member sees a "check your email" message instead of being logged in

Duplicate usernames and emails are rejected with a clear error message before the record is written.

Email verification

Optional email verification can be enabled in Admin → Settings → Members by toggling Require Email Verification. When enabled:

  1. New members receive a verification email with a 24-hour token link after registration
  2. They are not automatically logged in — they must click the link first
  3. Clicking the link opens /outpost/member-pages/verify-email.php?token=... which marks the email as verified
  4. After verification, the member can sign in normally

If an unverified member tries to sign in, the login form shows an error message and a Resend verification email button.

Grandfathering — Members who registered before email verification was enabled are not affected. They can continue to sign in normally. Only members with a pending verification token (created after the setting was turned on) are blocked from login.

Admins can manually verify a member from Admin → Members by clicking the checkmark icon that appears on hover next to unverified accounts. The Members list also shows an "Unverified" counter in the stats bar when unverified members exist.

SMTP required. Verification emails require a working SMTP configuration in Admin → Settings → Email. Without it, the token is stored in the database but no email is delivered.

Managing members in the admin

Go to Admin → Members to view all registered members. From this screen you can:

  • See each member's username, email, role, status, and signup date
  • Toggle a member between free_member and paid_member
  • Toggle a member between active and suspended
  • Set or clear a member_expires date for time-limited access
  • Delete a member permanently
Note — Only admin users with the admin or super_admin role can access the Members screen. Developers and editors do not have member management access.

Member sessions

Outpost runs two completely independent session systems:

Session Cookie name Duration Who uses it
Admin session outpost_session 24 hours Admin panel users
Member session outpost_member 30 days Front-end members

Admin login does not grant member access. Member login does not grant admin access. The two systems share nothing.
