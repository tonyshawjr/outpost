Admin Roles

Admin panel users are assigned one of four roles that control what they can see and do. Roles are completely separate from front-end member roles — an admin user is not automatically a member and vice versa.

Role overview

Each admin user account has exactly one role. The role determines which sections of the admin are visible and which API endpoints the user can call.

RoleDescription
super_admin Unrestricted access to everything. The only role that can manage other super_admin accounts, change site-level settings, and perform destructive operations like database resets. Typically reserved for the site owner or lead developer.
admin Full access to all content, code editor, user management, and member management. Cannot change site settings or perform super-level operations. Can create users up to and including the admin role.
developer Full content management access plus the code editor and template files. Cannot manage other users, members, or change site settings. Suited for developers who need to modify templates and styles but should not manage accounts.
editor Content management only — pages, collections and items, media, globals, navigation, folders, and form submissions. No access to the code editor, analytics, channels, form builder, settings, or user management. Editors can optionally be restricted to specific collections and cannot edit locked pages. Suited for content writers and marketing teams.

Creating and assigning roles

  1. Go to Admin → Users → New User (requires admin or super_admin).
  2. Fill in the name, email, and initial password.
  3. Select the role from the dropdown.
  4. Save. The new user can log in immediately.

To change an existing user's role, go to Admin → Users → [User] → Edit and change the Role field.

Role assignment restrictions

You cannot assign a role higher than your own. The table below shows what roles each admin role can create:

The first super_admin

The first account created during Outpost setup is always assigned the super_admin role automatically. There is no way to self-assign super_admin through the admin UI — it can only be granted by an existing super_admin.

Note — If you ever lose access to the only super_admin account and cannot log in, you can recover access by updating the account directly in the SQLite database. Open data/cms.db with any SQLite tool and run: UPDATE users SET role = 'super_admin', password = [new bcrypt hash] WHERE email = 'your@email.com'.

Developer role security note

Warning — The developer role includes full access to the code editor, which allows editing PHP template files on the server. A user with this role can execute arbitrary PHP code. Only grant this role to people you trust with equivalent access to the server filesystem.

Admin roles vs. member roles

Admin roles and front-end member roles are completely separate systems:

Admin rolesMember roles
Who it applies toAdmin panel usersFront-end site visitors who have registered
Rolessuper_admin, admin, developer, editorfree_member, paid_member
Login URL/outpost/admin//outpost/login
Session namespaceoutpost_sessionoutpost_member

An admin user logging into the admin panel does not automatically have a front-end member session. Similarly, a front-end member cannot access the admin panel regardless of their member role.

Session security

PropertyDetail
Session duration 24 hours from login. Sessions are not extended on activity — the admin must log in again after expiry.
Server-side sessions Sessions are stored server-side. Deleting or suspending an admin user account invalidates their session on the next request.
CSRF protection All mutating API requests (POST, PUT, DELETE) require a valid CSRF token in the X-CSRF-Token header. The token is issued at login and retrieved via GET auth/me.
Session isolation Admin sessions use the outpost_session namespace, separate from front-end member sessions (outpost_member).
Password hashing bcrypt at cost factor 12 via password_hash(). Passwords are never stored in plain text.

Recommended role setups by use case

ScenarioRecommended setup
Solo freelancer or developer super_admin for yourself. No other users needed.
Agency delivering a site to a client Keep yourself as super_admin. Give the client an editor account so they can manage content without accessing the code editor or settings.
Agency + client needs to update settings Give the client's technical lead an admin account. Give content writers editor accounts.
Development team with multiple devs Lead developer as admin or super_admin. Other developers as developer (code access, no settings). Content team as editor.
Large content team One admin account to manage users and settings. Multiple editor accounts for writers. One developer account for the template developer.
Client self-managing content, no dev access Give the client an editor account. Template changes only through you with super_admin.

Activity log

All admin actions are logged with the acting user's ID, the action type, the affected resource, and a timestamp. The activity log is viewable from Admin → Activity. Logged actions include:

  • Content changes — page publishes, collection item saves, media uploads and deletions
  • Settings changes — any change to site settings or SMTP configuration
  • User management — user creation, role changes, password resets, deletions
  • Template actions — file creation, editing, deletion via code editor
  • Login and logout events for all admin users

The activity log is read-only. Entries cannot be deleted from the admin — they persist indefinitely in the database.

Tip — The activity log is particularly useful when you have multiple editors working on a site and need to track who changed what. It is accessible to admin and super_admin roles.

Collection-scoped editors

By default, editors can access all collections. An admin or super_admin can restrict an editor to specific collections from the editor's user profile page.

How it works

  1. Go to Admin → Users → [Editor User].
  2. Under Collection Access, check the collections the editor should have access to.
  3. Save. If no boxes are checked, the editor has unrestricted access to all collections (backward compatible).

When an editor has collection grants:

  • Only granted collections appear in the sidebar
  • API calls to list, create, update, or delete items in non-granted collections return 403
  • Bulk operations (status change, delete) are blocked for items in non-granted collections
Note — Collection grants only apply to the editor role. All other roles have unrestricted collection access. Grants are stored in the user_collection_grants table and managed via the users/grants API endpoint.

Per-page locks

Admins can lock individual pages to prevent non-admin roles from editing them. This is useful for protecting critical pages like the homepage or landing pages from accidental changes by editors.

Locking a page

  1. Open the page in Admin → Pages → [Page].
  2. In the sidebar Page tab, check Lock page.
  3. Save. The page is now locked.

What locked pages do

  • super_admin and admin roles can always edit locked pages
  • developer and editor roles see a "This page is locked" banner and cannot edit the title, fields, or save changes
  • The locked field is only settable by admin or above — it is silently stripped from update requests by other roles
  • Locked pages show a lock icon in the pages list
  • Deleting a locked page also requires admin or above
