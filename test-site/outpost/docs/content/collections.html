Collections

Custom content types for structured, repeating content — blog posts, portfolio projects, team members, products, and anything else that follows a consistent schema.

What is a collection?

A collection is a custom content type made up of multiple items that share the same structure (schema). Where a page is a single unique piece of content, a collection item is one entry in a list of many.

Common examples:

Collection nameSlugURL pattern
Blog Postspost/post/{slug}
Portfolio Projectsproject/portfolio/{slug}
Team Membersmember/team/{slug}
Productsproduct/product/{slug}
Case Studiescase-study/case-studies/{slug}
Eventsevent/events/{slug}

Creating a collection

Go to Admin → Collections → New Collection. Fill in the following required fields:

FieldDescription
NameDisplay name used throughout the admin (e.g., "Blog Posts").
SlugURL-safe identifier used in template tags and API calls (e.g., post). Immutable after creation.
Singular NameUsed for admin UI buttons and labels (e.g., "Post" for "New Post").
URL PatternDefines how item URLs are constructed and how single-item routes are matched (e.g., /post/{slug}).
Warning — The slug cannot be changed after creation because it is referenced by template tags (collection.post), stored data, folder associations, and API routes. Choose it carefully.

URL patterns

The URL pattern controls two things: the public URL of each item, and which template file handles the single-item view.

/post/{slug}          → /post/hello-world
/articles/{slug}      → /articles/my-article
/portfolio/{slug}     → /portfolio/project-name
/{slug}               → /hello-world  (root level)
/news/{slug}          → /news/breaking-story

Only the {slug} placeholder is supported. The slug is derived from the item's title when created and can be edited in the item editor.

Template file naming from URL pattern

The CMS derives the single-item template filename from the first path segment of the URL pattern:

URL patternTemplate file
/post/{slug}post.html
/articles/{slug}articles.html
/portfolio/{slug}portfolio.html (not project.html)
/{slug}post.html (fallback)
Note — The template file name comes from the URL pattern, not the collection slug. A collection with slug project and pattern /portfolio/{slug} needs a portfolio.html template, not project.html.

Schema — field definitions

The schema defines the fields that editors fill out when creating or editing items. Open a collection in admin and go to the Schema tab to add and reorder fields.

Available field types

TypeUse caseTemplate filter
textSingle-line text — title, headline, name, label(none)
textareaMulti-line plain text — address, short bio, notes| textarea
richtextWYSIWYG HTML body content (TipTap editor)| raw
imageSingle image via media picker| image
linkURL input| link
selectDropdown with predefined choices(none)
toggleOn/off boolean(conditional)
colorColor picker — outputs hex value(none)
numberNumeric input(none)
dateDate picker(none)
repeaterOrdered list of rows with typed sub-fields{% for row in item.field %}
flexible contentOrdered blocks with named layouts, each with its own sub-fields{% for block in flexible.field %}
relationshipReferences to items from another collection{% for item in relationship.field %}

Per-field options

Every field in the schema has these configuration options:

  • Label — display label shown in the admin editor
  • Machine name — the key used in template tags and the data JSON (e.g., featured_image)
  • Placeholder — hint text shown in empty fields
  • Description — helper text shown below the field in admin
  • Default value — pre-filled value for new items
  • Required — whether the field must be filled before saving
  • Conditional logic — optionally show or hide this field based on another field's value. See Conditional Logic.

For select fields, you define the list of choices in the field settings. Each choice has a label and a value.

For relationship fields, you configure the target collection, whether multiple or single items can be selected, and an optional max-items cap. See Relationship Loops.

For flexible content fields, you define named layouts, each with its own set of typed sub-fields. See Flexible Content Loops.

Item statuses

StatusBehavior
draftNot publicly visible. Not returned in {% for %} loops on the front end.
publishedPublicly visible and returned in loops. Requires a published_at timestamp (set automatically when first published).
scheduledSet a future publish date. The item auto-publishes when the scheduled time arrives. Not visible until then.
pending_reviewSubmitted for review by an editor on a collection with Require Review enabled. Not publicly visible. Admins approve or reject pending items.

See Scheduling for details on the scheduled publish workflow.

See Editorial Workflow for review & approval, bulk scheduling, and the editorial calendar.

Require review

Enable Require Review on a collection to prevent editors from publishing directly. When enabled, editors who click "Publish" will see their item set to pending_review instead — an admin must then approve it before it goes live. Admins and super_admins can always publish directly, even on review-required collections.

Toggle this setting in Admin → Collections → Edit Collection → Require Review. Only admins can change this setting.

Item data structure

All custom field values are stored as a JSON object in the collection_items.data column. These standard fields are always present on every item:

FieldDescription
titleItem title — also shown in the admin list view
slugURL slug — unique within the collection (e.g., hello-world)
bodyRichtext body — output from the block editor
excerptShort plain-text description — shown in list views and used for meta description
featured_imageImage URL for the item's thumbnail/hero image
custom fieldsAny fields defined in the schema, stored alongside the standard fields

Additionally, the collection_items table row has:

  • status — draft, published, scheduled, or pending_review
  • published_at — timestamp of when the item was or will be published
  • url — the built public URL (derived from URL pattern + slug)

The collection item editor

Open any item in Admin → Collections to edit it. The editor has three areas:

Block editor (main content)

The primary body editor uses a block-based format. Available block types:

  • Text block — rich text via TipTap (bold, italic, links, headings, lists, etc.)
  • Image block — single image with optional caption
  • Markdown block — raw Markdown converted to HTML on render
  • HTML block — raw HTML for custom embeds and components
  • Divider block — horizontal rule separator

Blocks can be reordered by drag-and-drop (SortableJS).

Metadata section

Below the block editor is the metadata section, which shows all custom fields defined in the collection schema. Fill these in per item.

Top bar

The toolbar includes a History button that opens a revision drawer — browse and restore previous versions of the item. See Revision History.

Right sidebar

The sidebar contains:

  • Status — toggle between draft, published, scheduled, and pending review
  • Slug — the URL segment for this item (auto-generated from title, editable)
  • Featured image — image picker for the item's thumbnail
  • Excerpt — short description for list views
  • Folders — assign folder labels (if any folders are associated with this collection)
  • SEO — per-item meta title and meta description

Accessing collections in templates

Use {% for %} loops to output collection items in list templates:

<!-- Basic loop -->
{% for post in collection.post %}
  <article>
    <a href="{{ post.url }}">{{ post.title }}</a>
    <p>{{ post.excerpt }}</p>
  </article>
{% endfor %}

<!-- With featured image -->
{% for post in collection.post %}
  <div class="card">
    <img src="{{ post.featured_image | image }}" alt="{{ post.title }}">
    <h2><a href="{{ post.url }}">{{ post.title }}</a></h2>
    <p>{{ post.excerpt }}</p>
  </div>
{% endfor %}

<!-- With custom schema fields -->
{% for member in collection.member %}
  <div class="team-card">
    <img src="{{ member.photo | image }}" alt="{{ member.name }}">
    <h3>{{ member.name }}</h3>
    <p>{{ member.role }}</p>
  </div>
{% endfor %}

Single-item template

Use {% single %} in the single-item template to fetch the item matching the current URL slug:

<!-- post.html -->
{% single post from collection.post %}
  <article>
    <h1>{{ post.title }}</h1>
    <img src="{{ post.featured_image | image }}" alt="{{ post.title }}">
    <div class="body">{{ post.body | raw }}</div>
  </article>
{% else %}
  <p>Post not found.</p>
{% endsingle %}

See Collection Loops for pagination, ordering, limiting, and folder filtering options. See Single Items for the full {% single %} reference.

Practical examples

Blog

Collection: Blog Posts
Slug: post
Singular: Post
URL Pattern: /post/{slug}
Schema: title, excerpt, featured_image, body (richtext), author (text)
Template files: blog.html (list), post.html (single)

Portfolio

Collection: Projects
Slug: project
Singular: Project
URL Pattern: /portfolio/{slug}
Schema: title, excerpt, featured_image, body (richtext), client (text),
        year (number), tags (text), live_url (link)
Template files: portfolio.html (list), portfolio.html also handles /portfolio/{slug}

Team members (no single-item page)

Collection: Team
Slug: member
Singular: Member
URL Pattern: /team/{slug}
Schema: name (text), role (text), photo (image), bio (textarea), linkedin (link)
Template: displayed only via loop on team.html — no single-item template needed
Tip — Not every collection needs a single-item template. If visitors never need to visit an individual item's page (e.g., team members listed on a single page), you can skip creating the single-item template entirely.
