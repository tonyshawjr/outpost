Caching

Outpost has two independent caching layers — a template compile cache that turns templates into PHP, and an HTML page cache that stores fully rendered pages on disk.

Overview

Both caches live in the cache/ directory (configurable via OUTPOST_CACHE_DIR) and are file-based. There is no Redis, Memcached, or APCu dependency — just the filesystem.

Cache LayerWhat It StoresLocationInvalidation
Template cache Compiled PHP from .html templates cache/templates/ Auto — mtime comparison on source file
HTML page cache Fully rendered HTML output for each URL cache/ (root, .html files) TTL expiry, manual clear, or admin action

Template Cache (Compile Cache)

Every theme template (.html file) is compiled to PHP the first time it is rendered. The compiled PHP is stored in cache/templates/ and reused on all subsequent requests until the source file changes.

How It Works

  1. When a request arrives for a page, OutpostTemplate::render() is called with the template path.
  2. The method calculates the compiled path: cache/templates/{md5(realpath($templateFile))}.php
  3. If the compiled file doesn't exist, or if the source file's mtime is newer than the compiled file's mtime, the template is recompiled.
  4. The compiled PHP is written to disk with LOCK_EX to prevent race conditions on concurrent first requests.
  5. The compiled file is included with PHP's include.

Cache Key

The cache key for a template is the MD5 hash of its real (resolved) filesystem path:

$hash = md5(realpath($templateFile) ?: $templateFile);
$compiledPath = OUTPOST_TEMPLATE_CACHE_DIR . $hash . '.php';

This means moving or renaming a template file produces a new cache entry. Old orphaned entries are harmless but can be cleared manually.

Auto-Invalidation

The template cache invalidates itself automatically when you save a template file. The mtime check runs on every request — if the source is newer, it recompiles. No manual clearing is needed during normal theme development.

// Recompile if source is newer than cache
if (!file_exists($compiled) || filemtime($templateFile) > filemtime($compiled)) {
    $source = file_get_contents($templateFile);
    $php    = OutpostTemplate::compile($source);
    file_put_contents($compiled, $php, LOCK_EX);
}

Partials Are Cached Separately

Each partial (partials/*.html) has its own cache entry and its own mtime check. When you edit a partial, only that partial's cache entry is invalidated. Template files that include the partial still use their own (unmodified) cache entry — but since partials are included via a PHP function call at runtime, the updated partial content is always rendered.

Disabling the Template Cache

During theme development, you can disable the template cache to force recompilation on every request:

// In outpost/config.php
define('OUTPOST_TEMPLATE_CACHE', false);
When to disable. Disable template caching during active theme development to avoid the mtime delay — file system timestamps can have 1-second resolution on some systems, which can cause the old compiled version to be served if you edit and reload very quickly. Re-enable it in production.

Clearing the Template Cache Manually

# Delete all compiled template files
rm -f outpost/cache/templates/*.php

Or through PHP:

OutpostTemplate::clearCache();

Or through the admin: Settings → Clear Cache.

HTML Page Cache (Output Cache)

The HTML page cache stores the complete rendered HTML output of each public page. On a cache hit, the raw HTML file is streamed directly to the browser — no PHP execution, no database queries, no template compilation. This makes cached pages extremely fast.

How It Works

  1. On request, outpost_init() checks whether a cached file exists for the current URL.
  2. If a cache file exists and is within the TTL window (OUTPOST_CACHE_TTL seconds, default 3600), it is served immediately with readfile() and PHP exits.
  3. If the cache is stale or missing, normal rendering proceeds: field resolution, template execution, database queries.
  4. The rendered output is buffered using ob_start('outpost_cache_output'). The buffer callback writes the output to the cache file, injects the GA4 snippet (into the cached copy), and also injects the admin bar (not cached — admin-session-only).

Cache Key

The cache key is the MD5 of the full URL path including sorted query string. This means paginated pages, filtered pages, and search results each get their own cache entry:

function outpost_cache_path(string $page_path): string {
    $query = $_GET;
    if (!empty($query)) {
        ksort($query);
        $page_path .= '?' . http_build_query($query);
    }
    return OUTPOST_CACHE_DIR . md5($page_path) . '.html';
}

The query string is sorted before hashing so /blog?page=2&category=js and /blog?category=js&page=2 produce the same cache file.

When Pages Are NOT Cached

Several conditions bypass the cache entirely — the page is always rendered fresh:

ConditionReason
Admin is logged in Admins see the admin toolbar, which contains session-specific data. The toolbar is injected into the live response and never written to cache.
?nocache=1 in URL Forces a fresh render. The new output is written to cache, refreshing it.
OUTPOST_CACHE_ENABLED is false Cache globally disabled (set in config.php).
Cache file doesn't exist First visit after deploy or cache clear — renders fresh and writes cache.
Cache file is older than OUTPOST_CACHE_TTL TTL expired — renders fresh and writes a new cache file.
Members are not cached out. Member session presence does not automatically bypass the HTML page cache. If you have member-gated content on a page, that page should not be cached — or the member-specific content will leak to unauthenticated visitors. Use the visibility setting to gate full pages, which are served by a separate code path before caching is checked.

GA4 Injection and Caching

If a GA4 tracking ID is set in Global Fields, the GA4 <script> snippet is injected into the output buffer before the page is written to cache. This means the cached HTML includes GA4 — it's part of the cached content, not added dynamically on each request. Changing the GA4 ID requires clearing the cache for all pages to update the embedded snippet.

Admin Bar and Caching

The admin bar (the floating "Edit in Outpost" button) is injected into the live response by the outpost_cache_output buffer callback, but only if an admin session is active. It is never written to the cache file. This is correct and intentional — visitors never see the admin bar, and the cached HTML is clean.

Clearing the Cache

Method 1 — Admin Panel

Admin → Settings → Clear Cache. Clears both the template cache and the HTML page cache in one action.

Method 2 — Admin Bar

When logged into Outpost and viewing any front-end page, the admin bar shows a refresh icon. Clicking it calls the API's cache/clear endpoint and reloads the page.

Method 3 — URL Parameter

Append ?nocache=1 to any URL to bypass the cache for that single request. The page renders fresh and a new cache entry is written. Useful for previewing content changes without affecting the cached versions for other visitors.

https://mysite.com/blog?nocache=1
https://mysite.com/post/my-article?nocache=1

Method 4 — Shell

# Clear all HTML page cache files
rm -f outpost/cache/*.html

# Clear compiled template files
rm -f outpost/cache/templates/*.php

# Clear everything
rm -f outpost/cache/*.html outpost/cache/templates/*.php

Method 5 — PHP

// Clear all page cache
outpost_clear_cache();

// Clear cache for a specific page only
outpost_clear_cache('/blog');
outpost_clear_cache('/post/hello-world');

// Clear template cache
OutpostTemplate::clearCache();

Configuring Cache TTL

The default TTL is 3600 seconds (1 hour). Adjust it in config.php:

// Cache pages for 30 minutes
define('OUTPOST_CACHE_TTL', 1800);

// Cache pages for 24 hours (high-traffic, infrequently updated sites)
define('OUTPOST_CACHE_TTL', 86400);

// 10 minutes (frequently updated content like news sites)
define('OUTPOST_CACHE_TTL', 600);

Disabling the HTML Cache Entirely

// Disable HTML page caching (template cache still active)
define('OUTPOST_CACHE_ENABLED', false);

With page caching disabled, every request goes through the full PHP rendering pipeline. Suitable for development environments or sites with highly dynamic content that can't tolerate any staleness.

Stale Content Warning

Published content is not immediately visible. After publishing or updating content in the Outpost admin, visitors will continue seeing the cached (old) version until the TTL expires or you manually clear the cache. For time-sensitive changes, always clear the cache after publishing — use the admin bar refresh button or Admin → Settings → Clear Cache.

Automatic scheduled publishing (status = 'scheduled') also triggers a full cache clear when items are auto-published, because outpost_auto_publish_scheduled() calls outpost_clear_cache() after updating items.

Cache Directory Structure

outpost/cache/
├── templates/                    ← compiled template PHP files
│   ├── a3f8b2c1d4e5f6a7.php      ← index.html compiled
│   ├── b2c3d4e5f6a7b8c9.php      ← blog.html compiled
│   └── c3d4e5f6a7b8c9d0.php      ← partials/nav.html compiled
├── a1b2c3d4e5f6a7b8.html         ← cached / (homepage)
├── b2c3d4e5f6a7b8c9.html         ← cached /blog
├── c3d4e5f6a7b8c9d0.html         ← cached /blog?page=2
└── d4e5f6a7b8c9d0e1.html         ← cached /post/my-first-post

The cache/ directory and both subdirectories must be writable by the web server process (www-data, apache, or whatever user PHP runs as).

# Set correct permissions
chmod 755 outpost/cache outpost/cache/templates
chown www-data:www-data outpost/cache outpost/cache/templates

Performance optimization for high-traffic sites

Outpost uses SQLite, which is fast for read-heavy workloads but has limitations under high concurrency. Here are strategies to maximize performance.

1. Enable the HTML page cache

This is the single biggest performance win. With the page cache enabled, subsequent requests for the same URL are served from a static HTML file — no PHP, no SQLite, no template compilation. Set OUTPOST_CACHE_ENABLED to true in config.php (enabled by default).

With the page cache, a high-traffic site can handle thousands of requests per second because most requests never touch PHP at all — the web server serves the cached HTML file directly.

2. SQLite WAL mode

Outpost enables SQLite's Write-Ahead Logging (WAL) mode automatically. WAL allows concurrent reads while a write is in progress, which is critical for sites with both visitors (reads) and editors (writes) active simultaneously.

You can verify WAL mode is active by checking for a cms.db-wal file alongside your database. If it exists, WAL is working.

3. Minimize API calls from the front end

The Public Content API is designed for headless use cases. If you are building a traditional server-rendered site, use template tags instead of client-side API calls — template tags resolve data server-side in a single PHP process, avoiding HTTP overhead.

If you must use the API, apply these optimizations:

  • Use limit and offset to paginate — never fetch all items at once
  • Use fields parameter to request only the fields you need
  • Cache API responses on the client side or in a CDN

4. Use a reverse proxy or CDN

For truly high-traffic sites, place a reverse proxy (Nginx, Varnish) or CDN (Cloudflare, Fastly) in front of Outpost. With Outpost's page cache generating static HTML files, a CDN can serve them from edge locations worldwide with sub-10ms response times.

Configure cache headers or CDN rules to cache public pages for a short TTL (e.g. 5 minutes) and bypass the cache for:

  • URLs starting with /outpost/ (admin, API)
  • POST requests
  • Requests with the outpost_session or outpost_member cookies

5. Template cache warm-up

The first request to a page after a code change triggers a template recompilation, which is slower than subsequent cached requests. After deploying a new version, visit your key pages once to warm the template cache before traffic hits.

6. SQLite tuning

For advanced use cases, you can tune SQLite's performance in a custom bootstrap script:

// These are already set by Outpost's default configuration:
PRAGMA journal_mode = WAL;       // concurrent reads during writes
PRAGMA synchronous = NORMAL;     // good durability with better speed
PRAGMA cache_size = -2000;       // 2MB page cache in memory
PRAGMA busy_timeout = 5000;      // wait up to 5s for locks
Tip — SQLite handles millions of rows and hundreds of concurrent readers without issue. Most Outpost sites will never hit SQLite's limits. If you are running a site with 100K+ pages or 1000+ concurrent editors, consider offloading writes to a queue or switching to PostgreSQL with a custom adapter.
