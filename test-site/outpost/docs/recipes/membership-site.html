Membership Site

Build a site with free and paid content tiers. Outpost handles registration, login, session management, and per-page or per-item access control — no third-party services needed.

Members Access Control Visibility

1. Enable member registration

Go to Admin → Settings → Members and enable:

  • Allow registration — lets visitors create a free account at /outpost/register
  • Require email verification (optional) — sends a confirmation email before activating the account

Member-facing URLs that Outpost provides automatically:

URLPurpose
/outpost/loginLogin form
/outpost/registerRegistration form
/outpost/profileMember profile / account settings
/outpost/logoutClears session and redirects to homepage
/outpost/password-resetPassword reset request form
Note These are Outpost's own admin-area URLs. You can link to them from your templates. They use the admin CSS, not your site's CSS. If you want custom-styled login/register pages, create template pages at your site root that post to the Member API.

3. Page-level visibility

For standalone pages (not collection items), you can set visibility in the page editor under the SEO panel. Three options are available:

VisibilityWho can see it
publicEveryone (default)
membersAny logged-in member (free or paid)
paidPaid members only

When a visitor hits a page they are not authorised to view, Outpost redirects them to /outpost/login automatically. No template changes needed for page-level protection.

Tip Page-level visibility is the simplest form of gating — flip a dropdown in the editor and the page is protected. For more fine-grained control (e.g. showing a teaser to free members and the full article to paid members), use the template-level approach below.

4. Template-level content gating

Use {% if paid_member %} and {% if member %} conditionals to show different content to different tiers within the same template. This is the standard pattern for a "freemium" newsletter or magazine.

{% include 'head' %}
{% include 'nav' %}

<main>
  {% single article from collection.article %}

    <article class="article-single">
      <header>
        <h1>{{ article.title }}</h1>
        <p class="article-meta">{{ article.published_at }}</p>
      </header>

      {# Check the custom tier field to decide what to render #}
      {% if article.tier == "free" %}

        {# Free article — everyone can read the full content #}
        <div class="article-body">
          {{ article.body | raw }}
        </div>

      {% else %}

        {# Paid article — show full body to paid members, teaser + CTA to others #}
        {% if paid_member %}

          <div class="article-body premium">
            {{ article.body | raw }}
          </div>

        {% else %}

          {# Show excerpt / teaser to free and logged-out visitors #}
          <div class="article-teaser">
            {% if article.excerpt %}
              <p>{{ article.excerpt }}</p>
            {% endif %}
          </div>

          <div class="upgrade-cta">
            <p>This article is for paid members.</p>
            {% if member %}
              {# Logged in as free member — show upgrade prompt #}
              <a href="/upgrade" class="btn-upgrade">Upgrade to Paid</a>
            {% else %}
              {# Not logged in — show login and register options #}
              <a href="/outpost/register" class="btn-primary">Create a free account</a>
              <a href="/outpost/login" class="btn-secondary">Sign in</a>
            {% endif %}
          </div>

        {% endif %}
      {% endif %}
    </article>

  {% else %}
    <p>Article not found.</p>
  {% endsingle %}
</main>

{% include 'footer' %}

5. Protecting a whole page

To require a paid membership to view an entire page — for example a members-only dashboard — set the page's visibility in the editor to Paid. Outpost enforces this at the router level; the page template is never rendered for unauthorised visitors.

  1. Open the page in Admin → Pages
  2. Scroll to the SEO & Meta section
  3. Set Visibility to Members or Paid
  4. Save. Non-authorised visitors will be redirected to /outpost/login
Warning Page-level visibility only applies to pages managed in Admin → Pages. It does not automatically restrict collection item URLs. To gate individual collection items, use template-level conditionals as shown above.

Show different navigation links depending on the member's login state. This goes in your partials/nav.html.

7. The two-tier model

Outpost supports two membership tiers out of the box. The template variables are:

VariableTrue when
{% if member %}Visitor is logged in (free or paid)
{% if paid_member %}Visitor is logged in with a paid subscription
{% if free_member %}Visitor is logged in but does not have paid status

A typical content-gating decision tree:

  1. Is this a free article? — show full content to everyone
  2. Is the visitor a paid member? — show full content
  3. Is the visitor a free member? — show teaser + upgrade CTA
  4. Is the visitor logged out? — show teaser + register + login CTA

9. Upgrade flow with a payment provider

Outpost does not include a built-in payment gateway. The recommended approach is to use a third-party payment provider (such as Stripe or Lemon Squeezy) and update the member's status via the Member API after a successful payment.

  1. Visitor clicks "Upgrade to Paid" and lands on your checkout page (or a Stripe payment link)
  2. Payment provider processes the payment and fires a webhook to a URL on your server
  3. Your webhook handler calls the Member API to set the member's paid flag to true
  4. Member can now access paid content on their next page load
// Example webhook handler (standalone PHP file)
// webhook.php — called by Stripe after successful payment

<?php
$payload = file_get_contents('php://input');
$event = json_decode($payload, true);

if ($event['type'] === 'checkout.session.completed') {
  $email = $event['data']['object']['customer_details']['email'];

  // Call Outpost Member API to upgrade the member
  $ch = curl_init('https://yoursite.com/outpost/api.php?action=member_upgrade');
  curl_setopt($ch, CURLOPT_POST, true);
  curl_setopt($ch, CURLOPT_POSTFIELDS, json_encode([
    'email' => $email,
    'paid'  => true,
  ]));
  curl_setopt($ch, CURLOPT_HTTPHEADER, [
    'Content-Type: application/json',
    'X-API-Key: ' . getenv('OUTPOST_API_KEY'),
  ]);
  curl_setopt($ch, CURLOPT_RETURNTRANSFER, true);
  curl_exec($ch);
  curl_close($ch);
}

http_response_code(200);
Note Always verify the webhook signature from your payment provider before processing the payload. Never trust the raw POST body without validation.

10. Customising the login page appearance

The built-in login page at /outpost/login uses the admin CSS. To create a branded login experience, create a login.html template at your site root that posts to the Member API login endpoint directly.

{# login.html — custom branded login page #}
{% include 'head' %}
{% include 'nav' %}

<main class="auth-page">
  <div class="auth-box">
    <h1>Sign in</h1>
    <form method="POST" action="/outpost/member/login">
      <label>
        Email
        <input type="email" name="email" required autocomplete="email">
      </label>
      <label>
        Password
        <input type="password" name="password" required autocomplete="current-password">
      </label>
      <button type="submit">Sign in</button>
    </form>
    <p>No account? <a href="/register">Create one free</a></p>
    <p><a href="/outpost/password-reset">Forgot your password?</a></p>
  </div>
</main>

{% include 'footer' %}

11. Conditional layout based on member state

You can use member conditionals anywhere in a template — not just around content. A common pattern is to show different homepage layouts to logged-in members versus visitors:

{% include 'head' %}
{% include 'nav' %}

<main>
  {% if member %}
    {# Personalised dashboard for logged-in members #}
    <section class="member-dashboard">
      <h1>Welcome back</h1>
      <h2>Latest for members</h2>
      <div class="articles-grid">
        {% for article in collection.article limit:6 %}
          <article>
            <h3><a href="{{ article.url }}">{{ article.title }}</a></h3>
            {{ article.excerpt }}
          </article>
        {% endfor %}
      </div>
    </section>
  {% else %}
    {# Marketing landing page for logged-out visitors #}
    <section class="hero">
      <h1>{{ @site_name }}</h1>
      <p>{{ @site_tagline }}</p>
      <a href="/outpost/register" class="btn-primary">Get started free</a>
    </section>
  {% endif %}
</main>

{% include 'footer' %}

12. Routing for member pages

In v5, routing is automatic. Create .html files at your site root and the router maps them to URLs. For example, create login.html at the site root and it is accessible at /login:

site root/
  index.html        ← /
  articles.html     ← /articles
  article.html      ← /articles/{slug} (collection URL pattern)
  login.html        ← /login (custom branded login)
  register.html     ← /register (custom branded registration)
  upgrade.html      ← /upgrade
Note Custom /login and /register pages at the site root serve your branded templates. The built-in /outpost/login and /outpost/register URLs continue to work alongside them — you can redirect one to the other using a simple JavaScript redirect or a PHP header.
