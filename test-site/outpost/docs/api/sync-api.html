Sync API

A machine-to-machine API used by the Outpost Builder desktop app to sync theme files between your local development environment and a live server. Authenticated with a static API key — no browser session required.

Overview

The Sync API lives at sync-api.php, separate from all other Outpost API files. It is designed for use by the Outpost Builder Electron app, but you can also use it from any script or CI pipeline that needs to deploy theme files programmatically.

GET  https://example.com/outpost/sync-api.php?action=pull
POST https://example.com/outpost/sync-api.php?action=push
GET  https://example.com/outpost/sync-api.php?action=backup/list
POST https://example.com/outpost/sync-api.php?action=backup/restore
HTTPS required The Sync API enforces HTTPS on all non-localhost requests. Requests over plain HTTP from a non-local host will receive a 403 Forbidden response.

Authentication

Every request must include the sync API key in the X-Outpost-Key header:

X-Outpost-Key: your-64-character-hex-key

Getting the sync key

  1. Log in to the Outpost admin panel.
  2. Navigate to Settings.
  3. Find the "Builder Sync" section and copy the API key, or generate a new one.
  4. Alternatively, use the Admin API: GET api.php?action=sync/key to view (masked), or POST api.php?action=sync/key/regenerate to rotate the key.
Keep the key secret The sync key grants write access to your theme files and (optionally) your database. Store it in environment variables or a secrets manager — never commit it to version control.

Rate limiting

Failed authentication attempts are rate-limited per IP address: 5 failed attempts per hour. After 5 failures the server returns 429 Too Many Requests with a Retry-After header indicating how many seconds to wait. Successful authentication resets the counter.

Missing the header entirely (no X-Outpost-Key) is not counted as a failed attempt — it returns 401 immediately.

File handling

Theme files are transported as base64-encoded strings. The Sync API enforces these constraints on all files:

ConstraintValue
Maximum file size2 MB per file
Allowed extensionsphp, html, htm, css, js, json, svg, txt, md, yml, yaml, xml
Path validationAll paths must stay within themes/{theme-name}/ — traversal attempts are rejected

Endpoints

Pull — fetch a full site snapshot

GET sync-api.php?action=pull

Returns a complete snapshot of the server: all theme files (base64-encoded), a list of uploaded media files (paths and URLs, no binary content), a database snapshot of safe content, and the raw database file (base64-encoded). Used by Outpost Builder when opening a remote site or pulling in the latest server state.

Response:

{
  "outpost_version": "1.0.0",
  "site_url": "https://example.com/outpost",
  "pulled_at": "2024-03-01T10:00:00+00:00",
  "theme": {
    "active": "personal",
    "all": ["personal", "starter"],
    "files": {
      "themes/personal/index.html": "<base64-encoded content>",
      "themes/personal/post.html": "<base64-encoded content>",
      "themes/personal/assets/style.css": "<base64-encoded content>",
      "themes/personal/theme.json": "<base64-encoded content>"
    }
  },
  "uploads": [
    {
      "path": "uploads/photo-abc123.jpg",
      "url": "https://example.com/outpost/uploads/photo-abc123.jpg",
      "size": 204800
    }
  ],
  "db_snapshot": {
    "settings": { "site_name": "My Blog", "active_theme": "personal" },
    "pages": [ ... ],
    "collections": [ ... ],
    "collection_items": [ ... ],
    "folders": [ ... ],
    "labels": [ ... ]
  },
  "db_file": "<base64-encoded SQLite database binary>"
}
Note The db_snapshot excludes the sync API key and all admin user credentials. The db_file contains the raw SQLite binary — it is included for Builder's local preview but should be handled with care.

curl example:

curl -H "X-Outpost-Key: your-key-here" \
  "https://example.com/outpost/sync-api.php?action=pull" \
  -o snapshot.json

Node.js example — decode and save a theme file:

const res = await fetch('https://example.com/outpost/sync-api.php?action=pull', {
  headers: { 'X-Outpost-Key': process.env.OUTPOST_SYNC_KEY },
});

const snapshot = await res.json();

for (const [path, b64] of Object.entries(snapshot.theme.files)) {
  const content = Buffer.from(b64, 'base64').toString('utf8');
  const fullPath = `./local-theme/${path}`;
  await fs.mkdir(path.dirname(fullPath), { recursive: true });
  await fs.writeFile(fullPath, content, 'utf8');
}

console.log(`Pulled ${Object.keys(snapshot.theme.files).length} files`);

Push — deploy theme files to the server

POST sync-api.php?action=push

Deploys theme files to the server. Before writing any files, the server automatically creates a timestamped backup of the current theme. The push is aborted if the backup fails. After writing, all page and template caches are cleared.

Request body:

{
  "theme": {
    "name": "personal",
    "files": {
      "themes/personal/index.html": "<base64-encoded content>",
      "themes/personal/assets/style.css": "<base64-encoded content>"
    }
  },
  "push_content": false
}
FieldRequiredDescription
theme.nameYesThe theme directory name. Letters, numbers, hyphens, and underscores only.
theme.filesYesObject mapping relative file paths to base64-encoded content.
push_contentNoIf true, also applies the db_file or db_snapshot from the body. Default: false.
db_fileNoBase64-encoded raw SQLite database binary. Applied when push_content: true. Takes priority over db_snapshot.
db_snapshotNoStructured content snapshot. Applied when push_content: true and db_file is absent.

File paths must start with themes/{theme-name}/. Any path that attempts directory traversal, uses a disallowed extension, or exceeds 2 MB is skipped and reported in the diff.skipped array.

Response:

{
  "success": true,
  "backup_id": "2024-03-01T10-05-00-a1b2c3d4",
  "diff": {
    "added": ["themes/personal/new-page.html"],
    "modified": ["themes/personal/index.html", "themes/personal/assets/style.css"],
    "skipped": [
      { "path": "themes/personal/../../../etc/passwd", "reason": "invalid path" }
    ]
  },
  "push_content_applied": false
}
Content push caution When push_content: true with db_file, the entire database on the server is replaced with the local copy. This will overwrite any member data, form submissions, analytics, and content created on the live server since the last pull. Only use this when you intentionally want the local database to be the authoritative source.

curl example — push a single file:

CONTENT=$(base64 -i themes/personal/index.html)

curl -X POST "https://example.com/outpost/sync-api.php?action=push" \
  -H "X-Outpost-Key: your-key-here" \
  -H "Content-Type: application/json" \
  -d "{
    \"theme\": {
      \"name\": \"personal\",
      \"files\": {
        \"themes/personal/index.html\": \"${CONTENT}\"
      }
    }
  }"

Node.js example — push all local theme files:

import { readdir, readFile } from 'fs/promises';
import { join, relative } from 'path';

async function walkDir(dir, baseDir = dir) {
  const entries = await readdir(dir, { withFileTypes: true });
  const files = {};
  for (const entry of entries) {
    const full = join(dir, entry.name);
    if (entry.isDirectory()) {
      Object.assign(files, await walkDir(full, baseDir));
    } else {
      const rel = relative(baseDir, full).replace(/\\/g, '/');
      const content = await readFile(full);
      files[`themes/personal/${rel}`] = content.toString('base64');
    }
  }
  return files;
}

const files = await walkDir('./themes/personal');

const res = await fetch('https://example.com/outpost/sync-api.php?action=push', {
  method: 'POST',
  headers: {
    'X-Outpost-Key': process.env.OUTPOST_SYNC_KEY,
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    theme: { name: 'personal', files },
    push_content: false,
  }),
});

const result = await res.json();
console.log(`Added: ${result.diff.added.length}`);
console.log(`Modified: ${result.diff.modified.length}`);
console.log(`Skipped: ${result.diff.skipped.length}`);

List backups

GET sync-api.php?action=backup/list

Returns all server-side theme backups, newest first. Automatically prunes backups older than 30 days before returning the list. Each backup was created automatically before a push operation.

Response:

{
  "backups": [
    {
      "id": "2024-03-01T10-05-00-a1b2c3d4",
      "theme": "personal",
      "created_at": "2024-03-01T10:05:00+00:00",
      "file_count": 12,
      "size_bytes": 48320
    },
    {
      "id": "2024-02-28T15-30-00-e5f6a7b8",
      "theme": "personal",
      "created_at": "2024-02-28T15:30:00+00:00",
      "file_count": 11,
      "size_bytes": 46100
    }
  ]
}

curl example:

curl -H "X-Outpost-Key: your-key-here" \
  "https://example.com/outpost/sync-api.php?action=backup/list"

Restore a backup

POST sync-api.php?action=backup/restore

Restores a specific backup by its ID. Before restoring, the server automatically creates a new backup of the current state so the restore can itself be rolled back. All page and template caches are cleared after the restore.

FieldRequiredDescription
backup_idYesThe backup ID from backup/list.

Request body:

{ "backup_id": "2024-03-01T10-05-00-a1b2c3d4" }

Response:

{
  "success": true,
  "restored": 12,
  "theme": "personal"
}

Error responses:

{ "error": "Backup not found" }        // HTTP 404
{ "error": "Backup manifest missing" } // HTTP 500

curl example:

curl -X POST "https://example.com/outpost/sync-api.php?action=backup/restore" \
  -H "X-Outpost-Key: your-key-here" \
  -H "Content-Type: application/json" \
  -d '{"backup_id":"2024-03-01T10-05-00-a1b2c3d4"}'

Backup storage

Server-side backups are stored in outpost/backups/ (one subdirectory per backup). Each backup directory contains:

PathContents
{backup-id}/manifest.jsonMetadata: ID, theme name, creation time, file count.
{backup-id}/files/Full copy of the theme directory at backup time.

Backups older than 30 days are automatically pruned when backup/list is called. There is no hard limit on the number of backups retained within that window.

Database snapshot format

The db_snapshot object included in a pull response (and accepted in a push body when push_content: true and no db_file is provided) has this structure:

{
  "settings": {
    "site_name": "My Blog",
    "site_url": "https://example.com",
    "active_theme": "personal"
  },
  "pages": [
    {
      "path": "/",
      "title": "Home",
      "meta_title": "",
      "meta_description": "",
      "fields": [
        {
          "field_name": "hero_title",
          "field_type": "text",
          "content": "Welcome",
          "default_value": "",
          "options": "",
          "sort_order": 0
        }
      ]
    }
  ],
  "collections": [
    {
      "slug": "blog",
      "name": "Blog Posts",
      "schema": "...",
      "url_pattern": "/blog/{slug}"
    }
  ],
  "collection_items": [
    {
      "slug": "my-first-post",
      "collection_slug": "blog",
      "status": "published",
      "data": "...",
      "published_at": "2024-02-01 08:00:00"
    }
  ],
  "folders": [ ... ],
  "labels": [ ... ]
}
Content push safety When applying a db_snapshot (not a full db_file), only safe developer-owned data is updated. The following are never overwritten: admin user passwords, the sync API key, member accounts, analytics data, and form submissions. Settings are only updated for safe keys: site_name, site_url, active_theme, and cache_enabled.

Security considerations

The Sync API is a powerful interface — it can overwrite theme files and (optionally) replace the entire database. Keep these points in mind:

  • The API key should be rotated if you suspect it has been leaked. Use POST api.php?action=sync/key/regenerate to generate a new key immediately.
  • Requests over HTTP (not HTTPS) from non-localhost hosts are rejected with 403.
  • All incoming file paths are validated to prevent directory traversal. Paths must stay within themes/{name}/.
  • Only files with permitted extensions (PHP, HTML, CSS, JS, JSON, SVG, etc.) are accepted. Binary image uploads should be done through the Admin API media upload endpoint.
  • Push operations always create a backup before modifying any files. If the backup fails, the push is aborted.
  • Rate limiting prevents brute-force attacks on the API key.
