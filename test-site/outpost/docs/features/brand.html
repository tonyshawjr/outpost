Brand Settings

Set your site-wide colors, fonts, logo, and favicon in one place. Brand settings serve as the baseline identity for your entire site — the Theme Customizer and CSS Framework both read from them.

How it works

Brand is available at Admin → Brand (in the Design & Build section). It requires the settings.* capability.

Brand settings define your site's core visual identity:

  • Palette — six color roles: primary, secondary, accent, neutral, background, and surface
  • Typography — heading font, body font, and a type scale ratio (Minor Second through Golden Ratio)
  • Identity — logo and favicon images

These values are stored in content/data/brand.json and are included in backups.

Where brand values go

Brand settings flow into your site through two paths, depending on how your theme is set up:

Path 1: Theme Customizer (via brand_key)

If your theme has a Theme Customizer section in theme.json, individual customizer fields can map to brand values using the brand_key property. When a field has a brand_key, the brand value becomes the baseline default instead of the field's hardcoded default.

Value priority (highest wins):

  1. Customizer saved value (user customized this specific theme)
  2. Brand value (site-wide baseline via brand_key)
  3. Field default in theme.json (theme developer's fallback)

This means: set your brand colors once, and every theme that maps its customizer fields to brand keys will inherit those colors automatically. Override per-theme in the Customizer when needed.

Tip — The built-in Personal theme already maps its accent, text, background, surface colors, heading/body fonts, logo, and favicon to brand keys. Set your brand once and Personal inherits it.

Path 2: CSS Framework (via "framework": true)

If your theme opts into the Outpost CSS Framework by adding "framework": true to theme.json, brand values are injected directly as --brand-* CSS tokens before </head>. Your theme CSS uses these tokens directly:

/* Brand tokens injected automatically */
:root {
  --brand-primary: #2D5A47;
  --brand-accent: #14B8A6;
  --brand-neutral: #27272A;
  --brand-bg: #FFFFFF;
  --brand-surface: #FAFAFA;
  --brand-font-heading: 'Inter', -apple-system, sans-serif;
  --brand-font-body: 'Inter', -apple-system, sans-serif;
  --text-xs: 0.64rem;
  --text-sm: 0.8rem;
  --text-base: 1rem;
  --text-lg: 1.25rem;
  /* ... through text-5xl */
}

The framework path is independent of the customizer — it reads directly from brand.json every time a page renders.

The brand_key property

Add brand_key to any customizer field in theme.json to map it to a brand value. The format is "section.field" using dotted notation:

{
  "key": "color-accent",
  "label": "Accent Color",
  "type": "color",
  "default": "#14B8A6",
  "var": "--color-accent",
  "brand_key": "colors.accent"
}

The framework flag

Add "framework": true to your theme.json to opt into the Outpost CSS Framework:

{
  "name": "My Theme",
  "version": "1.0.0",
  "framework": true
}

When enabled, Outpost injects three things before </head> on every page:

  1. Google Fonts links for the brand's heading and body fonts
  2. Framework CSS (/outpost/framework/outpost-framework.css) — lightweight reset, grid system, utility classes, button styles
  3. Brand tokens — :root CSS custom properties for all brand values + computed type scale

Your theme CSS then uses var(--brand-primary), var(--brand-font-heading), var(--text-lg), etc.

Note — The framework flag is independent of the customizer. A theme can use the framework without a customizer section, or vice versa, or both.

Storage

Brand settings are stored in content/data/brand.json:

{
  "colors": {
    "primary": "#2D5A47",
    "secondary": "#C4785C",
    "accent": "#14B8A6",
    "neutral": "#27272A",
    "background": "#FFFFFF",
    "surface": "#FAFAFA"
  },
  "typography": {
    "heading_font": "Inter",
    "body_font": "Inter",
    "type_scale": "1.25"
  },
  "identity": {
    "logo": "",
    "favicon": ""
  }
}

This file lives inside content/, so it is included in backups and safe from the auto-updater.

Logo and favicon sync

When you set a logo or favicon in Brand settings, the values are automatically synced to the @site_logo and @site_favicon global fields. This means your existing template tags — {{ @site_logo | image }} and {{ @site_favicon | image }} — work without any changes.
