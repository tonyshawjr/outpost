Analytics

Outpost includes built-in, privacy-respecting analytics. No external service, no cookies, no personal data — traffic data stays on your server and is viewable directly in the admin panel.

Built-in tracker

Outpost automatically injects a lightweight tracking snippet before the closing </body> tag on every page rendered by the template engine. No changes to your theme templates are required. The snippet fires a request to /outpost/track.php using fetch() with an Image() pixel as a fallback for older browsers.

Note — The tracker only fires on pages rendered through Outpost's template engine. Static files (images, CSS, JS) are not tracked.

What is tracked

  • Page path — the URL path of the visited page
  • Referrer domain — extracted from the browser's Referer header
  • Device type — desktop, mobile, or tablet (inferred from user agent string)
  • Session ID — a random ID stored in sessionStorage to identify unique sessions; not persisted across browser sessions
  • Bot detection — common bot and crawler user agents are detected and flagged; bot hits are recorded with is_bot=1 and excluded from all reports
  • Country code — 2-letter country code, only when geo enrichment is enabled (see below)
  • Timestamp — when the pageview occurred

What is not tracked

  • IP addresses are not stored in the analytics table (they are used briefly for rate limiting, then discarded)
  • No personally identifiable information
  • No cookies — the session ID uses sessionStorage, which does not persist across browser sessions and is not a cookie
  • No cross-site tracking
Tip — Because Outpost analytics uses no cookies and collects no personal data, it generally does not require a cookie consent banner for GDPR compliance. Consult your legal adviser for your specific situation.

JavaScript tracking API

The injected tracking snippet exposes a global outpost object with methods for tracking custom interactions beyond pageviews.

Custom events

outpost.track('button_click', { label: 'signup-cta' })

Sends a custom event with an arbitrary name and optional properties object. Events appear in the Events tab.

Search tracking

// Track a search query with the number of results returned
outpost.trackSearch('keyword', 5)

// Track when a user clicks a search result
outpost.trackSearchClick('keyword', '/path/to/result')

Search events appear in the Search tab. See Search Analytics below for details.

Auto-tracked events

The tracking snippet automatically captures the following without any code changes:

  • Outbound clicks — links to external domains
  • File downloads — clicks on links to files (PDF, ZIP, DOCX, etc.)
  • Form submissions — any <form> submit event
  • Search queries from URL params — pages loaded with ?q=, ?search=, or ?s= query parameters are automatically recorded as search events (results count defaults to 0)

Data storage

Analytics data is stored across several tables in your SQLite database (outpost/content/data/cms.db). No data is sent to any external service.

TablePurpose
analytics_hitsPageviews — path, referrer, device, session, timestamp
analytics_eventsCustom events and auto-tracked interactions
analytics_goalsGoal definitions and conversion tracking
analytics_searchesSearch queries, result counts, and click-throughs
member_eventsMember lifecycle events (signup, login, role_change, email_verified)

The Analytics dashboard

Navigate to Admin → Analytics (requires developer or admin role). Use the period selector to view data for the last 7 days, 30 days, 90 days, or 12 months.

The dashboard is organized into six tabs: Traffic, Events, Goals, Search, Content, and Funnels.

Traffic tab

  • Pageviews & Unique Visitors — a line chart comparing total pageviews against unique visitor sessions over the selected period
  • Top pages — table listing each path with pageviews, unique visitors, and average time on page
  • Referrers — domains that sent traffic, with a traffic share percentage for each
  • Device breakdown — proportional split between desktop, mobile, and tablet visits
  • Top countries — visitor geography with flag emoji and percentage bars (only visible when geo enrichment is enabled; see Geo Enrichment below)

Events tab

Shows custom events sent via outpost.track() and auto-tracked interactions (outbound clicks, file downloads, form submissions). Events are grouped by name with total counts and unique session counts.

Goals tab

Define conversion goals and track completion rates. Goals can be triggered by pageviews matching a URL pattern or by specific events.

Search tab v2.3

Track what visitors search for on your site. Search data is collected two ways:

  1. JavaScript API — call outpost.trackSearch('keyword', 5) from your search results page, passing the query string and the number of results returned. Call outpost.trackSearchClick('keyword', '/path') when a user clicks a search result.
  2. Automatic URL detection — pages loaded with ?q=, ?search=, or ?s= query parameters are automatically recorded as search events. The results count defaults to 0 since the tracker cannot know how many results your page displays.

The Search tab displays:

  • Total searches — count of all search events in the period
  • Unique queries — count of distinct search terms
  • Search-to-click rate — percentage of searches that resulted in a click on a result
  • Top queries — most searched terms, with search count and click count
  • Zero-result queries — terms that returned no results (results_count = 0), useful for identifying content gaps
  • Top clicked results — paths that received the most clicks from search results
  • Daily trend chart — search volume over time

Database schema

Search data is stored in the analytics_searches table:

ColumnTypeDescription
queryTEXTThe search term entered by the visitor
results_countINTEGERNumber of results returned (0 for auto-detected URL params)
clicked_pathTEXTPath the user clicked from search results (NULL if no click)
session_idTEXTSession identifier
created_atDATETIMETimestamp of the search event

API endpoint

GET api.php?action=dashboard/search&period=30days

Returns search metrics, top queries, zero-result queries, top clicked paths, and daily trend data for the selected period.

Content tab

  • Publishing cadence — a chart showing how many collection items were published per week over time, useful for understanding content output consistency
  • Top content by pageviews — cross-references analytics hit data with your collection items to show which pieces of content are driving the most traffic
  • SEO health — automated checks across all pages and collection items producing a score from 0 to 100 (missing meta descriptions, title length, Open Graph images, short content, duplicate descriptions, missing excerpts). Results are grouped into Critical, Warnings, and Passing.

Content performance cohorts v2.3

Groups pages by publish date to reveal how traffic differs for new versus established content. This helps you understand whether recent content is gaining traction and how older content continues to perform.

Cohort buckets:

  • Last 7 days — pages published in the past week
  • Last 30 days — pages published in the past month
  • Last 90 days — pages published in the past quarter
  • Last 6 months — pages published in the past half-year
  • Older — everything published before that

Each cohort shows: page count, total views, average views per page, and the top pages within that cohort. No new database tables are required — cohorts are computed from existing analytics_hits and pages tables.

API endpoint

GET api.php?action=dashboard/cohorts&period=30days

Returns cohort breakdowns with page counts, view totals, averages, and top pages per bucket.

Funnels tab v2.3

Visualizes the member lifecycle funnel. This tab is only visible when the members feature is enabled.

Funnel stages:

  1. Visitors — unique sessions in the period
  2. Sign Ups — new member registrations
  3. Logins — members who logged in
  4. Upgrades — members who changed to a paid role

Between each stage the funnel shows conversion rates and drop-off percentages, making it easy to see where visitors fall out of the lifecycle. Below the funnel chart is a recent member activity feed showing the latest signup, login, and upgrade events.

Database schema

Member events are stored in the member_events table:

ColumnTypeDescription
user_idINTEGERThe member's user ID
event_typeTEXTOne of: signup, login, role_change, email_verified
detailsTEXTJSON string with event-specific metadata
session_idTEXTSession identifier
created_atDATETIMETimestamp of the event

API endpoint

GET api.php?action=dashboard/funnels&period=30days

Returns funnel stage counts, conversion rates, drop-off percentages, and recent member activity.

Geo enrichment v2.3

Optional country-level visitor geography. Geo enrichment is double-gated for privacy:

  1. You must upload a MaxMind GeoLite2-Country.mmdb database file
  2. You must enable the geo_enabled setting

Both conditions must be met before any geo data is recorded. Configure this in Settings → Advanced → Geo Analytics.

How it works

When a pageview is recorded and geo enrichment is active, the tracker looks up the visitor's IP against the MaxMind database to resolve a 2-letter country code (e.g. US, DE, JP). Only the country code is stored — the raw IP address is never written to the analytics table.

Country data appears in the Traffic tab as a "Top countries" section showing flag emoji, country name, visit count, and a percentage bar.

Management endpoints

EndpointMethodDescription
api.php?action=dashboard/geoGETTop countries for the selected period (&period=30days)
api.php?action=dashboard/geo/statusGETWhether the MMDB file is present and geo is enabled
api.php?action=dashboard/geo/uploadPOSTUpload a GeoLite2-Country.mmdb file
api.php?action=dashboard/geo/deleteDELETERemove the MMDB file and disable geo enrichment
Privacy note — Geo enrichment only stores a 2-letter country code. No city, region, postal code, or IP address is retained. You can delete the MMDB file at any time to stop geo tracking; existing country codes in the database remain but no new ones are recorded.

GA4 integration

Outpost's built-in analytics and Google Analytics 4 can run in parallel. To add GA4, store your Measurement ID in a global field and inject the GA4 snippet conditionally in your theme's head.html partial.

Setup

  1. Create a global field named ga4_id in Admin → Globals.
  2. Enter your GA4 Measurement ID (e.g. G-XXXXXXXXXX) as the field value.
  3. Add the following to partials/head.html in your theme:
{% if @ga4_id %}
<script async src="https://www.googletagmanager.com/gtag/js?id={{ @ga4_id }}"></script>
<script>
  window.dataLayer = window.dataLayer || [];
  function gtag(){dataLayer.push(arguments);}
  gtag('js', new Date());
  gtag('config', '{{ @ga4_id }}');
</script>
{% endif %}

When @ga4_id is empty the block is omitted entirely, so you can leave the field blank on development sites without modifying your theme files.

Note — GA4 tracks in addition to Outpost's built-in analytics. Both systems record pageviews independently. Numbers will differ slightly because GA4 uses different bot filtering and session definitions.

Excluding system paths from reports

The analytics reports automatically exclude the following paths:

  • /outpost/ and all admin sub-paths
  • /sync-api (the builder sync endpoint)
  • Bare collection slug paths (e.g. /post) — only individual item URLs are counted

These exclusions are built into the report queries and cannot be configured.
