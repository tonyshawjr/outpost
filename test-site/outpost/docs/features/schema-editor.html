Collection Schema Editor

A full-page editor for defining collection fields, settings, and behavior. Drag-sortable field builder with all 15 field types, visual repeater sub-fields, conditional logic, and Lodge configuration.

Accessing the schema editor

Navigate to Admin → Collections and click Edit on any collection. This opens the full-page schema editor at the collection-schema route.

The schema editor replaces the previous inline modal with a spacious, dedicated page where you can configure every aspect of a collection in one place.

Collection settings

The top section of the schema editor contains collection-level settings:

SettingDescription
NameDisplay name shown in the admin sidebar and lists (e.g. “Blog Posts”).
SlugURL-safe identifier used in templates and API calls (e.g. post). Immutable after creation.
URL patternRoute pattern for single item pages (e.g. /post/{slug}). The {slug} placeholder is replaced with the item’s slug.
Sort fieldDefault sort column for collection items. Options: created_at, updated_at, published_at, sort_order.
Items per pagePagination limit for <outpost-each> collection loops on the front end.

Schema field builder

Below the collection settings is the field builder — a drag-sortable list of all fields in the collection’s schema. Each field is displayed as a collapsible row showing the field slug, type badge, and controls.

Adding a field

Click Add Field at the bottom of the field list. Enter a field slug (lowercase, underscores allowed) and select a type from the dropdown. The field appears in the list and can be expanded to configure type-specific options.

Reordering fields

Drag fields by their handle to reorder them. The order in the schema editor determines the order fields appear in the collection item editor.

Removing a field

Click the remove button on any field row. Removing a field from the schema does not delete existing data — the data remains in the database but is no longer displayed in the editor.

Type-specific options

When you expand a field row, the schema editor shows options relevant to that field type:

Select fields

A Choices editor where you define the available options. Enter choices as comma-separated values (e.g. news,event,product).

Relationship fields

A Collection dropdown to choose which collection to link items from. Additional options for multiple (allow selecting more than one item) and max (maximum number of linked items).

Flexible fields

A Layouts JSON editor for defining named layouts, each with its own set of fields. This is the ACF Pro-style flexible content builder.

Repeater fields

A visual sub-field builder (see below).

Visual repeater sub-field builder

Repeater fields get a dedicated sub-field editor. Each sub-field has three properties:

  • Name — the sub-field slug (e.g. skill, level, url)
  • Type — any of the standard field types (text, select, image, etc.)
  • Label — display label shown in the collection item editor

Click Add Sub-Field to add a new sub-field row. Click the remove button to delete a sub-field.

Switch to JSON

For advanced users, a Switch to JSON toggle reveals the raw JSON definition of the sub-fields. This is useful for copy-pasting field definitions between collections or for defining complex nested structures. The visual builder and JSON editor stay in sync — changes in one are reflected in the other.

Example sub-fields JSON:

{
  "skill": { "type": "text", "label": "Skill Name" },
  "level": { "type": "select", "label": "Level", "options": "beginner,intermediate,advanced" }
}

Conditional logic

Any field can have conditions that control its visibility in the collection item editor. Conditions reference other fields in the same schema.

Adding a condition

Expand a field and click Add Condition. Select the target field, choose an operator, and (for == and !=) enter the expected value.

Operators

OperatorDescription
==Equals — field value matches exactly
!=Not equal — field value does not match
not_emptyField has any value (not empty, not 0, not false)
emptyField is empty, 0, or false

Negation

The NOT modifier inverts any condition. For example, “Show this field when category is NOT event” — the field appears for all categories except “event”.

Multiple conditions

When a field has multiple conditions, all must be true (AND logic) for the field to be visible.

Note: Conditional logic is editor-side only. It controls field visibility in the admin UI but does not affect template rendering or API output.

Lodge settings

For Lodge-enabled collections, the schema editor includes a Lodge configuration panel. This is the same configuration available at Admin → Collections → Lodge tab, but accessible directly from the schema editor for convenience.

Lodge settings include:

  • Enable/disable Lodge on the collection
  • Allow create, edit, and delete toggles
  • Require approval toggle (items need admin approval before publishing)
  • Max items per member limit
  • Editable fields whitelist (which fields members can write)
  • Read-only fields (visible but not editable by members)
  • Required tiers (restrict Lodge access to specific member tiers)

See the Lodge documentation for full details on member portal configuration.

Require review

Toggle the Require review option to enable editorial review for the collection. When enabled, items saved by non-admin users are set to pending_review status instead of published. An admin must then approve the item before it appears on the live site.

Schema JSON format

The schema is stored as a JSON object in the collections.schema column. Keys are field slugs and values are objects with type, label, and type-specific options.

{
  "title": { "type": "text", "label": "Title" },
  "body": { "type": "richtext", "label": "Body" },
  "category": {
    "type": "select",
    "label": "Category",
    "options": "news,event,product"
  },
  "featured_image": { "type": "image", "label": "Featured Image" },
  "skills": {
    "type": "repeater",
    "label": "Skills",
    "sub_fields": {
      "skill": { "type": "text", "label": "Skill Name" },
      "level": { "type": "select", "label": "Level", "options": "beginner,intermediate,advanced" }
    }
  },
  "related_posts": {
    "type": "relationship",
    "label": "Related Posts",
    "collection": "post",
    "multiple": true,
    "max": 5
  },
  "event_date": {
    "type": "date",
    "label": "Event Date",
    "conditions": [
      { "field": "category", "operator": "==", "value": "event" }
    ]
  }
}

The schema editor reads and writes this JSON directly. The admin API normalizes incoming schema data — if you send the array format ({fields: [{name, type}]}), it is automatically converted to the flat object format shown above.
