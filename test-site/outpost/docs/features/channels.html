Channels

Connect to external REST APIs, RSS feeds, and CSV files — pull data into your templates with the same template syntax you use for collections. If Collections are content you create, Channels are content that flows in from somewhere else.

What are Channels?

Collections store content you write inside Outpost. Channels do the opposite — they pull content in from external sources and make it available to your templates with familiar data-attribute syntax. No plugins, no Composer dependencies, no build step.

Connect to any JSON API, RSS feed, or CSV file — a headless CMS, a product catalog, a job board, a podcast feed, a Google Sheets spreadsheet — and render the data in your templates using the same <outpost-each> and <outpost-single> tags you already know from collections. Outpost caches the external data in SQLite so pages render instantly, and re-fetches automatically when the cache expires.

Common use cases:

  • Display product listings from a Shopify or WooCommerce API
  • Show job openings from an ATS (Greenhouse, Lever, etc.)
  • Pull real-estate listings from an MLS feed
  • Aggregate blog posts from a headless CMS like Contentful or Strapi
  • Show upcoming events from an external calendar API
  • Display weather data, stock quotes, or any public JSON endpoint
  • Pull blog posts or news from any RSS/Atom feed
  • Render podcast episodes with enclosure links from a podcast RSS feed
  • Import data from a Google Sheets spreadsheet (published as CSV)
  • Load team members, pricing, or any structured data from a CSV file
Note — Channels are read-only. They pull data in from external sources but do not push data back. If you need to create or edit content directly, use collections.

Channel Types

Outpost supports three channel source types. You choose the type when creating a channel — it cannot be changed after creation. All three types use the same caching, sync, and template rendering pipeline, so your templates work identically regardless of the source.

Type Source Best for
REST API Any JSON API endpoint Product catalogs, job boards, headless CMSes, any API with JSON responses
RSS Feed RSS 2.0 or Atom 1.0 feeds Blog aggregation, news feeds, podcast episodes, GitHub releases
CSV Any URL serving a CSV file Google Sheets exports, data files, team directories, pricing tables

RSS Feed Channels

RSS channels parse RSS 2.0 and Atom 1.0 feeds automatically. Outpost maps standard feed fields to item properties:

Field RSS 2.0 source Atom 1.0 source
title<title><title>
link<link><link rel="alternate" href="...">
description<description><summary>
pubDate<pubDate><published> or <updated>
author<author> or <dc:creator><author><name>
content<content:encoded><content>
category<category> (joined with commas)<category term="...">
guid<guid><id>
enclosure_url<enclosure url="...">—

In your templates, access these fields directly: {{ item.title }}, {{ item.content | raw }}, etc.

CSV Channels

CSV channels download and parse a CSV file from any URL. This works with Google Sheets (use the File → Share → Publish to web → CSV link), hosted CSV files, or any endpoint that returns CSV data.

Configuration options:

  • Delimiter — Comma (default), Tab, Semicolon, or Pipe
  • First row is headers — When enabled (default), the first row becomes field names. When disabled, fields are named col_0, col_1, etc.

Header names are automatically sanitized to lowercase with underscores (e.g. First Name becomes first_name). UTF-8 BOM characters are stripped automatically.

Tip — To use a Google Sheet as a channel source, publish the sheet to the web as CSV. The URL will look like https://docs.google.com/spreadsheets/d/.../pub?output=csv. Updates to the sheet will be picked up on the next sync.

Creating a Channel

Navigate to Admin → Channels and click New Channel. Outpost walks you through a four-step wizard to connect, map, configure, and preview your external data source.

Step 1 — Connect

First, choose the source type — REST API, RSS Feed, or CSV. The type cannot be changed after the channel is created. Then enter the source URL and configure any authentication or type-specific options.

REST API channels:

  1. Enter the full API URL (e.g. https://api.example.com/v2/listings)
  2. Choose an authentication method — None, API Key, Bearer Token, or Basic Auth
  3. Add any custom headers your API requires (e.g. Accept: application/json)
  4. Add query parameters that should be included on every request (e.g. status=active, per_page=100)
  5. Click Test Connection to verify the API responds with a 200 status and valid JSON

RSS Feed channels:

  1. Enter the RSS or Atom feed URL (e.g. https://example.com/feed)
  2. Optionally configure authentication if the feed requires it
  3. Click Test Connection to verify the feed parses correctly

CSV channels:

  1. Enter the CSV file URL (e.g. a Google Sheets published CSV link)
  2. Choose the delimiter — Comma (default), Tab, Semicolon, or Pipe
  3. Toggle whether the first row contains headers (enabled by default)
  4. Click Test Connection to verify the CSV parses correctly
Tip — The test connection button sends a real request to the source and shows you the parsed response. Use this to verify your URL, auth credentials, and settings are correct before proceeding.

Step 2 — Schema

After a successful test connection, Outpost displays the discovered fields. This step lets you tell Outpost which fields to use.

  1. Data path (REST API only) — set the dot-notation path to the array of items in the response (see Data Path Extraction). For example, if your API returns {"data": {"listings": [...]}}, enter data.listings. RSS and CSV channels skip this step as the data structure is auto-detected.
  2. Select fields — check the fields you want to include from each item. Unchecked fields are ignored and not cached.
  3. ID field — choose the field that uniquely identifies each item (e.g. id, uuid, _id). For RSS channels, this defaults to guid.
  4. Slug field — choose or generate the field used for URL-friendly slugs (e.g. slug, handle). If the source does not provide a slug, Outpost can auto-generate one from a text field like title or name.

Step 3 — Configure

Set the channel’s identity and behaviour.

Setting Description
Name Human-readable label shown in the admin (e.g. Job Listings)
Slug URL-safe identifier used in templates — channel.slug (e.g. jobs)
Cache TTL How long cached data is considered fresh before Outpost re-fetches from the API. Options: 5 minutes, 15 minutes, 1 hour, 6 hours, Daily, Manual only
Max items Maximum number of items to store from the API. Leave blank for unlimited
Sort field Which field to sort items by when rendering in templates
Sort direction Ascending or descending
URL pattern Route pattern for single-item pages (e.g. /listing/{slug}). Leave blank if you do not need single-item pages

Step 4 — Preview

Outpost fetches the first batch of items and displays them in a table so you can verify the data looks correct. This step also provides auto-generated template code you can copy directly into your templates.

Note — The preview performs a real sync. After completing the wizard, data is already cached and ready for template rendering.

Authentication Methods

Outpost supports four methods for authenticating with external APIs. Choose the method your API provider requires.

Method How it works Example
None No authentication headers are sent. Use this for public APIs that do not require credentials. —
API Key Sends a custom header with your API key. You specify both the header name and the key value. X-API-Key: sk_live_abc123
Bearer Token Sends an Authorization header with a Bearer token prefix. Authorization: Bearer eyJhbGci...
Basic Auth Sends an Authorization header with Base64-encoded username:password credentials. Authorization: Basic dXNlcjpwYXNz
Warning — API keys and tokens are stored in your SQLite database. Keep your outpost/data/ directory protected and never expose it to the public web. Use server-level restrictions to limit access to the data/ directory.

Data Path Extraction

Most APIs do not return a bare JSON array. Instead, the items you care about are nested inside the response under one or more keys. The data path tells Outpost where to find the array of items using dot notation.

Example

Given this API response:

{
  "status": "ok",
  "meta": {
    "total": 142,
    "page": 1
  },
  "data": {
    "listings": [
      { "id": 1, "title": "Oceanfront Villa", "price": 850000 },
      { "id": 2, "title": "Downtown Loft", "price": 425000 },
      { "id": 3, "title": "Mountain Cabin", "price": 310000 }
    ]
  }
}

Set the data path to data.listings and Outpost extracts the three listing objects. The status and meta keys are ignored.

Common data paths

API response shape Data path
[{...}, {...}] (bare array at the top level) (leave blank)
{"data": [{...}]} data
{"results": [{...}]} results
{"response": {"items": [{...}]}} response.items
{"data": {"listings": [{...}]}} data.listings
{"feed": {"entries": [{...}]}} feed.entries
Tip — If you are unsure of the response shape, use the Test Connection button in Step 1. The raw JSON response is displayed in a collapsible panel so you can inspect the structure. In Step 2, Outpost auto-detects the most likely data path for you to confirm or adjust.

Rendering in Templates

Channel data is rendered in your templates using the same loop syntax as collections, with channel. instead of collection..

Basic channel loop

{% for item in channel.listings %}
  <div class="listing-card">
    <h2>{{ item.title }}</h2>
    <p class="price">${{ item.price }}</p>
    <p>{{ item.description }}</p>
    <a href="{{ item.url }}">View details</a>
  </div>
{% else %}
  <p>No listings found.</p>
{% endfor %}

The {% else %} block is optional and renders when the channel has no cached items.

Loop options

Channel loops support the same inline options as collection loops:

Option Description Example
limit Maximum number of items to render {% for item in channel.listings limit:6 %}
orderby Sort by a specific field (overrides the default sort configured on the channel) {% for item in channel.listings orderby:price %}

Accessing fields

Inside the loop, use {{ item.field_name }} to output any field you selected during the schema mapping step. Field values are auto-escaped by default. Use the | raw filter for fields that contain HTML:

{{ item.title }}              <!-- auto-escaped -->
{{ item.body | raw }}         <!-- HTML rendered as-is -->
{{ item.featured_image | image }}  <!-- image tag -->

Full example — job board page

{#--- Template: Careers | Route: /careers ---#}
{% include 'head' %}
{% include 'nav' %}

<main class="careers-page">
  <h1>Open Positions</h1>

  {% for job in channel.jobs limit:20 %}
    <article class="job-posting">
      <h2><a href="/careers/{{ job.slug }}">{{ job.title }}</a></h2>
      <div class="job-meta">
        <span>{{ job.department }}</span>
        <span>{{ job.location }}</span>
        <span>{{ job.type }}</span>
      </div>
      <p>{{ job.excerpt }}</p>
    </article>
  {% else %}
    <p>No open positions at this time. Check back soon.</p>
  {% endfor %}
</main>

{% include 'footer' %}

For full loop syntax details and additional examples, see Channel Loops.

Single Item Pages

Channels support single-item detail pages, just like collections. To enable them, set a URL pattern when creating the channel (Step 3 of the wizard). The URL pattern uses {slug} as a placeholder for the item’s slug value.

Setting up

  1. Set the URL pattern in the channel configuration (e.g. /listing/{slug})
  2. Create a matching template file in your site root (e.g. listing.html)
  3. Use the {% single %} tag in the template to load the item by its slug

The front router extracts the slug from the URL, sets $_GET['slug'], and renders the matching template — the same routing mechanism used for collections.

The single tag

{% single listing from channel.listings %}
  <article class="listing-detail">
    <h1>{{ listing.title }}</h1>
    <div class="price">${{ listing.price }}</div>
    <div class="description">{{ listing.description | raw }}</div>
    <ul class="details">
      <li>Bedrooms: {{ listing.bedrooms }}</li>
      <li>Bathrooms: {{ listing.bathrooms }}</li>
      <li>Square feet: {{ listing.sqft }}</li>
    </ul>
  </article>
{% else %}
  <h1>Listing not found</h1>
  <p>The listing you're looking for may have been removed or the URL is incorrect.</p>
{% endsingle %}

The {% else %} block renders when no item matches the slug in the URL — useful for displaying a 404-style message within your site’s layout.

Full example — single job page

{#--- Template: Job Detail | Route: /careers/{slug} ---#}
{% include 'head' %}
{% include 'nav' %}

<main class="job-detail-page">
  {% single job from channel.jobs %}
    <article>
      <h1>{{ job.title }}</h1>
      <div class="job-meta">
        <span>{{ job.department }}</span> &middot;
        <span>{{ job.location }}</span> &middot;
        <span>{{ job.type }}</span>
      </div>
      <div class="job-body">
        {{ job.description | raw }}
      </div>
      <a href="{{ job.apply_url }}" class="btn">Apply Now</a>
    </article>
  {% else %}
    <h1>Position not found</h1>
    <p>This position may have been filled. <a href="/careers">View all open positions</a>.</p>
  {% endsingle %}
</main>

{% include 'footer' %}

Caching and Sync

Channels store fetched data in your SQLite database, so templates never make HTTP requests at render time. Instead, Outpost manages a local cache that is refreshed based on the TTL you configure for each channel. This applies to all channel types — REST API, RSS, and CSV.

How caching works

  1. When a channel is first created (or manually synced), Outpost fetches data from the external source and stores each item as a row in the channel_items table.
  2. On every template render that includes a channel loop, Outpost checks whether the cache has expired based on the channel’s TTL setting.
  3. If the cache is still fresh, items are served directly from SQLite — no HTTP request is made.
  4. If the cache has expired, Outpost re-fetches from the source in the background. The stale data is served immediately to avoid slow page loads, and the cache is updated for the next request.

Cache TTL options

TTL Re-fetch interval Best for
5 minutes Every 5 min Rapidly changing data (stock prices, live feeds)
15 minutes Every 15 min Frequently updated APIs (news, social feeds)
1 hour Every 60 min Moderately updated data (product catalogs, job boards)
6 hours Every 6 hours Slow-changing data (event calendars, directory listings)
Daily Once per day Rarely changing data (partner lists, static reference data)
Manual only Never auto-refreshes Data you want to control explicitly from the admin panel

Manual sync

Navigate to Admin → Channels → select a channel and click Sync Now to force an immediate re-fetch from the source, regardless of the TTL. This is useful when you know the upstream data has changed and you do not want to wait for the cache to expire.

Sync dashboard

Each channel displays sync status information in the admin:

  • Last synced — timestamp of the most recent successful sync
  • Item count — how many items are currently cached
  • Sync history — log of recent sync operations with timestamps, item counts (added, updated, removed), duration, and any errors
  • Errors — if the last sync failed, the error message and HTTP status code are displayed. Common errors include authentication failures (401/403), rate limiting (429), and connection timeouts
Note — If a sync fails, Outpost keeps serving the previously cached data. Your templates will not break because of a temporary source outage — they will continue to display the last successfully fetched data until the next successful sync.

Pagination Support

Many REST APIs return data in pages rather than all at once. Outpost supports two pagination strategies to automatically fetch all pages during a sync. Pagination is only applicable to REST API channels — RSS and CSV channels always return all items in a single fetch.

Offset-based pagination

The most common pattern. The API accepts page or offset query parameters, and Outpost increments them until all items are fetched.

Configure these settings when creating or editing a channel:

Setting Description Example
Page param The query parameter name for the page number page
Per-page param The query parameter name for items per page per_page
Per-page value Number of items to request per page 100
Total path Dot-notation path to the total count in the response meta.total

Outpost starts at page 1 and keeps requesting subsequent pages until it has fetched all items (determined by the total count) or hits the Max items limit.

Cursor-based pagination

Some APIs use a cursor or token to indicate the next page of results. Outpost extracts the cursor from each response and includes it in the next request.

Setting Description Example
Cursor param The query parameter name to send the cursor value cursor or after
Cursor path Dot-notation path to the next cursor value in the response meta.next_cursor or paging.next

Outpost continues fetching pages until the cursor value is null, empty, or absent from the response, or until the Max items limit is reached.

Tip — If your API does not paginate (it returns all items in a single response), leave the pagination settings blank. Outpost will make a single request and cache whatever items are returned.

Channel vs Collection

Channels and collections serve different purposes but share the same template interface. Here is a side-by-side comparison:

Feature Collections Channels
Data source Created and edited in the Outpost admin Fetched from an external REST API, RSS feed, or CSV file
Template loop syntax {% for item in collection.slug %} {% for item in channel.slug %}
Single item syntax {% single var from collection.slug %} {% single var from channel.slug %}
URL patterns Yes Yes
Editing items Full WYSIWYG editor in admin Read-only — data is managed at the source
Storage SQLite (collection_items table) SQLite cache (channel_items table) with TTL
Freshness Always current (stored locally) Depends on cache TTL and sync schedule
Authentication N/A None, API Key, Bearer Token, or Basic Auth
Folders Yes No
Revisions Yes No — historical data is not tracked
Scheduling Yes (publish/unpublish dates) No — items appear as soon as they are synced

API Endpoints

Channels are managed through the admin API at api.php?action=endpoint. All endpoints require session authentication and a valid CSRF token. For full details, see Admin API.

Method Endpoint Description
GET channels List all channels
GET channels&id=N Get a single channel with its configuration and schema
POST channels Create a new channel
PUT channels&id=N Update an existing channel’s configuration
DELETE channels&id=N Delete a channel and all its cached data
POST channels/test Test an API connection (URL, auth, headers) without saving
POST channels/sync&id=N Trigger a manual sync for a specific channel
GET channels/items&id=N List cached items for a channel (with pagination)
GET channels/history&id=N Get sync history log for a channel
Note — Channel items are read-only through the API. There is no endpoint to create or update individual items — data flows in one direction, from the external API into the Outpost cache.
