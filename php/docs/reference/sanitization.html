Sanitization

How Outpost cleans user-submitted HTML, prevents XSS, handles SVG uploads safely, and ensures every output path is escaped by default.

The Two-Layer Defence

Outpost uses a two-layer approach to prevent cross-site scripting (XSS) and HTML injection:

  1. Input sanitization — richtext HTML is cleaned through an allowlist filter before being stored in the database.
  2. Output escaping — all template output is HTML-escaped by default. Raw HTML is only output when explicitly requested with the | raw filter, and only for richtext fields where HTML is intentional.
Stored clean, not escaped-on-output. Richtext fields store sanitized HTML. When you use {{ body | raw }}, the stored HTML renders as-is. The sanitizer ran when the content was saved — not when it's displayed.

HTML Sanitization — Allowed Tags

When an admin saves a richtext field, the HTML is passed through the Outpost sanitizer. Only the following tags are permitted. All other tags are stripped.

CategoryAllowed Tags
Text & inline p, br, strong, em, b, i, u, s, span
Headings h2, h3, h4
Lists ul, ol, li
Block elements blockquote, div, hr, figure, figcaption
Code pre, code
Links a
Images img
Tables table, thead, tbody, tr, th, td
No h1 in richtext. <h1> is intentionally excluded. Each page should have only one h1, defined in the page template — not inside content fields.

Allowed Attributes

Only specific attributes are permitted on each allowed tag. All other attributes are stripped.

Tag / ScopeAllowed AttributesNotes
a href, target, rel href values starting with javascript: are stripped entirely. External links (not starting with / or #) have noopener noreferrer added to rel automatically.
img src, alt, width, height, class src values starting with javascript: or data: are stripped. Only http, https, and relative paths are allowed.
All block elements class, id Allows editors to apply CSS classes. id permits anchor links within content.
Table cells colspan, rowspan, class Standard table spanning attributes are permitted.

Always Stripped

The following are removed unconditionally from all richtext input, regardless of context:

Element / PatternReason
<script> JavaScript execution vector.
<iframe> Frame injection / clickjacking vector.
<object>, <embed> Plugin execution vectors.
<form>, <input>, <button>, <select> Form injection / phishing vectors.
<style> CSS injection — can be used for UI redressing attacks.
<meta>, <link> Document manipulation.
All on* event attributes onclick, onmouseover, onerror, onload, and all other event handlers are stripped from every tag.
href="javascript:..." JavaScript execution via link click.
Inline style attributes Can contain expression(), url(), and CSS injection vectors.
data: URIs in src Can embed executable content in img tags.

SVG Sanitization

SVG files uploaded through the Media manager receive additional processing because SVG is XML and can contain inline scripts and event handlers.

Removed from SVGWhy
<script> elements Direct script execution within SVG.
All on* event handler attributes SVG supports the same event handlers as HTML — all are stripped.
xlink:href and href pointing to external URLs Can load external resources or trigger navigation.
Inline style containing url() or expression() Can load external resources or execute CSS expressions.
<foreignObject> Embeds HTML within SVG — allows full HTML injection.
SVG sanitization runs at upload time. The cleaned SVG is stored on disk and the original is discarded. When an SVG is displayed with {{ field | image }}, the URL is output-escaped but the SVG content itself is served as-is from disk.

Template Output Escaping

The template compiler generates htmlspecialchars() calls for all text output by default. The only escape hatch is the | raw filter.

Template TagOutputEscaped?
{{ title }} Plain text Yes — htmlspecialchars() with ENT_QUOTES
{{ title | textarea }} Plain text with line breaks Yes — escaped then nl2br()
{{ body | raw }} Sanitized HTML No — outputs stored HTML as-is
{{ hero | image }} File URL Yes — htmlspecialchars()
{{ @site_name }} Global text Yes — htmlspecialchars()
{{ @bio | raw }} Global richtext HTML No — outputs stored HTML as-is
{{ item.title }} Collection item field Yes — htmlspecialchars()
{{ item.body | raw }} Collection item richtext No — richtext fields only
{{ item.excerpt | or_body }} Plain text excerpt Yes — strips HTML, then escapes
Never use | raw on user-controlled text fields. Only use it on richtext fields (field_type = 'richtext') that have already been sanitized on save. Using | raw on a plain text or textarea field would allow stored XSS.

The | or_body Filter

The or_body filter is used in collection loops to generate a plain-text excerpt. It takes the field value (typically an excerpt field), and if that's empty it falls back to the item's body richtext, strips all HTML tags, and returns the first 40 words.

<!-- Uses excerpt if set, otherwise auto-generates from body -->
{{ item.excerpt | or_body }}

The PHP equivalent:

function outpost_auto_excerpt($text, $words = 40): string {
    $text = strip_tags($text);          // remove all HTML tags
    $text = preg_replace('/\s+/', ' ', trim($text));  // collapse whitespace
    $arr = explode(' ', $text);
    if (count($arr) <= $words) return $text;
    return implode(' ', array_slice($arr, 0, $words)) . '…';
}

The output of or_body is always HTML-escaped before rendering — it is safe to use in any context.

Form Submission Sanitization

All data submitted through Outpost forms is sanitized before storage in form_submissions:

  • HTML tags are stripped from all field values with strip_tags().
  • Values are trimmed of leading and trailing whitespace.
  • Output in the admin submissions view is HTML-escaped.
  • Notification emails use plain-text values — no HTML rendering of submitted data.
// Sanitize all submitted form fields before storing
$clean = [];
foreach ($_POST as $key => $val) {
    $clean[$key] = strip_tags(trim((string)$val));
}
$data = json_encode($clean);

noopener for External Links

When the richtext editor inserts a link with target="_blank", the sanitizer automatically ensures rel="noopener noreferrer" is present. This prevents the opened tab from accessing the opener window via window.opener, closing a common tab-napping vector.

<!-- Input (from editor) -->
<a href="https://example.com" target="_blank">Visit us</a>

<!-- After sanitization (stored) -->
<a href="https://example.com" target="_blank" rel="noopener noreferrer">Visit us</a>

What Sanitization Does Not Cover

SQL injection is prevented separately by using PDO prepared statements throughout — all database queries use parameterized placeholders. The sanitizer handles HTML injection; PDO handles SQL injection. These are independent protection layers.

Sanitization does not protect against:

  • CSRF — handled by per-session CSRF tokens on all API-mutating requests.
  • Brute force / rate limiting — no built-in rate limiting on the login endpoint; use a web server or reverse-proxy level solution.
  • Path traversal in file uploads — upload filenames are regenerated server-side to avoid directory traversal; the original filename is stored for display only.
