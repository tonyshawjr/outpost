Architecture

How Outpost CMS is structured, what it intentionally does not include, and how to extend it when you need custom behavior.

Design philosophy

Outpost is a zero-dependency, file-based CMS. It uses PHP 8, SQLite, and Liquid-style HTML templates. There is no Composer, no Node.js runtime, no Redis, no external services. The entire system fits in a single outpost/ directory.

This simplicity is intentional. Outpost prioritizes:

  • One-click deploy to any PHP host
  • No build step for themes
  • No package manager dependencies
  • Self-contained — everything in one directory

Request lifecycle

Every front-end request follows this path:

  1. Web server receives the request. Static files (CSS, JS, images) are served directly — PHP never sees them.
  2. Router (front-router.php or index.php) maps the URL to a template file in the active theme.
  3. Template engine compiles the Liquid .html template to PHP (cached in cache/templates/), then executes it.
  4. Engine functions (engine.php) resolve field values from SQLite, render collection loops, evaluate conditionals.
  5. HTML page cache (optional) stores the fully rendered output for subsequent requests.

No middleware, hooks, or plugin system

Outpost does not have:

  • A middleware pipeline
  • Action/filter hooks (like WordPress)
  • A plugin or extension system
  • Event dispatchers or subscribers
  • Dependency injection containers

This is a deliberate design decision. Hooks and plugins add architectural complexity, security surface area, and maintenance burden. Outpost keeps the codebase small and auditable instead.

How to add custom logic

If you need behavior that Outpost does not provide out of the box, here are the recommended approaches:

Custom API endpoint

Create a standalone PHP file at your document root. It can query Outpost's SQLite database directly or call Outpost's PHP functions:

<?php
// /custom-api.php — standalone endpoint, not part of Outpost
require_once __DIR__ . '/outpost/db.php';

header('Content-Type: application/json');

// Query Outpost's database directly
$items = OutpostDB::fetchAll("SELECT * FROM items WHERE collection_id = ? AND status = 'published'", [1]);

echo json_encode(['items' => $items]);
?>

PHP partial in a Liquid template

Create a .php file in your theme's partials/ directory and include it in a Liquid template. PHP partials execute server-side before the template output is sent:

<?php
// partials/weather-widget.php
$weather = json_decode(file_get_contents('https://api.example.com/weather?city=nyc'), true);
?>
<div class="weather">
  <span><?= htmlspecialchars($weather['temp'] ?? 'N/A') ?>°F</span>
</div>
{% include 'weather-widget' %}

Pre-processing with a cron job

For periodic tasks (syncing data, sending emails, cleaning up), create a PHP script and run it via cron. You can use Outpost's database classes:

# Run every hour
0 * * * * php /var/www/my-task.php

JavaScript on the front end

For interactive features (dynamic filtering, search, real-time updates), use JavaScript in your theme's assets/main.js. Call Outpost's Public Content API for data:

// Fetch published blog posts via the public API
const res = await fetch('/outpost/api.php?action=content/items&collection=blog');
const data = await res.json();
// data.data contains the items array
Note — engine.php is part of Outpost's core and is overwritten during auto-updates. Do not modify it for custom logic — your changes will be lost on the next update. Use standalone files instead.

Key PHP files

FilePurpose
api.phpREST API router — handles all api.php?action= requests

form.phpForm submission handler

Database

Outpost uses a single SQLite file at outpost/content/data/cms.db. SQLite is bundled with PHP — no database server to install or configure. Migrations run automatically on the next request after an update.
