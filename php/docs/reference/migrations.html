Migrations

How Outpost evolves its SQLite schema safely — idempotent migrations that run automatically on every boot with no manual intervention.

The Idempotent Migration Pattern

Outpost does not use a migration version table or a numbered sequence of migration files. Instead, every schema change is written as a self-checking idempotent operation: the code inspects the current schema and applies the change only if it hasn't been applied yet.

This means migrations can run on every single request — and they do. The performance cost is negligible because SQLite's PRAGMA table_info is an in-memory operation, and a static $checked guard ensures each migration function runs its inspection at most once per request.

No migration commands to run. Deploy a new version of Outpost, load any page, and the schema updates automatically. No php artisan migrate, no manual SQL, no downtime.

How PRAGMA table_info Works

SQLite exposes schema information through the PRAGMA command. PRAGMA table_info(table_name) returns one row per column with fields including name, type, notnull, dflt_value, and pk.

-- Check columns on the pages table
PRAGMA table_info(pages);
-- Returns rows like:
-- cid | name        | type    | notnull | dflt_value | pk
--   0 | id          | INTEGER |       0 |       NULL |  1
--   1 | title       | TEXT    |       0 |       NULL |  0
--  ...

In PHP, this looks like:

$db = OutpostDB::connect();
$cols = $db->query("PRAGMA table_info(pages)")->fetchAll(PDO::FETCH_ASSOC);
$colNames = array_column($cols, 'name');
// $colNames = ['id', 'title', 'slug', 'body', ...]

Pattern 1 — Adding a Column

The most common migration: add a column to an existing table only if it doesn't already exist.

function ensure_pages_visibility_column(): void {
    static $checked = false;
    if ($checked) return;
    $checked = true;

    $db = OutpostDB::connect();
    $cols = array_column(
        $db->query("PRAGMA table_info(pages)")->fetchAll(PDO::FETCH_ASSOC),
        'name'
    );

    if (!in_array('visibility', $cols)) {
        $db->exec("ALTER TABLE pages
                   ADD COLUMN visibility TEXT DEFAULT 'public'");
    }
}

Call this function from the API init section or from outpost_init(). The static $checked guard means the PRAGMA query only runs once per PHP process, even if the function is called multiple times.

Pattern 2 — Creating a Table

Use CREATE TABLE IF NOT EXISTS — SQLite's built-in idempotency guard for table creation.

$db->exec("CREATE TABLE IF NOT EXISTS activity_log (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id    INTEGER,
    action     TEXT NOT NULL,
    target     TEXT,
    created_at TEXT DEFAULT (datetime('now'))
)");

This is safe to run every time — SQLite silently does nothing if the table already exists.

Pattern 3 — Adding Multiple Columns at Once

Check all expected columns in one pass and add any that are missing:

function ensure_collection_items_columns(): void {
    static $checked = false;
    if ($checked) return;
    $checked = true;

    $db = OutpostDB::connect();
    $cols = array_column(
        $db->query("PRAGMA table_info(collection_items)")->fetchAll(PDO::FETCH_ASSOC),
        'name'
    );

    // Add each column only if missing
    $toAdd = [
        'scheduled_at'    => "TEXT DEFAULT NULL",
        'meta_title'      => "TEXT DEFAULT ''",
        'meta_description' => "TEXT DEFAULT ''",
        'og_image'        => "TEXT DEFAULT ''",
        'published_at'    => "TEXT DEFAULT NULL",
    ];

    foreach ($toAdd as $col => $def) {
        if (!in_array($col, $cols)) {
            $db->exec("ALTER TABLE collection_items ADD COLUMN {$col} {$def}");
        }
    }
}

Pattern 4 — Table Rebuild (Changing Constraints)

SQLite does not support ALTER TABLE ... DROP COLUMN (in older versions) or changing column constraints in place. The only way to rename columns, change types, or modify UNIQUE constraints is to rebuild the table.

This is the approach used by the ensure_fields_theme_column() migration in engine.php, which added a theme column and changed the UNIQUE constraint:

$db->exec("PRAGMA foreign_keys = OFF");
$db->exec("BEGIN");

// 1. Create the new table with the desired schema
$db->exec("CREATE TABLE fields_new (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    page_id      INTEGER NOT NULL,
    theme        TEXT NOT NULL DEFAULT '',
    field_name   TEXT NOT NULL,
    field_type   TEXT NOT NULL,
    content      TEXT DEFAULT '',
    default_value TEXT DEFAULT '',
    options      TEXT DEFAULT '',
    sort_order   INTEGER DEFAULT 0,
    updated_at   TEXT DEFAULT (datetime('now')),
    UNIQUE(page_id, theme, field_name),
    FOREIGN KEY (page_id) REFERENCES pages(id) ON DELETE CASCADE
)");

// 2. Copy data from old table to new table
$db->exec("INSERT INTO fields_new
    (id, page_id, theme, field_name, field_type, content, default_value, options, sort_order, updated_at)
    SELECT id, page_id, '', field_name, field_type, content, default_value, options, sort_order, updated_at
    FROM fields");

// 3. Drop the old table and rename the new one
$db->exec("DROP TABLE fields");
$db->exec("ALTER TABLE fields_new RENAME TO fields");

// 4. Recreate indexes
$db->exec("CREATE INDEX IF NOT EXISTS idx_fields_page_theme ON fields(page_id, theme)");

$db->exec("COMMIT");
$db->exec("PRAGMA foreign_keys = ON");
Disable foreign keys during rebuild. Always disable PRAGMA foreign_keys before a table rebuild and re-enable it after. Foreign key enforcement would block the DROP TABLE step if other tables reference the one being rebuilt.

Pattern 5 — One-Time Data Migrations

Some migrations need to transform existing data — not just the schema. Use the settings table to record whether the migration has run:

function migrate_backfill_published_at(): void {
    static $checked = false;
    if ($checked) return;
    $checked = true;

    // Check if this migration has already run
    $ran = OutpostDB::fetchOne(
        "SELECT value FROM settings WHERE key = 'migration_backfill_published_at'"
    );
    if ($ran) return;

    // Backfill published_at from created_at for all published items
    OutpostDB::exec("
        UPDATE collection_items
        SET published_at = created_at
        WHERE status = 'published' AND published_at IS NULL
    ");

    // Mark migration as done
    OutpostDB::query(
        "INSERT OR REPLACE INTO settings (key, value) VALUES (?, ?)",
        ['migration_backfill_published_at', '1']
    );
}

SQLite ALTER TABLE Limitations

SQLite's ALTER TABLE is more limited than MySQL or PostgreSQL. Keep these constraints in mind:

OperationSupportWorkaround
ADD COLUMN Supported Native — use directly.
DROP COLUMN SQLite 3.35+ only Leave unused columns in place, or do a full table rebuild if removal is essential.
RENAME COLUMN SQLite 3.25+ only Table rebuild for older versions.
Change column type Not supported Table rebuild required.
Add/change UNIQUE constraint Not supported Table rebuild required.
Add/change foreign key Not supported Table rebuild required.
RENAME TABLE Supported Native — used in rebuild pattern.
Leave old columns in place. When you no longer need a column, it is safest to simply stop using it in code rather than dropping it. Dropping requires a table rebuild, which is riskier and more complex. Unused columns have negligible storage cost in SQLite.

Where Migrations Run

Outpost runs migrations in two places:

  1. outpost_init() in engine.php — runs on every front-end page request. Only lightweight ADD COLUMN checks that use the static $checked guard go here, so they have no perceptible latency impact.
  2. API init in api.php — runs on every admin API request. More migrations can be placed here since admin requests are less latency-sensitive.

Naming Convention

Follow this naming pattern for migration functions and settings keys:

TypePatternExample
Add column ensure_{table}_{column}_column() ensure_pages_visibility_column()
Add multiple columns ensure_{table}_columns() ensure_collection_items_columns()
Create table ensure_{table}_table() ensure_activity_log_table()
Data migration migrate_{description}() migrate_backfill_published_at()
Settings key migration_{description} migration_backfill_published_at
