PHP Classes

The core PHP classes that power Outpost CMS — database access, template compilation, member sessions, email delivery, and HTML sanitization.

Outpost has no Composer dependencies. All classes are hand-written, zero-dependency PHP. Source files live in outpost/: engine.php, template-engine.php, and db.php.

These classes are internal to the CMS engine. Templates interact with them indirectly through template tag functions (cms_text(), cms_global(), etc.). Custom PHP that you write can use OutpostDB directly.

OutpostDB

File: outpost/db.php

A lightweight static wrapper around PDO SQLite. Manages a single shared connection per request, provides helper methods for common query patterns, and ensures WAL mode and foreign keys are enabled on every connection.

Static Methods

MethodReturnsDescription
connect(): PDO PDO Returns the shared PDO connection, creating it on first call. Enables WAL journal mode, sets foreign keys on, and configures error mode to exceptions.
query(string $sql, array $params = []): PDOStatement PDOStatement Prepares and executes a parameterized query. Returns the statement for further fetching.
fetchOne(string $sql, array $params = []): ?array array|null Executes a query and returns the first row as an associative array, or null if no rows found.
fetchAll(string $sql, array $params = []): array array Executes a query and returns all rows as an array of associative arrays. Returns an empty array if no rows.
insert(string $table, array $data): int int Inserts a row into the given table. Keys in $data are column names, values are bound parameters. Returns the new row's auto-increment ID.
update(string $table, array $data, string $where, array $params = []): void void Updates rows matching the $where clause. $data is column→value pairs. $params are bound to the WHERE clause placeholders.
exec(string $sql): void void Executes a raw SQL statement (no parameters). Used for DDL statements like CREATE TABLE, PRAGMA, and migration scripts.

Usage Examples

// Fetch a single row
$page = OutpostDB::fetchOne(
    'SELECT * FROM pages WHERE slug = ?',
    [$slug]
);

// Fetch multiple rows
$items = OutpostDB::fetchAll(
    'SELECT * FROM collection_items WHERE collection_id = ? AND status = ? ORDER BY created_at DESC LIMIT ?',
    [$id, 'published', 10]
);

// Insert a row
$newId = OutpostDB::insert('pages', [
    'title'      => 'My New Page',
    'slug'       => 'my-new-page',
    'status'     => 'draft',
    'created_at' => date('Y-m-d H:i:s'),
]);

// Update rows
OutpostDB::update('pages',
    ['status' => 'published', 'updated_at' => date('Y-m-d H:i:s')],
    'id = ?',
    [$id]
);

// Raw access to PDO for transactions
$pdo = OutpostDB::connect();
$pdo->beginTransaction();
// ... multiple operations ...
$pdo->commit();
Always use parameterized queries. Never interpolate user input into SQL strings. All OutpostDB methods use PDO prepared statements — pass values as the $params array, never via string concatenation.

OutpostTemplateV2 Default

File: outpost/template-engine.php

Compiles HTML templates with data-outpost attributes and <outpost-*> custom elements into cached PHP files. Auto-detected when index.html contains data-outpost= attributes.

Static Methods

MethodReturnsDescription
render(string $templateFile, string $themePath): void void Entry point for rendering a template file. Checks the compiled cache, recompiles if the source is newer, then includes the compiled PHP. Also injects the analytics tracker snippet before </body>.
renderPartial(string $name): void void Renders a partial from the site's partials/ directory. Called automatically when the compiler encounters <outpost-include partial="name" />. Partials are compiled and cached separately.
compile(string $source): string string Compiles a template source string to PHP source. Applies all tag transformations in the correct order and prepends the engine require_once. Can be called directly for testing or programmatic compilation.
clearCache(): void void Deletes all .php files from the template cache directory (cache/templates/). Called by Admin → Settings → Clear Cache.

Compilation Pipeline

The compile() method processes the template source through a series of transformations. Custom elements and data attributes are converted to PHP function calls in a specific order.

  1. <outpost-include partial="name" /> → inlines partial file contents
  2. <outpost-seo /> → generates SEO meta tags
  3. <outpost-form slug="name" /> → renders form with fields
  4. data-outpost="field" → outpost_resolve_field('field', 'text', 'Default')
  5. data-outpost="field" data-type="image" → outpost_resolve_field('field', 'image', 'default.jpg')
  6. data-outpost="field" data-type="link" → sets href attribute from stored value
  7. data-outpost="field" data-type="richtext" → unescaped HTML content
  8. data-outpost="field" data-scope="global" → global field
  9. <outpost-each collection="slug"> → collection loop
  10. <outpost-if field="name" exists> → conditional
  11. <!-- outpost:blockname -->...<!-- /outpost:blockname --> → block grouping

Cache Key

Compiled files are stored as cache/templates/{md5(realpath($templateFile))}.php. The mtime of the source file is compared to the mtime of the compiled file on every request — if the source is newer, it recompiles automatically.

Usage Example

// Render a template file (used by front-router.php)
$siteRoot = OUTPOST_SITE_ROOT;
$template = $siteRoot . '/blog.html';
OutpostTemplateV2::render($template, $siteRoot);

// Compile source manually (useful for testing)
$source = '<h1 data-outpost="title">Hello</h1>';
$compiled = OutpostTemplateV2::compile($source);
// Produces: <h1><?php echo htmlspecialchars(outpost_resolve_field('title', 'text', 'Hello'), ENT_QUOTES, 'UTF-8'); ?></h1>

// Clear the template cache
OutpostTemplateV2::clearCache();

Legacy v1 Engine

The v1 Liquid-style engine (OutpostTemplate in template-engine.php) is still supported for backward compatibility but is not recommended for new sites. It uses {{ }} and {% %} syntax. New projects should use the v2 data-attribute engine described above.

OutpostMedia

File: outpost/media.php

Handles file uploads, deletion, thumbnail generation, and image transformations. All media operations go through this class.

Static Methods

MethodReturnsDescription
upload(array $file): array array Processes an uploaded file — validates extension and MIME type, sanitizes the filename, auto-resizes oversized images, generates a thumbnail, and inserts a database record. Returns the new media row or an ['error' => '...'] array.
delete(array $media): void void Deletes the original file and thumbnail from disk, then removes the database record.
getAbsolutePath(string $relativePath): string string Converts a web-root-relative media path (e.g., /outpost/uploads/photo.jpg) to an absolute filesystem path. Used by the transform handler to locate files on disk.
getRelativePath(string $absolutePath): string string Converts an absolute filesystem path back to a web-root-relative URL path (with leading slash). Used to store updated paths after transformations.
regenerateThumbnail(string $sourcePath, string $filename): string string Generates (or regenerates) a thumbnail for the given image file. Returns the absolute path to the thumbnail, or an empty string if GD is not available or the image format is unsupported.

OutpostPagination

File: outpost/engine.php

A simple static state holder that coordinates between cms_collection_list() and cms_pagination(). When a collection loop uses the paginate: option, the list function sets the total count and page info here so the pagination renderer can use it.

Static Properties

PropertyTypeDescription
$totalintTotal number of items in the collection (before pagination slicing).
$perPageintNumber of items per page.
$currentintCurrent page number (from $_GET['page'], minimum 1).

Static Methods

MethodDescription
set(int $total, int $perPage, int $current): void Called internally by cms_collection_list() when paginating. Sets all three values atomically.

Template development note. You never call these functions directly in templates — the template compiler converts tags like <span data-outpost="title">Default</span> into <?php cms_text('title', 'Default'); ?> automatically. The functions are listed here for PHP developers extending the engine or building custom endpoints.
