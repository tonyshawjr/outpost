Repeater Loops

Loop over structured, multi-row data within a single collection item. Repeater fields let editors add as many rows as needed — each with its own typed sub-fields — making them ideal for feature lists, team members, FAQ items, pricing tiers, and more.

What Is a Repeater Field?

A repeater is a schema field type that stores an ordered list of structured rows. Each row contains the same set of typed sub-fields that you define when building the schema. Unlike a gallery (which stores only images), a repeater can hold any combination of text, image, link, richtext, and other field types per row.

Think of a repeater as a mini-table attached to a collection item: you define the columns (sub-fields) once in the schema, and the editor adds as many rows as needed — without requiring a separate collection.

Note — Repeaters are defined per collection schema. Go to Admin → Collections → select a collection → Edit Schema → Add Field → choose type Repeater, then add sub-fields to it.

Basic Syntax

Loop over a repeater field the same way you would any other loop. The outer variable is the collection item; the field name is the repeater's name in the schema; the inner variable gives you access to each row's sub-fields.

{% for row in item.features %}
  {{ row.title }}
  {{ row.description }}
{% endfor %}

Where:

  • item is the loop variable from a collection loop or {% single %} block
  • features is the name of the repeater field in the schema
  • row is the loop variable representing one row of the repeater
  • row.title, row.description, etc. are the sub-fields defined in that repeater

Sub-Field Filters

Sub-fields use the same output filters as regular collection fields. Apply the appropriate filter based on the sub-field type.

Sub-field type Template syntax Notes
Text {{ row.label }} Auto-escaped. No filter needed.
Textarea {{ row.description | textarea }} Preserves line breaks as <br> tags.
Richtext {{ row.body | raw }} Outputs HTML as-is. Only use | raw for trusted editor content.
Image {{ row.icon | image }} Resolves to the full public URL of the uploaded image.
Link {{ row.cta_url | link }} Ensures the URL is safe to output in an href attribute.
Select {{ row.tier | select }} Outputs the selected option value.

Repeater Inside a Collection Loop

Nest the repeater loop inside an outer collection loop to render structured data for each collection item on a listing page.

{% for service in collection.service %}
  <article class="service-card">
    <h2>{{ service.title }}</h2>
    <p>{{ service.description }}</p>

    {% if service.features %}
      <ul class="feature-list">
        {% for feature in service.features %}
          <li>
            {% if feature.icon %}
              <img src="{{ feature.icon | image }}" alt="" aria-hidden="true">
            {% endif %}
            {{ feature.label }}
          </li>
        {% endfor %}
      </ul>
    {% endif %}

    <a href="{{ service.url }}">Learn more</a>
  </article>
{% endfor %}

Repeater Inside a Single Item

Use a repeater on a detail page to render structured data that belongs to just that item.

{% single product from collection.product %}
  <article>
    <h1>{{ product.title }}</h1>
    <p class="price">{{ product.price }}</p>
    {{ product.description | raw }}

    {% if product.specs %}
      <table class="spec-table">
        <thead>
          <tr><th>Specification</th><th>Value</th></tr>
        </thead>
        <tbody>
          {% for spec in product.specs %}
            <tr>
              <td>{{ spec.name }}</td>
              <td>{{ spec.value }}</td>
            </tr>
          {% endfor %}
        </tbody>
      </table>
    {% endif %}
  </article>
{% endsingle %}

Empty Repeater

If an editor has not added any rows to a repeater field, the loop body is silently skipped. No error is thrown. Use an {% if item.repeater_field %} check if you need to conditionally show a section heading alongside the loop.

{% if product.specs %}
  <section class="specs">
    <h2>Specifications</h2>
    {% for spec in product.specs %}
      <div class="spec-row">
        <strong>{{ spec.name }}</strong>
        <span>{{ spec.value }}</span>
      </div>
    {% endfor %}
  </section>
{% endif %}

Row Order

Rows are returned in the order the editor arranged them in the admin. The repeater field supports drag-and-drop reordering — the template always reflects the saved order with no additional configuration.

Common Use Cases

The following table shows typical repeater field configurations for common content patterns.

Use case Repeater field name Sub-fields
Feature list features icon (image), label (text), description (text)
FAQ faqs question (text), answer (richtext)
Team members team name (text), role (text), photo (image), bio (textarea)
Pricing tiers tiers name (text), price (text), features (textarea), cta_label (text), cta_url (link)
Testimonials testimonials quote (textarea), author (text), company (text), avatar (image)
Timeline timeline date (text), event (text), description (textarea)
Product specs specs name (text), value (text)

Example: FAQ Accordion

A service collection item with an faqs repeater (sub-fields: question, answer). The HTML uses the native <details>/<summary> elements for a no-JavaScript accordion.

{% single service from collection.service %}
  <h1>{{ service.title }}</h1>
  {{ service.description | raw }}

  {% if service.faqs %}
    <section class="faq-section">
      <h2>Frequently Asked Questions</h2>
      {% for faq in service.faqs %}
        <details class="faq-item">
          <summary class="faq-question">{{ faq.question }}</summary>
          <div class="faq-answer">
            {{ faq.answer | raw }}
          </div>
        </details>
      {% endfor %}

  {% endif %}
{% endsingle %}

Example: Pricing Table

A pricing collection item with a tiers repeater (sub-fields: name, price, description, features, cta_label, cta_url, highlighted).

{% single page from collection.pricing %}
  <div class="pricing-grid">
    {% for tier in page.tiers %}
      <div class="pricing-card {% if tier.highlighted %}is-featured{% endif %}">
        <h3>{{ tier.name }}</h3>
        <div class="price">{{ tier.price }}</div>
        {% if tier.description %}
          <p class="tier-desc">{{ tier.description }}</p>
        {% endif %}
        {% if tier.features %}
          <ul class="tier-features">
            {% for line in tier.features %}
              <li>{{ line.item }}</li>
            {% endfor %}
          </ul>
        {% endif %}
        {% if tier.cta_label %}
          <a href="{{ tier.cta_url | link }}" class="btn {% if tier.highlighted %}btn-primary{% else %}btn-secondary{% endif %}">
            {{ tier.cta_label }}
          </a>
        {% endif %}
      </div>
    {% endfor %}
  </div>
{% endsingle %}
Tip — The highlighted sub-field above is a text field containing "true" or "false" as set by the editor (or a checkbox-style select). Use a conditional like {% if tier.highlighted == "true" %} to apply special styling to the recommended tier.

Conditionals on Sub-Fields

Not every row in a repeater is guaranteed to have every sub-field filled in — editors may leave optional sub-fields blank. Always guard optional sub-fields with {% if row.field %} before rendering them.

{% for step in guide.steps %}
  <div class="step">
    <h3>{{ step.title }}</h3>
    {% if step.image %}
      <img src="{{ step.image | image }}" alt="{{ step.title }}">
    {% endif %}
    {% if step.note %}
      <aside class="step-note">{{ step.note }}</aside>
    {% endif %}
    {{ step.body | raw }}
  </div>
{% endfor %}
Warning — Outputting an image sub-field without checking {% if row.image %} first will render a broken <img> tag when the editor leaves the field blank. Always guard image and link sub-fields with a conditional.

Repeaters vs. Separate Collections

Use a repeater when the rows belong to a single parent item and do not need their own URL, status, or search visibility. Use a separate collection when each row needs to be independently published, filtered, or linked to from other pages.

Use repeater when... Use a separate collection when...
Items only make sense as part of their parent (e.g., a product's spec rows) Items have their own detail pages (e.g., blog posts, team member bios)
The number of rows is modest and managed by one editor Items need individual publish status (draft / scheduled / published)
Items do not need to be filtered or queried independently Items need to appear in multiple places (e.g., related posts, category pages)
Simple structured data — testimonials, FAQs, bullet points Complex structured data with folder labels, SEO fields, or media
