Repeater Loops

Loop over structured, multi-row data within a single collection item. Repeater fields let editors add as many rows as needed — each with its own typed sub-fields — making them ideal for feature lists, team members, FAQ items, pricing tiers, and more.

What Is a Repeater Field?

A repeater is a schema field type that stores an ordered list of structured rows. Each row contains the same set of typed sub-fields that you define when building the schema. Unlike a gallery (which stores only images), a repeater can hold any combination of text, image, link, richtext, and other field types per row.

Think of a repeater as a mini-table attached to a collection item: you define the columns (sub-fields) once in the schema, and the editor adds as many rows as needed — without requiring a separate collection.

Note — Repeaters are defined per collection schema. Go to Admin → Collections → select a collection → Edit Schema → Add Field → choose type Repeater, then add sub-fields to it.

Basic Syntax

Loop over a repeater field the same way you would any other loop. The outer variable is the collection item; the field name is the repeater's name in the schema; the inner variable gives you access to each row's sub-fields.

Where:

  • item is the loop variable from a collection loop or <outpost-single> block
  • features is the name of the repeater field in the schema
  • row is the loop variable representing one row of the repeater
  • row.title, row.description, etc. are the sub-fields defined in that repeater

Sub-Field Filters

Sub-fields use the same output filters as regular collection fields. Apply the appropriate filter based on the sub-field type.

Sub-field type Template syntax Notes
Text data-outpost="label" Auto-escaped. No filter needed.
Textarea data-outpost="description" data-type="textarea" Preserves line breaks as <br> tags.
Richtext data-outpost="body" data-type="richtext" Outputs HTML as-is. Only use | raw for trusted editor content.
Image data-outpost="icon" data-type="image" Resolves to the full public URL of the uploaded image.
Link data-outpost="cta_url" data-type="link" Ensures the URL is safe to output in an href attribute.
Select data-outpost="tier" data-type="select" Outputs the selected option value.

Repeater Inside a Collection Loop

Nest the repeater loop inside an outer collection loop to render structured data for each collection item on a listing page.

Repeater Inside a Single Item

Use a repeater on a detail page to render structured data that belongs to just that item.

Empty Repeater

If an editor has not added any rows to a repeater field, the loop body is silently skipped. No error is thrown. Use an <outpost-if field="repeater_field" exists> check if you need to conditionally show a section heading alongside the loop.

Row Order

Rows are returned in the order the editor arranged them in the admin. The repeater field supports drag-and-drop reordering — the template always reflects the saved order with no additional configuration.

Conditionals on Sub-Fields

Not every row in a repeater is guaranteed to have every sub-field filled in — editors may leave optional sub-fields blank. Always guard optional sub-fields with <outpost-if field="field" exists> before rendering them.

Warning — Outputting an image sub-field without checking <outpost-if field="image" exists> first will render a broken <img> tag when the editor leaves the field blank. Always guard image and link sub-fields with a conditional.

Repeaters vs. Separate Collections

Use a repeater when the rows belong to a single parent item and do not need their own URL, status, or search visibility. Use a separate collection when each row needs to be independently published, filtered, or linked to from other pages.

Use repeater when... Use a separate collection when...
Items only make sense as part of their parent (e.g., a product's spec rows) Items have their own detail pages (e.g., blog posts, team member bios)
The number of rows is modest and managed by one editor Items need individual publish status (draft / scheduled / published)
Items do not need to be filtered or queried independently Items need to appear in multiple places (e.g., related posts, category pages)
Simple structured data — testimonials, FAQs, bullet points Complex structured data with folder labels, SEO fields, or media

Template Engine (Outpost 4.1.0+) uses <outpost-each repeat="..."> to loop over repeater fields. The repeater field name must match the name defined in the collection's schema.

Basic Syntax

<outpost-each repeat="features">
 <div class="feature">
 <img data-outpost="icon" data-type="image" alt="">
 <h3 data-outpost="title">Feature Title</h3>
 <p data-outpost="description">Feature description</p>
 </div>
</outpost-each>

Each iteration represents one row of the repeater. Sub-fields are bound to elements using data-outpost with the sub-field name. Filters work the same as in collection loops:

Sub-field typeSyntax
Text<span data-outpost="label">Default</span>
Richtext<div data-outpost="body" data-type="richtext"></div>
Image<img data-outpost="icon" data-type="image">
Link<a data-outpost="cta_url" data-type="link">...</a>
Textarea<p data-outpost="description" data-type="textarea"></p>
