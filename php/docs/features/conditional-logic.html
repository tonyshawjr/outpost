Conditional Logic

Show and hide fields in the collection editor based on other field values. Conditional logic keeps the editing experience clean by only displaying fields that are relevant to the editor's current choices.

What Is Conditional Logic?

Conditional logic lets you define rules on individual fields in a collection schema that control whether the field is visible in the admin editor. When the condition is not met, the field is hidden from the editor — reducing clutter and guiding editors through complex content structures.

Note — Conditional logic is UI-only. It affects field visibility in the admin editor but has no effect on template rendering. All fields and their values are always available in templates regardless of conditional logic rules. Use template {% if %} tags if you need to conditionally render content on the front end.

Setting Up Conditional Logic

Conditional logic is configured per field in the collection schema builder:

  1. Go to Admin → Collections → select a collection → Edit Schema
  2. Select the field you want to conditionally show or hide
  3. In the field settings, find the Conditional Logic section
  4. Choose the controlling field (the field whose value determines visibility)
  5. Choose the operator (how to evaluate the controlling field's value)
  6. Set the comparison value (if the operator requires one)

Available Operators

Operator Description Requires value?
equals Show the field when the controlling field's value exactly matches the comparison value. Yes
not equal Show the field when the controlling field's value does not match the comparison value. Yes
has value Show the field when the controlling field is non-empty (any value at all). No
is empty Show the field when the controlling field has no value. No

Common Patterns

Toggle-driven fields

The most common pattern: use a toggle field to show or hide a group of related fields. This is ideal for optional sections that have their own sub-fields.

Schema:
  show_cta       (toggle)
  cta_title      (text)      → show when: show_cta equals "true"
  cta_url        (link)      → show when: show_cta equals "true"
  cta_label      (text)      → show when: show_cta equals "true"

When the editor turns Show CTA off, the three CTA fields disappear from the editor. When they turn it on, the fields appear — making it clear that these fields are part of a CTA block.

Select-driven fields

Show different fields depending on a select dropdown value. Useful when the same collection has items with different structures.

Schema:
  layout_type    (select: "standard", "featured", "minimal")
  subtitle       (text)      → show when: layout_type equals "featured"
  badge_label    (text)      → show when: layout_type equals "featured"
  hero_image     (image)     → show when: layout_type not equal "minimal"

Presence-based visibility

Show a field only when another field has been filled in. Useful for caption or alt text fields that only matter when an image has been uploaded.

Schema:
  hero_image     (image)
  image_caption  (text)      → show when: hero_image has value
  image_alt      (text)      → show when: hero_image has value

Emptiness check

Show a fallback field only when another field is empty. Useful for providing alternative content paths.

Schema:
  video_url      (link)
  fallback_image (image)     → show when: video_url is empty

Practical Examples

Blog post with optional CTA

Collection: Blog Posts (slug: post)
Schema:
  title           (text)
  body            (richtext)
  author          (text)
  show_cta        (toggle)         ← controlling field
  cta_title       (text)           → show when: show_cta equals "true"
  cta_url         (link)           → show when: show_cta equals "true"
  cta_button_text (text)           → show when: show_cta equals "true"

The corresponding template uses standard {% if %} tags to conditionally render the CTA section on the front end:

{% single post from collection.post %}
  <article>
    <h1>{{ post.title }}</h1>
    <div class="prose">{{ post.body | raw }}</div>

    {% if post.show_cta %}
      <div class="post-cta">
        <h3>{{ post.cta_title }}</h3>
        <a href="{{ post.cta_url | link }}" class="btn btn-primary">
          {{ post.cta_button_text }}
        </a>
      </div>
    {% endif %}
  </article>
{% endsingle %}

Event with conditional pricing

Collection: Events (slug: event)
Schema:
  title           (text)
  date            (date)
  body            (richtext)
  is_paid         (toggle)         ← controlling field
  price           (text)           → show when: is_paid equals "true"
  ticket_url      (link)           → show when: is_paid equals "true"
  max_attendees   (number)         → show when: is_paid equals "true"

Team member with role-specific fields

Collection: Team (slug: member)
Schema:
  name            (text)
  role            (select: "executive", "engineer", "designer", "marketing")
  bio             (textarea)
  photo           (image)
  department      (text)           → show when: role not equal "executive"
  reports_to      (text)           → show when: role not equal "executive"
  executive_bio   (richtext)       → show when: role equals "executive"
  board_member    (toggle)         → show when: role equals "executive"

Important Notes

  • UI-only — Conditional logic controls field visibility in the admin editor only. It does not affect data storage or template rendering. A hidden field retains its value — it is not cleared when hidden.
  • No cascading — Conditions reference one controlling field at a time. You cannot create compound conditions (e.g., "show when field A equals X AND field B equals Y"). For complex scenarios, consider using a single controlling field with more specific values.
  • Template independence — Always use template {% if %} tags for front-end visibility. The conditional logic system in the schema builder and the template conditional system are completely independent — they serve different purposes.
  • Toggle values — Toggle fields store "true" or "" (empty string). Use equals "true" to check for an enabled toggle.
Tip — When using conditional logic with toggle fields, pair the admin-side condition (show when: show_cta equals "true") with a matching template condition ({% if post.show_cta %}). This keeps the editor and the front end in sync — fields hidden from the editor will also be hidden from visitors.
