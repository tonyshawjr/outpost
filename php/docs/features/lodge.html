Lodge

Member-owned content portals. Enable Lodge on any collection and your members can create, edit, and delete their own items from the front end — with optional approval, tier gating, and field whitelisting.

What is Lodge?

Lodge turns any collection into a member-facing content portal. Instead of requiring admin access to create content, members get their own dashboard where they can manage items they own. Every request is ownership-scoped — members can only see and modify their own items.

Use cases:

  • User-generated blog posts or articles
  • Member directories and profiles
  • Job boards where employers post listings
  • Community classifieds or marketplace listings
  • Student or employee portfolios
  • Event submissions with admin approval

Enabling Lodge on a collection

Navigate to Admin → Collections → [Your Collection] → Lodge tab. Toggle Lodge on and configure the options below. Once enabled, the Lodge API endpoints become available and template tags render for logged-in members.

Lodge configuration options

OptionTypeDefaultDescription
allow_createbooleantrueMembers can create new items.
allow_editbooleantrueMembers can edit their own items.
allow_deletebooleanfalseMembers can delete their own items.
require_approvalbooleanfalseNew items and edits are held in pending status until an admin approves them.
max_items_per_memberinteger0 (unlimited)Maximum number of items a single member can own. 0 means no limit.
editable_fieldsarray[] (all)Whitelist of field slugs the member can write. Empty array means all schema fields are editable.
readonly_fieldsarray[]Fields visible to the member but not editable (e.g. admin notes, approval status).
required_tiersarray[] (any member)Member tiers allowed to use Lodge. Empty array means any logged-in member. Example: ["paid", "premium"].

Lodge API endpoints

All Lodge endpoints live under member-api.php?action=lodge/*. They require an active member session (cookie or JWT bearer token). CSRF tokens are required for POST, PUT, and DELETE requests when using session auth.

GET member-api.php?action=lodge/dashboard

Returns an overview of the member’s Lodge activity across all Lodge-enabled collections.

      "pending_items": 2,
      "published_items": 10,
      "max_items": 0,

  ],

      "item_id": 45,
      "title": "My Latest Article",
      "status": "published",
      "updated_at": "2026-03-19 14:30:00"
    }
  ]
}

GET lodge/items

GET member-api.php?action=lodge/items&collection=slug

Returns the member’s own items for the specified collection. Supports pagination.

ParamRequiredDescription

  "page": 1,
  "pages": 1
}

POST lodge/items

POST member-api.php?action=lodge/items&collection=slug

Creates a new item owned by the current member. Only fields in editable_fields are accepted (if configured). Returns the created item.

Request body:

{
  "title": "My New Post",
  "body": "<p>Hello world</p>",
  "category": "tutorials"
}

Success response (HTTP 201):

{
  "success": true,
  "item": {
    "id": 46,
    "slug": "my-new-post",
    "status": "published",
    "owner_member_id": 50,
    "data": { "title": "My New Post", "body": "..." }
  }
}

When require_approval is enabled, status will be "pending" instead of "published".

Error responses:

{ "error": "Lodge is not enabled for this collection" }     // 403
{ "error": "You have reached the maximum number of items" }  // 403
{ "error": "Your membership tier does not have access" }     // 403
{ "error": "Title is required" }                              // 400

PUT lodge/items

PUT member-api.php?action=lodge/items&id=123

Updates an item owned by the current member. Only fields in editable_fields are accepted. The member must own the item.

Request body (partial update — only changed fields):

{
  "title": "Updated Title",
  "body": "<p>Revised content</p>"
}

Success response:

{ "success": true }

When require_approval is enabled, the item status reverts to "pending" after an edit.

Error responses:

{ "error": "Item not found or not owned by you" }  // 404
{ "error": "Editing is not allowed" }               // 403

DELETE lodge/items

DELETE member-api.php?action=lodge/items&id=123

Deletes an item owned by the current member. Only works when allow_delete is enabled.

Success response:

{ "success": true }

Error responses:

{ "error": "Item not found or not owned by you" }  // 404
{ "error": "Deleting is not allowed" }              // 403

POST lodge/upload

POST member-api.php?action=lodge/upload&item_id=123

Uploads a file and attaches it to an item owned by the current member. The item must exist and be owned by the member. Uses multipart/form-data.

FieldRequiredDescription
fileYesThe file to upload (multipart form field).
item_idYesQuery param — the item to attach the file to.

Success response:

{
  "success": true,
  "url": "/outpost/uploads/my-image.jpg",
  "filename": "my-image.jpg"
}

Error responses:

{ "error": "Item not found or not owned by you" }  // 404
{ "error": "File type not allowed" }                // 400
{ "error": "File too large" }                       // 400
MIME validation — Lodge uploads validate file MIME type against an allowlist (images, PDFs, common document types). Executable files and server-side scripts are always rejected.

Lodge template tags

Three custom elements render Lodge UI in your templates. They require a logged-in member session to display content — unauthenticated visitors see nothing.

<outpost-lodge-dashboard>

Renders a member dashboard showing item counts and recent activity across all Lodge-enabled collections.

<outpost-lodge-dashboard>
  <h2>Your Dashboard</h2>
  <div class="lodge-stats">
    <span data-outpost="total_items">0</span> items
    <span data-outpost="pending_items">0</span> pending
  </div>
  <div class="lodge-recent">
    <outpost-each var="activity">
      <div>
        <span data-outpost="title">Item</span>
        <span data-outpost="status">published</span>
        <span data-outpost="updated_at">date</span>
      </div>
    </outpost-each>
  </div>
</outpost-lodge-dashboard>

<outpost-lodge-items>

Renders a list of the member’s own items for a given collection.

<outpost-lodge-items collection="posts">
  <article>
    <h3 data-outpost="title">Post Title</h3>
    <span data-outpost="status">published</span>
    <a data-bind="href:edit_url">Edit</a>
    <a data-bind="href:delete_url">Delete</a>
  </article>
</outpost-lodge-items>
AttributeRequiredDescription
collectionYesCollection slug.
limitNoItems per page (default 20).
statusNoFilter: published, pending, draft, or all.

<outpost-lodge-form>

Renders a create or edit form for a Lodge-enabled collection. Respects the editable_fields whitelist.

<!-- Create new item -->
<outpost-lodge-form collection="posts">
  <label>Title</label>
  <input data-outpost="title" type="text">
  <label>Body</label>
  <textarea data-outpost="body"></textarea>
  <button type="submit">Submit</button>
</outpost-lodge-form>

<!-- Edit existing item (pass item ID via query param) -->
<outpost-lodge-form collection="posts" item="query:id">
  <label>Title</label>
  <input data-outpost="title" type="text">
  <label>Body</label>
  <textarea data-outpost="body"></textarea>
  <button type="submit">Save Changes</button>
</outpost-lodge-form>
AttributeRequiredDescription
collectionYesCollection slug.
itemNoItem ID for editing. Use query:id to read from the URL query string.

Routing

Lodge has its own sub-router built into front-router.php. All Lodge URLs live under a single configurable slug (default: /lodge). Lodge templates go in a lodge/ subdirectory inside your site root.

URL → template mapping

/lodge/edit/42lodge/edit.htmlEdit an item$_GET['item_id'] = 42
/lodge/create/postslodge/create.htmlCreate new item$_GET['collection'] = 'posts'

Custom slug

The Lodge URL prefix is configurable in Admin → Settings (stored as lodge_slug). Change it from /lodge to /portal, /members, or any other slug. The router will look for templates in both {site-root}/{custom-slug}/ and {site-root}/lodge/ (fallback).

Reserved slugs that cannot be used: outpost, admin, api, wp-admin, wp-login.

Authentication

All Lodge routes require an active member session. Unauthenticated visitors are automatically redirected to the member login page (/outpost/member-pages/login.php) with a ?return= parameter so they return to the Lodge page after logging in.

Tip — Lodge can be toggled on/off via the lodge feature flag in Settings → Features. When disabled, all Lodge URLs return a 404.

Starter templates

When you enable Lodge on a collection, the setup UI offers to install starter templates into your site root. These are fully styled, ready-to-use HTML pages that you can customize:

  • Dashboard — item counts, recent activity, quick-create button
  • Item list — filterable table with status badges, edit/delete links, pagination
  • Item form — create/edit form with all editable fields, file upload, and submit button

Approval workflow

When require_approval is enabled on a collection:

  1. Member creates or edits an item → item status is set to pending
  2. Admin sees pending items in the collection list (filtered by status)
  3. Admin approves → status changes to published, item appears on the public site
  4. Admin rejects → status changes to draft with an optional rejection note

Webhook events fire on both submission (lodge.item.submitted) and approval/rejection (lodge.item.approved, lodge.item.rejected).

Security model

  • Ownership scoping — every Lodge API query filters by owner_member_id = :current_member. Members cannot access, edit, or delete items they do not own.
  • Field whitelisting — only fields listed in editable_fields are accepted by the API. Unlisted fields in the request body are silently dropped. If editable_fields is empty, all schema fields are writable.
  • Rate limiting — Lodge endpoints are rate-limited per member session to prevent automated abuse. Create and upload endpoints have stricter limits.
  • Atomic transactions — create, update, and delete operations run inside database transactions to prevent partial writes on failure.
  • MIME validation — uploaded files are validated against an allowlist of safe MIME types. Executable files and PHP scripts are always rejected.
  • Slug sanitization — member-submitted slugs are stripped of special characters, path separators, and null bytes to prevent traversal attacks.
  • XSS protection — all member-submitted content is sanitized before storage. Output is escaped by the template engine.
  • Tier enforcement — when required_tiers is configured, the API checks the member’s tier before allowing any Lodge operation.
Tip — Lodge is designed for member-facing content. For admin-facing content workflows (multi-step approval with custom stages), see Collaboration & Workflows.
