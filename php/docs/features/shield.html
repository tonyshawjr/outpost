Shield Security Suite

Comprehensive security hardening for your Outpost site. Protects against brute force attacks, SQL injection, XSS, path traversal, and more — all configurable from Settings > Shield.

Overview

Shield is a built-in security suite that runs on every request. It provides seven layers of protection:

  1. Login Protection — lockout after failed attempts
  2. IP Blocklist — manual and automatic IP blocking
  3. WAF-lite Firewall — blocks common attack patterns
  4. File Integrity Monitoring — detects modified core files
  5. Security Headers — adds protective HTTP headers
  6. Traffic Logging — monitors recent requests
  7. Email Notifications — alerts on security events

Shield is enabled by default. All features can be toggled individually from Settings > Shield.

Login Protection

When enabled, Shield tracks failed login attempts by IP address. After a configurable number of failures (default: 5), the IP is temporarily blocked for a lockout period (default: 15 minutes).

Setting Default Description
login_max_attempts 5 Failed attempts before lockout triggers
login_lockout_minutes 15 Duration of temporary IP block
auto_block_after_lockouts 3 After this many lockout events, the IP is blocked permanently (no expiry)
Note — Failed attempt counts are based on a rolling time window. Attempts older than the lockout duration are not counted, so the counter resets naturally.

IP Blocklist

Blocked IPs are checked on every request, before any routing or authentication. Blocked requests receive a 403 Forbidden response.

IPs can be blocked in two ways:

  • Manual — add IPs from Settings > Shield > Blocked IPs tab
  • Automatic — IPs that trigger login lockouts are auto-blocked (temporary for single lockouts, permanent after repeated lockouts)

Temporary blocks have an expiry timestamp and are automatically removed after expiry. You can unblock any IP at any time from the admin UI.

Safety — Shield prevents you from blocking your own IP address.

Firewall (WAF-lite)

The firewall scans request URLs, query strings, and POST bodies for common attack patterns:

Rule Detects
sql_injection UNION SELECT, DROP TABLE, SLEEP(), tautologies, comment injection
xss <script> tags, javascript: URLs, inline event handlers (onerror=, etc.)
path_traversal ../, ..\\, URL-encoded variants
php_injection <?php, eval(, base64_decode(, system(, exec(, etc.
null_byte %00 null byte injection

Firewall modes

  • Block (default) — malicious requests are blocked with 403 and logged
  • Log only — requests are logged but allowed through. Use this to monitor before enabling blocking.
Performance — The firewall skips scanning for logged-in admin sessions, so it never interferes with normal admin operations.

File Integrity Monitoring

Shield computes MD5 hashes of all PHP files in the Outpost root directory. On the first scan, hashes are stored as a baseline. Subsequent scans compare current hashes against the stored baseline and report any changes.

Run a check anytime from Settings > Shield > Configuration > Run Check Now. Changed files are listed with their modification type (modified or deleted).

If email notifications are enabled, file integrity changes trigger an alert email.

Security Headers

When enabled, Shield adds the following headers to every API response:

Header Value
X-Content-Type-Options nosniff
X-Frame-Options SAMEORIGIN
X-XSS-Protection 1; mode=block
Referrer-Policy strict-origin-when-cross-origin
Permissions-Policy camera=(), microphone=(), geolocation=()

Traffic Logging

Shield logs the last 1,000 API requests with IP, path, HTTP method, user agent, and timestamp. View the traffic log from Settings > Shield > Traffic tab. Old entries are automatically pruned.

Email Notifications

When enabled, Shield sends email alerts for these events:

  • Login lockout — an IP triggered too many failed login attempts
  • Permanent IP block — an IP was auto-blocked after repeated lockouts
  • Firewall block — an attack pattern was detected and blocked
  • File integrity change — core PHP files were modified or deleted

Emails are sent using your configured SMTP settings from Settings > Email. If SMTP is not configured, Shield falls back to PHP mail().

API Endpoints

All endpoints require admin authentication.

Method Endpoint Description
GET shield/status Security dashboard: blocked IPs count, attacks in 24h, login attempts, last integrity check
GET shield/log Security event log (paginated: limit, offset)
GET shield/blocked-ips List all blocked IPs
POST shield/block-ip Block an IP. Body: { "ip": "...", "reason": "..." }
DELETE shield/block-ip&ip=X Unblock an IP
GET shield/traffic Recent traffic log (paginated)
POST shield/file-check Run file integrity check now
GET shield/config Get current Shield configuration
PUT shield/config Update Shield configuration
GET shield/login-attempts Recent login attempts

Configuration Object

Shield settings are stored as a JSON object in the settings table under the key shield_config:

{
  "enabled": true,
  "login_lockout": true,
  "login_max_attempts": 5,
  "login_lockout_minutes": 15,
  "auto_block_after_lockouts": 3,
  "firewall_enabled": true,
  "firewall_mode": "block",
  "file_integrity": true,
  "security_headers": true,
  "traffic_logging": true,
  "email_notifications": false,
  "notification_email": ""
}

Database Tables

Shield auto-creates these tables on first use:

  • shield_login_attempts — tracks login successes and failures by IP
  • shield_blocked_ips — blocked IPs with reason, type (manual/auto), and optional expiry
  • shield_log — security events (firewall hits, lockouts, blocked requests)
  • shield_file_hashes — MD5 hashes of core PHP files for integrity checks
  • shield_traffic — recent request log for traffic monitoring
