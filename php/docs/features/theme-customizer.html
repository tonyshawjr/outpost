Theme Customizer

Change your site's colors, fonts, logo, and favicon from a visual editor with a live preview — no code required. The customizer reads its configuration from the active theme's theme.json and stores overrides separately so your theme files are never modified.

How it works

The Theme Customizer is available at Admin → Customize (in the Build section of the sidebar) or via the Customize button on the active theme card in Themes. It requires the settings.* capability (admin or developer role).

Themes declare which properties are customizable by adding a customizer key to their theme.json. The admin panel reads this schema and builds the editor UI dynamically. Themes without a customizer key display a message saying customization is not yet supported.

What you can customize

The available options depend on what the theme declares. The built-in Personal theme supports:

  • Colors — accent, accent hover, text, background, surface, and muted text
  • Typography — heading font and body font (from a curated list of Google Fonts)
  • Site Identity — logo image and favicon

Theme developers can add any combination of color, font, and image fields to their themes.

Live preview

The right side of the customizer shows a live preview of your site in an iframe. As you change colors, fonts, or images, the preview updates instantly — no page reload required. This uses CSS custom property updates via postMessage between the admin panel and the preview iframe.

Note — The live preview only reflects CSS variable changes. Some visual changes (like a new logo replacing a text element) may require saving and refreshing the preview to see the full effect.

Saving and resetting

Click Save to persist your changes. Customizer values are saved to content/data/customizer.json and the page cache is cleared automatically so visitors see the new styles immediately.

Click Reset to discard all customizations and revert to the theme's default values. This deletes the saved values for the current theme only — other themes' customizations are unaffected.

Export and import presets

You can export your current customizer values as a JSON file and import them on another site running the same theme. This is useful for:

  • Backing up a color scheme before experimenting
  • Sharing a branded look across multiple sites
  • Distributing presets alongside a theme

Use the Export button to download a preset file, and Import to upload and apply one.

How customizer CSS is injected

When a visitor loads your site, Outpost's template engine checks for saved customizer values. If any exist, it injects a <style> tag before </head> that overrides the theme's CSS custom properties:

<style id="_outpost_customizer">
:root:not([data-theme="dark"]) {
  --color-accent: #E11D48;
  --color-ink: #1A1A2E;
  --font-display: 'Playfair Display', serif;
  --font-sans: 'Source Sans 3', sans-serif;
}
</style>

The :root:not([data-theme="dark"]) selector ensures customizations only apply to light mode. Dark mode continues to use the theme's built-in defaults.

Google Fonts

If custom fonts are selected, Outpost injects the appropriate Google Fonts <link> tags and removes any hardcoded Google Fonts links from the theme's output to avoid loading unnecessary fonts.

Favicon

If a custom favicon is set, Outpost injects a <link rel="icon"> tag before </head>.

Logo

When a logo is set via the customizer, it is automatically synced to the @site_logo global field. This means your theme's existing {{ @site_logo | image }} tags work without any changes.

Tip — Customizer values are stored per-theme. If you switch themes, each theme retains its own customizer values independently. Switching back restores the previously saved values.

Storage

Customizer values are stored in content/data/customizer.json, keyed by theme slug:

{
  "personal": {
    "colors": {
      "color-accent": "#E11D48",
      "color-ink": "#1A1A2E"
    },
    "typography": {
      "font-heading": "Playfair Display",
      "font-body": "Source Sans 3"
    },
    "identity": {
      "logo": "/outpost/uploads/logo.png"
    }
  }
}

This file lives inside content/, so it is included in backups and safe from the auto-updater.

For theme developers

To add customizer support to your theme, add a customizer key to your theme.json. The customizer key contains a sections array — each section becomes a tab in the editor UI.

Schema structure

{
  "name": "My Theme",
  "version": "1.0.0",
  "author": "Your Name",
  "description": "A customizable theme.",
  "customizer": {
    "sections": [
      {
        "id": "colors",
        "label": "Colors",
        "fields": [
          {
            "key": "color-accent",
            "label": "Accent Color",
            "type": "color",
            "default": "#14B8A6",
            "var": "--color-accent"
          }
        ]
      },
      {
        "id": "typography",
        "label": "Typography",
        "fields": [
          {
            "key": "font-heading",
            "label": "Heading Font",
            "type": "font",
            "default": "Inter",
            "var": "--font-display"
          }
        ]
      },
      {
        "id": "identity",
        "label": "Site Identity",
        "fields": [
          {
            "key": "logo",
            "label": "Site Logo",
            "type": "image"
          }
        ]
      }
    ]
  }
}

Section fields

PropertyTypeRequiredDescription
idstringYesUnique section identifier. Used as the key in the saved JSON.
labelstringYesDisplay name shown as a tab in the customizer UI.
fieldsarrayYesArray of field definitions (see below).

Field properties

PropertyTypeRequiredDescription
keystringYesUnique field identifier within the section.
labelstringYesDisplay label shown next to the input.
typestringYesField type: color, font, or image.
defaultstringNoDefault value. For colors, a hex string. For fonts, the font family name. Not used for images.
varstringNoCSS custom property name to override (e.g. --color-accent). Required for color and font types. Not used for image type.

Field types

color

Renders a color picker with a hex text input. The value is injected as a CSS custom property override using the var name. Always provide a default hex value so the reset button works.

{
  "key": "color-accent",
  "label": "Accent Color",
  "type": "color",
  "default": "#14B8A6",
  "var": "--color-accent"
}

font

Renders a dropdown with a curated list of Google Fonts plus a "System Default" option. The selected font is loaded via a Google Fonts <link> tag and injected as a CSS custom property. Provide a default font name and the var to override.

{
  "key": "font-heading",
  "label": "Heading Font",
  "type": "font",
  "default": "Inter",
  "var": "--font-display"
}

image

Renders a thumbnail preview with a media picker button. Used for logos and favicons. Image fields do not use var or default — they store the image URL path directly. The logo key has special behavior: its value is synced to the @site_logo global field on save.

{
  "key": "logo",
  "label": "Site Logo",
  "type": "image"
}

CSS custom property requirements

For the customizer to work, your theme's CSS must use CSS custom properties (variables) for the values you want to be customizable. Define them on :root in your stylesheet:

:root {
  --color-accent: #14B8A6;
  --color-accent-hover: #0D9488;
  --color-ink: #27272A;
  --color-body: #FAFAFA;
  --color-paper: #FFFFFF;
  --color-muted: #52525B;
  --font-display: 'Inter', sans-serif;
  --font-sans: 'Inter', sans-serif;
}

Then use var(--color-accent) etc. throughout your CSS. When the customizer overrides a variable, all elements using it update automatically.

Important — If your CSS uses hardcoded color values instead of custom properties, the customizer cannot override them. Always use var(--variable-name) for any value you want to be customizable.

Dark mode considerations

Customizer overrides use the selector :root:not([data-theme="dark"]), which means they only apply to light mode. If your theme supports dark mode via a [data-theme="dark"] attribute, the dark mode colors remain unchanged by the customizer. This is intentional — it ensures a user's light mode customizations do not break the dark mode palette.
