Stock Photo Search

Search royalty-free photos from Pexels and Unsplash without leaving the visual builder. Pick an image node, search, and insert — Pexels photos are saved to your media library, Unsplash photos are hotlinked from their CDN.

Overview

Stock Photo Search adds a photo picker to the visual builder’s image-node inspector. It talks to two providers — Pexels and Unsplash — through a small PHP file (stock-photos.php) that keeps your API keys server-side and encrypted. You bring a free API key for either (or both) providers; Outpost handles search, pagination, and inserting the chosen photo.

The two providers behave differently by design, because their licenses require it: Pexels photos are downloaded and self-hosted in your media library, while Unsplash photos are embedded by CDN URL and never copied locally. That difference is the most important thing to understand on this page.

Zero extra services. No proxy, no third-party account beyond the provider key itself. Keys are encrypted with Ranger’s crypto and stored per-provider in the settings table. The picker never sees your key — all provider calls happen on the server.

Where to find it

  1. Open a page in the Visual Builder and select an image node.
  2. In the image inspector, click Search stock photos.
  3. If more than one provider is configured, use the Pexels / Unsplash toggle in the dialog header to choose a source.
  4. Type a query and press Search. Results load as a grid; each tile shows the photographer’s name as a credit overlay. Use Load more to page through results.
  5. Click a photo to insert it. The image node’s src (and alt, when available) update immediately.

If no provider is connected yet, the dialog shows an empty state pointing you to Settings → Integrations.

Adding API keys

Go to Settings → Integrations → Stock Photos and paste a key for either provider. Both are free to obtain:

ProviderWhere to get a keyFree tier
Pexelspexels.com/api200 requests/hour
Unsplashunsplash.com/developersDemo apps allow 50 requests/hour

You only need one to start — the provider toggle in the picker only appears when both are configured. Keys are write-through encrypted on save (ranger_encrypt) and decrypted on demand server-side; the settings form re-displays the stored key so you can confirm or rotate it.

Pexels vs. Unsplash: the key difference

The two providers are inserted in fundamentally different ways. This is not a preference — it follows each provider’s API guidelines.

PexelsUnsplash
How it’s insertedDownloaded server-side and saved to your media libraryHotlinked directly from the Unsplash CDN
Stored in uploads/?Yes — self-hostedNo — served from images.unsplash.com
Media library recordCreated (returns a media_id)None
ProcessingResized to the max width and auto-converted to WebP by OutpostMedia::importFromFile()Served at request time with CDN sizing params (w=1600&q=80&fm=jpg&fit=max)
Extra requirementNoneA download_location trigger is fired on the Unsplash API when the photo is used (required by their guidelines)
Insert result typemediahotlink

The builder reflects this: inserting a Pexels photo shows a “Photo added to media” toast, while an Unsplash photo shows “Photo added (hotlinked)”.

Unsplash images are not on your server. A hotlinked photo depends on the Unsplash CDN staying reachable and the image staying live. It does not appear in your Media library and is not covered by your own backups. If you need a photo to be permanently yours, prefer Pexels — or download it and upload it manually.

Attribution

Both Pexels and Unsplash are free to use without attribution being legally required, but crediting the photographer is expected and appreciated — and required by Unsplash’s guidelines in spirit. The picker surfaces the credit you need:

  • Every result tile shows the photographer’s name as an overlay.
  • The picker footer reminds you to credit the photographer and links to both providers.
  • The insert response includes a credit object — author, author_url, provider, and provider_url — so credit data travels with the photo.
Coming soon. A fuller on-page / credits treatment (rendering attribution automatically near the image or on a credits page) is planned. For now, use the photographer name shown in the picker and the credit data returned on insert.

Endpoints

All routes are on the admin API (api.php?action=…). Search and import require the content.* capability; reading and writing keys require settings.*.

RouteMethodDescription
stock/providersGETWhich providers are configured. Returns { providers: [{ id, configured }] }.
stock/searchGETSearch a provider. Params: provider (pexels|unsplash, default pexels), q (query, capped at 120 chars), page (1–100). Returns { results, page, has_more }, 24 results per page.
stock/importPOSTInsert a photo. Body: { provider, id, alt } (alt capped at 500 chars). Returns the insert result (see below).
stock/settingsGET / PUTRead or write the per-provider API keys. GET returns decrypted keys; PUT accepts stock_api_key_pexels and/or stock_api_key_unsplash and stores them encrypted.

Security posture

Because import fetches a remote URL server-side, the download path is hardened against SSRF and abuse:

  • Host allow-list — downloads are only permitted from images.pexels.com and images.unsplash.com. Any other host is rejected as bad_source.
  • HTTPS only — the download curl handle restricts protocols to HTTPS; plain-HTTP URLs are refused.
  • No redirects — CURLOPT_FOLLOWLOCATION is disabled so a response can’t bounce the fetch to an internal address.
  • Size & type caps — downloads are capped at the upload size limit and the response content-type must start with image/.
  • Strict photo-id validation — Pexels ids must be all digits; Unsplash ids must match [A-Za-z0-9_-]{5,32}. Invalid references are rejected before any network call.
  • Encrypted keys — keys are stored encrypted (Ranger crypto), never exposed to the browser, and used only in server-side provider requests.
