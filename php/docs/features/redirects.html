Redirects

Manage URL redirects from the admin panel. Keep links working when pages move, consolidate old URLs, and track redirect usage — all without touching server config.

Overview

Redirects let you point old or changed URLs to new destinations. Every front-end request is checked against your redirect rules before page routing, so visitors (and search engines) are seamlessly forwarded.

Access redirects from the sidebar under Tools → Redirects. Requires the settings.* capability (admin role or higher).

Redirect Types

Code Name When to Use
301 Permanent The old URL is gone forever. Search engines transfer link equity to the new URL. Default.
302 Temporary The old URL will come back. Search engines keep the original indexed.
307 Temporary (preserve method) Like 302, but the browser must use the same HTTP method (POST stays POST). Useful for API or form endpoints.

Match Types

Outpost evaluates redirect rules in this order: exact match first, then wildcard, then regex. The first match wins.

Exact Match

The source URL must match the request path exactly (trailing slashes are normalized).

SourceTargetMatches
/old-page/new-page/old-page and /old-page/
/blog/my-post/articles/my-post/blog/my-post only

Wildcard Match

End the source URL with * to match any path that starts with the prefix. The matched remainder is appended to the target.

SourceTargetExample
/old-blog/*/blog/*/old-blog/my-post → /blog/my-post
/docs/*/help/*/docs/getting-started → /help/getting-started

Regex Match

Prefix the source URL with ~ to use a PCRE regular expression. Capture groups ($1, $2, etc.) can be used in the target URL.

SourceTargetExample
~^/post/(\d+)$/archive/$1/post/42 → /archive/42
~^/(\w+)/(\w+)$/$2/$1/category/item → /item/category
Note — Regex patterns are standard PCRE. The leading ~ is stripped before matching. Invalid patterns are silently skipped.

Adding a Redirect

  1. Navigate to Tools → Redirects
  2. Click Add Redirect
  3. Enter the Source URL (the old path)
  4. Enter the Target URL (where to send visitors)
  5. Choose the redirect Type (301, 302, or 307)
  6. Optionally add Notes for your reference
  7. Click Create

CSV Import

Bulk-import redirects by pasting CSV data. Click Import in the toolbar.

Format: one redirect per line with columns source, target, type. The type column is optional (defaults to 301). A header row is auto-detected and skipped if it contains the words "source" or "from".

/old-page,/new-page,301
/blog/*,/articles/*,301
/about-us,/about,302

Duplicate source URLs are silently skipped. The import reports how many were imported and how many were skipped.

Hit Tracking

Every time a redirect fires, the hit counter increments and the last-hit timestamp updates. This helps you identify which redirects are still in use and which can be cleaned up.

Testing Redirects

Click Test URL in the toolbar to check how a URL would be handled without actually navigating. Enter a path and the system reports whether it matches (and which rule), the match type (exact, wildcard, or regex), and the resolved target URL.

Enable / Disable

Each redirect can be toggled on or off without deleting it. Disabled redirects appear dimmed in the list and are not evaluated during front-end requests.

Security

Redirect targets are validated at execution time. Only relative paths and same-domain absolute URLs are allowed. External URLs are blocked to prevent open-redirect attacks. Protocol-relative URLs (//) are also blocked.

All redirect management endpoints require the settings.* capability, restricting access to admin-level users.

How It Works Internally

The redirect check runs in front-router.php after the config and database are loaded but before page routing. This ensures redirects take priority over page or collection routes. The redirects table is created automatically on first use (no migration step needed).

The evaluation order is: exact match → wildcard match → regex match. Only active rules are considered. On match, the response is sent immediately with the configured status code and Location header.
