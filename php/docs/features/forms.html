Forms

Build forms visually in the admin, or hand-code HTML forms — either way, submissions are stored, emailed, and viewable in the admin panel with no backend code required.

Visual Form Builder

Navigate to Admin → Forms to create forms visually. The drag-and-drop builder supports 14 field types and renders forms via a single template tag.

Creating a form

  1. Click New Form and enter a name and slug
  2. Add fields from the palette on the left — drag to reorder
  3. Click any field to edit its settings (label, name, required, placeholder, choices, etc.)
  4. Switch to Settings to configure submit button text, honeypot spam protection, and confirmation behaviour
  5. Switch to Preview to see a live rendering of your form
  6. Click Save

Rendering in templates

Add this template tag wherever you want the form to appear:

{% form 'contact' %}

This renders a complete <form> element with all fields, labels, validation attributes, honeypot, and a submit button. No JavaScript is required — the form submits via standard POST.

Styling builder forms

Builder forms output semantic HTML with BEM-style class names:

.outpost-form { /* form wrapper */ }
.outpost-form--contact { /* form-specific, uses the slug */ }
.outpost-field { /* field wrapper */ }
.outpost-field--text { /* type-specific field wrapper */ }
.outpost-field label { /* field label */ }
.outpost-field input,
.outpost-field textarea,
.outpost-field select { /* inputs */ }
.outpost-required { /* required asterisk */ }
.outpost-field-desc { /* help text below field */ }
.outpost-form-submit { /* submit button */ }
.outpost-section-title { /* section heading */ }
.outpost-radio-group, .outpost-checkbox-group { /* choice groups */ }

Honeypot spam protection

When enabled (default), the builder injects a hidden text field that is invisible to real users but filled in by bots. If the field has a value on submission, the form handler silently redirects as if the submission succeeded — bots never know they were caught.

URL pre-population

Form fields auto-fill from query parameters. For example, if your form has a field named email, visiting /contact?email=user@example.com will pre-fill that field.

Enhanced submissions inbox

Navigate to Admin → Submissions to view all form submissions. Features include:

  • Star important submissions for quick reference
  • Notes — add internal notes to any submission
  • Bulk actions — mark read, star, archive, or delete multiple submissions at once
  • Status filtering — view active, archived, or all submissions
  • Form filtering — filter by specific form or view all
  • Field labels — builder form submissions display human-readable field labels
  • Export to CSV

Manual HTML Forms

You can also hand-code forms in your templates without using the builder. This is useful for simple forms or forms with custom HTML structure.

How it works

Outpost's form handler lives at /outpost/form.php. Set your HTML form's action to that path and method to POST. Include a few hidden fields to tell Outpost where to redirect after submission and which form identifier to file the submission under. Everything else is stored automatically.

Basic form

A minimal contact form looks like this:

<form action="/outpost/form.php" method="POST">
  <input type="hidden" name="_form" value="contact">
  <input type="hidden" name="_redirect" value="/contact?sent=1">

  <input type="text" name="name" placeholder="Your name" required>
  <input type="email" name="email" placeholder="Email" required>
  <textarea name="message" placeholder="Message" required></textarea>

  <button type="submit">Send</button>
</form>
Note — The _form value becomes the form's identifier in the admin. Use alphanumeric characters, hyphens, and underscores only (e.g. contact, quote-request, newsletter_signup).

Reserved fields

These fields are processed by form.php and stripped from the stored submission data. All other fields are stored exactly as submitted.

Field Required? Purpose
_form Yes Form identifier — alphanumeric, hyphens, underscores only
_redirect Yes URL to redirect to after a successful submission
_notify No Override the notification email for this specific submission
g-recaptcha-response No reCAPTCHA token — validated automatically if a secret key is configured
_form_id No Builder form ID — auto-injected by {% form %} tag
_hp_* No Honeypot fields — auto-injected by builder forms

After submission

On success, form.php redirects the visitor to your _redirect URL with ?submitted=1 appended. Use that query parameter to show a thank-you message.

Showing a success message

In a Liquid template, detect the query parameter and show a confirmation:

<!-- In your template -->
<div id="form-success" style="display:none">
  Thank you! We'll be in touch.
</div>

<script>
  if (new URLSearchParams(window.location.search).get('submitted') === '1') {
    document.getElementById('form-success').style.display = 'block';
  }
</script>
Tip — You can also hide the form itself when ?submitted=1 is present, so the visitor doesn't accidentally submit again.

Error handling

If submission fails, the visitor is redirected back with ?form_error=CODE added to the URL.

Code Meaning
missing_form_name The _form field is absent or contains invalid characters
rate_limited The submitting IP has exceeded 5 submissions in 60 seconds
captcha_failed reCAPTCHA validation failed or the token was missing
required_field_missing A required field defined in the form builder schema was empty

Read and display these errors in your page:

<div id="form-error" style="display:none"></div>

<script>
  const params = new URLSearchParams(window.location.search);
  const errorMap = {
    missing_form_name: 'Form configuration error. Please contact the site owner.',
    rate_limited:      'Please wait a moment before submitting again.',
    captcha_failed:    'Security check failed. Please try again.',
  };
  const code = params.get('form_error');
  if (code && errorMap[code]) {
    const el = document.getElementById('form-error');
    el.textContent = errorMap[code];
    el.style.display = 'block';
  }
</script>
Warning — Always display error messages to users. Silent failures leave visitors confused about whether their message was sent.

Email notifications

Configure a notification email address in Admin → Forms → select a form → Notification Email. Outpost sends an email to that address every time a new submission arrives for that form.

You can also set a global fallback address in Admin → Settings → Forms → Default Notification Email. Forms without their own address fall back to this value.

Per-submission override

Include a _notify hidden field to route a specific form's notifications to a different address — useful for multi-department forms:

<!-- Route submissions to the sales team -->
<input type="hidden" name="_notify" value="sales@example.com">
Note — Email delivery requires a working PHP mail() setup on your server, or an SMTP plugin. Outpost does not bundle an SMTP library.

Viewing submissions

Navigate to Admin → Forms. Forms appear automatically after their first submission — there is nothing to create in advance. For each form you can:

  • View all submissions with field values, timestamp, and read/unread status
  • Mark individual submissions as read or unread
  • Delete individual submissions
  • Export all submissions to CSV

Exporting submissions

Click the Export button on any form's submissions page to download a CSV file containing all submissions for that form. The CSV includes all submitted field names as columns, plus a timestamp column.

reCAPTCHA

Outpost supports Google reCAPTCHA v2 ("I'm not a robot" checkbox) to reduce spam.

Setup

  1. Register your site at google.com/recaptcha to get a site key and a secret key.
  2. Enter the secret key in Admin → Settings → reCAPTCHA Secret Key.
  3. Add the reCAPTCHA widget to your form:
<!-- Load the reCAPTCHA script once, in your <head> or before the form -->
<script src="https://www.google.com/recaptcha/api.js"></script>

<form action="/outpost/form.php" method="POST">
  <input type="hidden" name="_form" value="contact">
  <input type="hidden" name="_redirect" value="/contact?sent=1">

  <input type="text" name="name" required>
  <input type="email" name="email" required>
  <textarea name="message" required></textarea>

  <div class="g-recaptcha" data-sitekey="YOUR_SITE_KEY"></div>

  <button type="submit">Send</button>
</form>

When a secret key is configured, form.php validates the g-recaptcha-response token on every submission. Submissions without a valid token are rejected with ?form_error=captcha_failed.

Tip — If no secret key is configured in Settings, reCAPTCHA validation is skipped entirely — useful during local development.

Rate limiting

Outpost enforces a limit of 5 submissions per IP address per 60 seconds across all forms. This limit is not configurable. Submissions that exceed the limit are rejected immediately with ?form_error=rate_limited and the visitor is redirected back to the form.

Note — Rate limiting is per IP, not per form. A single IP submitting 3 different forms in 60 seconds will be allowed; submitting the same form 6 times in 60 seconds will be blocked on the 6th attempt.

Multi-step forms

Outpost's form handler does not natively support multi-step forms. The recommended approach is to build multi-step behaviour entirely in JavaScript — show and hide sections of the form, validate each step client-side, and then submit the entire form as a single POST when the user reaches the final step.

<form action="/outpost/form.php" method="POST" id="multistep">
  <input type="hidden" name="_form" value="quote">
  <input type="hidden" name="_redirect" value="/quote?sent=1">

  <!-- Step 1 -->
  <div class="step" id="step-1">
    <input type="text" name="company" placeholder="Company name">
    <button type="button" onclick="showStep(2)">Next</button>
  </div>

  <!-- Step 2 -->
  <div class="step" id="step-2" style="display:none">
    <textarea name="details" placeholder="Project details"></textarea>
    <button type="submit">Submit</button>
  </div>
</form>

File uploads

File uploads are not supported by form.php. If your use case requires file attachments, implement a custom PHP endpoint to handle multipart/form-data submissions and store or email the uploaded files as needed.
