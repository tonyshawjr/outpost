Section Import & AI Generation

Paste a self-contained section of HTML, CSS, and JavaScript — or describe one and let AI write it — and Outpost explodes the markup into editable builder elements, merges the styles into your site stylesheet, and drops the section onto the canvas.

Overview

The Visual Builder toolbar has an Import button that opens the Section Import modal. Import takes three separate parts — HTML, CSS, and JavaScript — and turns them into a live section:

  • HTML is “exploded” into the builder’s node tree: every supported tag becomes an individually editable element.
  • CSS is parsed into named style classes and merged into the site stylesheet, so the section keeps its look and stays editable in the style panel.
  • JavaScript is appended to the page’s script file ({slug}.js) and runs on the published page only — not inside the builder canvas.

The new section is inserted at the currently selected container, or at the page root if nothing is selected.

Static except the holes. Imported markup bakes to plain static HTML. Any element you tag with data-outpost becomes an editable, server-rendered field — the rest of the section is fixed structure. This is the same node model the rest of the builder uses.

Where to find it

  1. Open any page in the Visual Builder.
  2. (Optional) Select the container you want the section to land inside. With nothing selected, the section is inserted at the page root.
  3. Click Import in the builder toolbar.
  4. Paste your HTML, CSS, and JavaScript into the three tabs — or use the Generate with AI bar at the top to have them written for you.
  5. Click Import section. The markup explodes onto the canvas, the styles merge in, and the JavaScript is written to the page’s script file.

Import-ready HTML

The importer is deliberately strict: only a fixed set of tags survive, and styling is class-only. Producing markup that follows these rules is what lets Outpost turn it into clean, editable nodes instead of an opaque blob.

Supported tags

One top-level element wraps the section — it becomes the section root. Only these tags survive the import; everything else is transformed or dropped.

RoleTagsNotes
Containers (hold children)div, section, main, header, footer, article, aside, nav, ul, ol, li, figureChildren are recursed into individual nodes.
Text (content only)p, span, h1–h6, strong, em, small, blockquote, labelText is taken whole — inner markup is flattened (see below).
MediaimgKeeps src and alt.
Interactivea, buttona keeps href; text is taken whole.
  • Any other tag (table, svg, input, select, form, video, iframe, web components…) collapses to a plain div and loses its attributes.
  • script, style, link, meta, br, and hr are dropped entirely. Put behavior in the JS part and decoration in the CSS part.
  • Inline style="" attributes are discarded on import. Style only with the class attribute.
Text elements flatten inner markup. A text element’s text is taken whole — do not nest elements inside it. <p>Save <strong>50%</strong></p> flattens to the plain text “Save 50%”. To style part of a line, make the pieces siblings inside a container — e.g. a <div> holding a <span> and a <span class="accent"> — not children of one <p>.

Dynamic holes

Add data-outpost="snake_case_name" to any text, image, link, or button element to make its content an editable, server-rendered field. The page bakes to static HTML except these holes, which fill from managed content at request time — editing a hole updates the live page with no rebuild.

AttributePurpose
data-outpost="name"Marks the element as an editable field. Name is normalized to snake_case letters, numbers, and underscores.
data-type="text|richtext|image|url"Optional. Declares how the field is edited and rendered.
data-scope="global"Optional. Makes it a site-wide field shared across every page (e.g. a shared CTA or contact number).

Use holes for anything editable or data-driven — hero headline, subhead, price, CTA label, hero image. Leave purely decorative or structural text as plain nodes.

CSS rules

The CSS parser maps single-class selectors onto the builder’s style model. It supports more than a flat list of classes, but the selector shapes it understands are limited.

SupportedIgnored (never depend on)
Single-class base rules: .hero-title { … }Descendant / child combinators: .a .b, .a > .b
Comma groups of single classes: .a, .b { … }Compound / multi-class selectors: .a.b
Pseudo-classes / elements on one class: .btn:hover, .card::beforeElement, id, and attribute-only selectors: div, #id, [data-x]
@media, @container, @supports blocks whose inner rules are single-class rules@keyframes, @import

Give every element its own class and style it by that class. Prefer CSS variables (design tokens) for color and spacing so the section stays on-brand. Values containing expression(), url(javascript:), or behavior are rejected. BEM class names are encouraged (.hero, .hero__title, .hero__cta--ghost).

Why single-class only? Hover states and responsive rules survive because they map onto the builder’s nested-style model — a hover or @media block becomes a nested variant of the class, not a separate selector. Descendant and compound selectors have no place in that per-element model, so they are dropped rather than silently half-applied.

JavaScript

  • Vanilla JS only — no build step, no framework.
  • It runs on the published page only, not in the builder canvas, and is appended to the page’s script file.
  • Scope every query to the section’s own classes so it can’t touch the rest of the page, and guard for elements that may not be present.

Worked example

A call-to-action section with an editable eyebrow, headline, and button link (three dynamic holes), a :hover lift on the button, and a responsive @media rule.

HTML

<section class="cta">
  <p class="cta__eyebrow" data-outpost="cta_eyebrow">Limited beta</p>
  <h2 class="cta__title" data-outpost="cta_title">Ship pages before lunch</h2>
  <a class="cta__btn" href="/signup" data-outpost="cta_href">Start free</a>
</section>

CSS

.cta { padding: var(--space-xl, 4rem) 1.5rem; text-align: center; background: var(--surface, #0b0b12); color: #fff; }
.cta__title { font-size: clamp(1.75rem, 4vw, 2.75rem); margin: 0 0 1rem; }
.cta__btn { display: inline-block; padding: 0.8rem 1.6rem; border-radius: 10px; background: var(--primary, #6d5efc); color: #fff; text-decoration: none; }
.cta__btn:hover { transform: translateY(-2px); }
@media (max-width: 600px) { .cta { padding: 2.5rem 1rem; } }

JavaScript

document.querySelectorAll('.cta__btn').forEach((b) => b.addEventListener('click', () => {}));

After import: the <section> becomes the section root; the eyebrow, title, and button are editable fields (edit their text or link inline, and the live page updates with no rebuild); .cta, .cta__title, and .cta__btn (with its hover and responsive variants) are added to the site stylesheet; and the click handler is written to the page’s {slug}.js to run on the published page.

Generate with AI

At the top of the modal is a Generate with AI bar. Describe the section you want — for example, “a pricing CTA with three tiers” — and the model fills the three panes with import-ready HTML, CSS, and JavaScript that you can review and edit before importing.

  • The model is instructed with the exact import-ready spec above, so its output is built to explode cleanly into editable nodes.
  • If your site defines design tokens, they are passed to the model so it prefers your CSS variables and stays on-brand.
  • Generation streams into the panes; you can stop it mid-flight. Nothing is imported until you click Import section.
Bring your own key. Generation uses the AI provider and key configured in Settings → Integrations (the same keys Ranger uses). The model is self-selected for the chosen provider. If no key is configured, generation returns a message asking you to add one. Usage and cost are reported when generation finishes.

For external tools (MCP)

The same import-ready spec is exposed as an MCP resource so an external LLM — Claude Code, Claude Desktop, or any MCP client — can fetch it and emit paste-ready sections directly.

URIDescription
outpost://import/guideThe full import-ready HTML/CSS/JS spec: supported tag set, class-only styling with hover + @media, dynamic holes (data-outpost), the parser limits, and a worked example.

An external tool reads this resource, generates HTML/CSS/JS that follows it, and you paste the three parts into the Import modal. See the MCP Server page for connection setup.

Limits & safeguards

  • Size caps — HTML is capped at 500,000 characters; CSS and JS at 200,000 each. AI prompts are capped at 8,000 characters.
  • PHP stripped — PHP tags are removed from imported HTML and JS before anything is stored or written.
  • URL sanitization — javascript:, data:, and vbscript: URLs are blocked in href/src and field values.
  • Tree limits — parsing stops at a depth of 100 or 5,000 nodes; dangling child references and cycles are broken during validation.
  • JS target — JavaScript is only written for real pages with a simple single-segment slug; it is never written for the globals owner.
