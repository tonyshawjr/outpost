Auto-Updater

Check for and apply core updates directly from the admin panel. No SSH, no FTP — one click.

How It Works

Outpost includes a built-in updater at Settings → Updates that lets administrators check for new versions and apply them without leaving the browser. The entire flow is self-contained — no command-line access, FTP clients, or manual file copying required.

  1. On every admin page load, the auth/me API response includes update_available and latest_version for admin and super_admin users. This uses the same 5-minute cache as the Updates page — no extra API calls.
  2. If an update is available, a green dot badge appears on Settings in the sidebar and mobile navigation
  3. Admin visits Settings → Updates in the admin panel
  4. The UI calls GET api.php?action=updates/check, which fetches the latest GitHub Release from the Outpost repository
  5. Outpost compares the installed OUTPOST_VERSION (from config.php) against the release tag using version_compare()
  6. If a newer version is available, the UI displays the release notes and an update button
  7. Admin clicks “Update now” — the UI calls POST api.php?action=updates/apply with the zip download URL
  8. PHP downloads the zip, validates the URL (must point to github.com/tonyshawjr/outpost), and validates all zip entry paths to prevent path traversal attacks
  9. Extracts the archive and copies only core files to the installation directory, then clears the template cache
  10. The admin SPA reloads automatically on the new version
Note — The update check result is cached for 5 minutes. Both the auth/me auto-check and the Updates page share the same cache, so GitHub is hit at most once per 5 minutes regardless of how often the admin panel is loaded.

What Gets Updated

When an update is applied, Outpost replaces only the core system files. Everything that makes your site unique — your theme, your content, your uploads — is left untouched.

Files that are replaced

Path Description
*.php (outpost root) All PHP files in the outpost root directory — api.php, engine.php, config.php, front-router.php, template-engine.php, etc.
admin/ The entire admin SPA — index.html, JavaScript bundles, and CSS assets
docs/ Developer documentation
member-pages/ Built-in member page templates (login, register, password reset)
tools/ Utility scripts and diagnostic tools

Files that are never touched

Path Why it is protected
content/ The entire user data directory is skipped. This includes your database (content/data/cms.db), uploaded media (content/uploads/), themes (content/themes/), and backups (content/backups/).
cache/ Compiled template cache. This directory is cleared after an update so templates are recompiled with the latest engine changes, but it rebuilds automatically on the next page load.
Tip — Because the entire content/ directory is protected, you can safely customize any theme file in a non-managed theme. Your changes will persist through every update. For managed themes, the updater uses conflict detection to preserve your edits — see below.

Theme Update System

Starting in v1.9.0, Outpost can update bundled themes alongside the CMS core. This keeps official themes in sync with engine changes, new features, and bug fixes — without overwriting your customizations.

Managed themes

A managed theme is any theme with "managed": true in its theme.json. This flag tells the updater that the theme is maintained by Outpost and should be kept up to date. The bundled Personal and Starter themes ship with this flag set.

Managed themes display a “Managed” badge on their card in Admin → Themes. User-created themes and duplicated themes never have this flag and are never touched by the updater.

How theme updates work

When you apply a CMS update, the update package may include updated versions of managed themes in its content/themes/ directory. The updater processes each one:

  1. New theme: If a managed theme in the package does not exist on the site, it is installed automatically as a fresh copy.
  2. Version check: If the theme already exists and is managed, the updater compares the package version against the installed version. If the installed version is the same or newer, the theme is skipped.
  3. Conflict detection: If an update is needed, the updater compares each file against the .outpost-manifest.json to determine which files the user has modified. See below for details.

The .outpost-manifest.json file

Every managed theme includes a hidden .outpost-manifest.json file at its root. This file contains an MD5 hash of every file as it was shipped:

{
  "index.html": "a1b2c3d4e5f6...",
  "post.html": "f6e5d4c3b2a1...",
  "assets/style.css": "1234abcd5678...",
  "partials/head.html": "abcd1234efgh..."
}

During an update, the updater uses these hashes to determine:

  • Unmodified files (installed hash matches manifest hash) → safely replaced with the new version
  • User-modified files where the update did not change the file → kept as-is (no conflict)
  • User-modified files where the update also changed the file → preserved as a conflict. The user’s version is kept, and the file is reported in the update results.
  • New files (exist in the update but not on the site) → copied in
Note — After a theme update, the new manifest replaces the old one. This means future updates will compare against the latest shipped version, not the original install.

What users see after an update

The update results screen in the admin panel shows a Theme Updates section when themes were processed. Each theme shows one of:

StatusMeaning
Installed A new managed theme was added to the site for the first time.
Updated The theme was updated from one version to another. Shows the version range (e.g. 1.0.0 → 2.0.0).
Updated with conflicts The theme was updated, but some files you edited were preserved instead of replaced. The conflicting file paths are listed so you can manually merge changes if needed.
Skipped The installed version is already up to date.

Duplicating a managed theme

If you want to customize a managed theme, duplicate it first. The duplicate has the managed flag stripped and becomes a fully independent theme. It will never be touched by future updates, giving you complete control over every file.

The original managed theme stays on your site and continues receiving updates. You can switch between the original (auto-updated) and your custom duplicate at any time.

Tip — For small CSS tweaks to a managed theme, consider using the Theme Customizer instead of editing files directly. Customizer values are stored separately and survive all updates without conflicts.

Database Migrations

After an update, your database schema may need changes — new columns, new tables, or modified indexes. Outpost handles this automatically.

On the next API request after an update, the bootstrap process checks for missing columns and tables using PRAGMA table_info and applies any necessary ALTER TABLE ADD COLUMN statements. No manual intervention is needed.

Migrations are idempotent — they check whether each change has already been applied before running, so they are safe to run multiple times.

Note — If a new version introduces schema changes, the first API request after the update may take slightly longer as migrations run. Subsequent requests return to normal speed.

API Endpoints

The auto-updater is powered by two API endpoints. Both require super_admin role and session authentication with a valid CSRF token.

Check for updates

GET api.php?action=updates/check

Checks the GitHub Releases API for the latest version and compares it against the installed version.

Requires: super_admin role

Response:

{
  "available": true,
  "current_version": "1.0.0-beta.12",
  "latest_version": "1.0.0-beta.13",
  "release_notes": "### Fixed\n- Fix package script...",
  "download_url": "https://github.com/tonyshawjr/outpost/releases/download/v1.0.0-beta.13/outpost-v1.0.0-beta.13.zip"
}

When no update is available:

{
  "available": false,
  "current_version": "1.0.0-beta.13",
  "latest_version": "1.0.0-beta.13"
}

The response is cached for 1 hour. Subsequent calls within that window return the cached result without hitting the GitHub API.

Apply update

POST api.php?action=updates/apply

Downloads and installs the specified update package.

Requires: super_admin role

Request body:

{
  "url": "https://github.com/tonyshawjr/outpost/releases/download/v1.0.0-beta.13/outpost-v1.0.0-beta.13.zip"
}

Validation:

  • The URL must use HTTPS
  • The hostname must be github.com
  • The path must begin with /tonyshawjr/outpost/releases
  • All zip entry paths are validated before extraction to prevent zip slip attacks

Response (success):

{
  "success": true,
  "message": "Update applied successfully"
}

Response (failure):

{
  "error": "Invalid download URL"
}

Security

The updater includes multiple layers of validation to ensure only legitimate updates from the official repository can be applied.

URL validation

Before downloading, the URL is parsed with parse_url() and checked against strict rules:

  • Scheme must be https
  • Hostname must be exactly github.com
  • Path must start with /tonyshawjr/outpost/releases

Any URL that does not match all three criteria is rejected immediately.

Zip slip prevention

Before extracting any file from the downloaded zip archive, Outpost validates every entry path to ensure it does not contain directory traversal sequences (e.g. ../). If any entry attempts to write outside the expected directory, the entire update is aborted.

Role-based access

Both updater endpoints require the super_admin role. Regular admin users and editors cannot check for or apply updates. This ensures that only the site owner can modify the installation.

HTTPS enforcement

The download URL must use HTTPS. Plain HTTP URLs are rejected to prevent man-in-the-middle attacks during the download.

Warning — The updater replaces core PHP files on the server. Always ensure you have a backup of your site before applying updates. While the updater is designed to be safe, having a backup provides an extra layer of protection.

Manual Updates

For sites that do not yet have the built-in updater (versions before v1.0.0-beta.11), updates must be applied manually:

  1. Download the latest release zip from the GitHub Releases page
  2. Extract the zip on your local machine
  3. Upload the core files to your outpost/ directory via FTP or SSH:
    • All .php files in the root
    • The entire admin/ directory
    • The docs/, member-pages/, and tools/ directories
  4. Clear the template cache: delete all files in outpost/cache/templates/
  5. Reload the admin panel

Once the manual update includes the updated api.php with the updater endpoints, all future updates can be done through Settings → Updates.

Tip — Do not overwrite the content/ directory during a manual update. This contains your database, uploads, themes, and backups. You may safely clear cache/templates/.

Configuration

The updater uses one key constant defined at the top of api.php:

define('OUTPOST_GITHUB_REPO', 'tonyshawjr/outpost');

This constant tells the updater which GitHub repository to check for releases. The GitHub Releases API URL is constructed as:

https://api.github.com/repos/{OUTPOST_GITHUB_REPO}/releases/latest

If the repository ever moves, update this constant and the URL validation logic in the updates/apply handler.

The installed version is read from config.php:

define('OUTPOST_VERSION', '1.0.0-beta.13');

This is the value compared against the latest release tag to determine whether an update is available.
