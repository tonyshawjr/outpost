Revision History

Every save automatically snapshots the current content so you can browse and restore previous versions of any page or collection item.

How it works

Outpost uses automatic snapshots. Every time you save a page or collection item, the current content is snapshotted into a revisions table before the new content is written. There is no manual "save version" step — revisions accumulate silently in the background.

Each entity (page or item) keeps the last 25 revisions. Older revisions are pruned automatically on each save, so the table stays compact.

Note — Global fields (the __global__ pseudo-page) are excluded from revision tracking. Only regular pages and collection items generate snapshots.

What is snapshotted

A revision stores two JSON objects:

Column Pages Collection items
data All field values: { field_name: content, ... } The item's data JSON (title, body, blocks, custom fields)
meta Page meta: title, meta_title, meta_description, visibility, status Item meta: slug, status, published_at, scheduled_at

Together, these two snapshots capture the full state of the content at the time of the save.

Browsing revisions

Pages

In the Page Editor, the right sidebar has three tabs: Page, SEO, and History. Click the History tab to see a list of all revisions for the current page, sorted newest first. Each entry shows:

  • A relative timestamp (e.g. "2m ago", "Yesterday")
  • The user who made the save
  • The number of fields that were stored in the snapshot

Collection items

In the Collection Editor, click the History button in the top bar (next to Preview). This opens an inline drawer below the toolbar with the same revision list.

Restoring a revision

  1. Open the History tab (pages) or History drawer (collection items).
  2. Click Restore on the revision you want to roll back to.
  3. Confirm the action. The current content is snapshotted first — so the restore itself is undoable.
  4. The editor reloads with the restored content.
Tip — Because the current state is snapshotted before every restore, you can always "undo" a restore by immediately restoring the most recent revision in the list.

API endpoints

The revision system exposes two endpoints via the Admin API. Both require session authentication.

List revisions

GET api.php?action=revisions&entity_type=page&entity_id=5

Returns up to 25 revisions, newest first. Pass entity_type (page or item) and entity_id (the page or item ID).

{
  "revisions": [
    {
      "id": 42,
      "entity_type": "page",
      "entity_id": 5,
      "created_at": "2025-03-01 10:00:00",
      "created_by": 1,
      "user": "Admin",
      "field_count": 4
    }
  ]
}

Restore a revision

POST api.php?action=revisions/restore
Body: { "entity_type": "page", "entity_id": 5, "revision_id": 42 }

Snapshots the current state (so the restore is undoable), then overwrites the entity's content and meta with the revision's data. Clears the template cache for pages.

{ "success": true }

Database table

Column Type Description
id INTEGER PK Auto-incrementing revision ID
entity_type TEXT page or item
entity_id INTEGER The page or collection item ID
data TEXT (JSON) Snapshot of all field content
meta TEXT (JSON) Snapshot of title, slug, status, and other metadata
created_by INTEGER The user who triggered the save (FK to users.id)
created_at TEXT Timestamp when the revision was created

An index on (entity_type, entity_id, created_at DESC) ensures listing revisions for a given entity is fast.

Retention & pruning

After every snapshot insert, Outpost runs a prune query that deletes any revisions beyond the most recent 25 for that entity. This keeps the database compact without requiring a cron job or manual cleanup.

DELETE FROM revisions
WHERE entity_type = ? AND entity_id = ?
  AND id NOT IN (
    SELECT id FROM revisions
    WHERE entity_type = ? AND entity_id = ?
    ORDER BY created_at DESC LIMIT 25
  )

Limitations

  • Global fields are not revisioned. Changes to site-wide settings like site name, tagline, or social links are not tracked.
  • Media files are not revisioned. Replacing or transforming an image is permanent.
  • Diff view is not available. Revisions show a summary (field count) but not a side-by-side comparison of what changed.
  • Retention is fixed at 25 revisions per entity. This is not configurable.
