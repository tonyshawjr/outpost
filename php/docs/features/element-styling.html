Element & Class Styling

The builder’s Style panel edits either a reusable CSS class or a single element directly. Pick a node, style it, and Outpost bakes the result — a shared .class rule for reusable styles, or a scoped [data-node-id] rule for one-offs.

Overview

Every element you select in the visual builder can be styled two ways, and the Style panel chooses the right one automatically:

  • Class styling — when the element has a class, the panel edits that class. The styles are reusable: every element carrying the class gets them.
  • Element styling — when the element has no class, the panel edits the element itself. The styles are a one-off, scoped to that single node.

The panel header tells you which mode you’re in at a glance: it reads Editing .class for class styling, or Editing this <tag> for element styling. Both modes offer the same controls — a visual field editor and a raw CSS box — toggled with the Visual / CSS tabs.

One panel, two targets. There is no separate “class editor” and “element editor.” The Style panel always styles the element’s active class if it has one (the last class in its list), and falls back to the element itself when it has none.

When to use each

Class stylingElement styling
Use forA look you’ll reuse — buttons, cards, section wrappersA one-off tweak on a single element
Reusable?Yes — shared by every element with the classNo — applies to this element only
How it’s storedIn the shared style-class store, keyed by class nameIn the node’s own styles field, in the page tree
How it bakesA .class { … } rule in the site stylesheetA scoped [data-node-id="…"] { … } rule

Rule of thumb: reach for a class the moment a style is worth repeating. Use element styling for the small, page-specific adjustments that don’t deserve a name.

Promoting an element style to a class

Element styling is a starting point, not a dead end. The Style panel has a class combobox at the top — search for an existing class or type a new name to create one. The moment you add a class, the panel switches its target to that class (the header flips from this <tag> to .class), and subsequent edits become reusable.

Panel hint. While you’re styling an element directly, the panel reminds you: “Styling this element directly. Add a class above to make these styles reusable across elements.”

Pseudo-classes, responsive rules, and nesting

Both modes use the same nested-style model, so both support more than a flat list of properties:

  • Pseudo-classes and pseudo-elements — :hover, :focus, ::before, and friends, written as nested &-selectors.
  • Responsive breakpoints — @media blocks. The builder’s breakpoint switcher (Desktop / Tablet / Mobile) writes and reads per-breakpoint declarations, and the panel header shows an @ Tablet / @ Mobile badge when you’re editing a non-default breakpoint.
  • Descendant selectors — nested & .child rules.

The raw CSS box

Switch to the CSS tab to edit the target’s declarations as plain, nested CSS. Scalar properties sit at the top; nested &-selectors and @media blocks follow. The two tabs stay in sync — edits in the CSS box update the visual fields and vice versa.

color: var(--primary);
padding: 1rem 1.5rem;
border-radius: 10px;

&:hover {
  opacity: 0.85;
  transform: translateY(-2px);
}

@media (max-width: 600px) {
  padding: 0.75rem 1rem;
}

The & stands for the target selector — the class (.card) in class mode, or the scoped element selector ([data-node-id="…"]) in element mode. Nesting a plain selector like .child without & is treated as a descendant (& .child).

How element styles bake

Element styles live in the node’s styles field inside the page tree. On render, Outpost gives any node that has styles a stable data-node-id attribute and emits a matching scoped rule — so the one-off CSS attaches to exactly that element and nothing else.

  • Attribute — outpost_render_node() adds data-node-id="n_…" only to nodes that actually carry styles (and whose id matches the internal n_… pattern). Styleless nodes stay clean.
  • Rule — outpost_node_styles_css() walks the tree and bakes a [data-node-id="n_…"] { … } rule for each styled node, expanding nested &-selectors and @media blocks through the same emitter that produces class rules.
  • Same sanitization as classes — element CSS runs through the identical property/value/selector validation used for class CSS. There is no weaker path for one-off styles.

Class styles bake the way you’d expect — a .class { … } rule in the stylesheet — using the same nesting-aware emitter. The only difference between the two is the selector: a class name versus a scoped data-node-id.

Scoped, not inline. Element styles are emitted as a real stylesheet rule keyed on data-node-id, not as an inline style="" attribute. That keeps :hover, @media, and nested selectors possible for a single element — things inline styles can’t express.
