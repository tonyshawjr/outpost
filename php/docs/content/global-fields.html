Global Fields

Site-wide content available in every template — site name, logo, social links, analytics IDs, contact information, footer text, and any other value shared across the whole site. Global fields auto-scaffold from your HTML templates and are edited from one place in the admin.

What are global fields?

Global fields hold content that is shared across the entire site, not scoped to a single page or collection item. Where a page field like <span data-outpost="hero_title">Default</span> only appears on the specific page that defines it, a global field like <span data-outpost="site_name" data-scope="global">Default</span> is available in every template and every partial across every page of the site.

They live in Admin → Globals. Like page fields, global fields are auto-scaffolded — the admin discovers them automatically by scanning your site's HTML templates for any tag using the data-scope="global" attribute. You never register them manually; just use them in your templates and they appear.

Template syntax

Global fields use the data-scope="global" attribute to distinguish them from page-scoped fields:

<!-- Plain text — auto-escaped -->
<span data-outpost="site_name" data-scope="global">My Site</span>

<!-- Image field — opens media picker in admin -->
<img data-outpost="logo" data-scope="global" data-type="image" src="default-logo.png" alt="Site logo">

<!-- Richtext — render HTML as-is (no escaping) -->
<div data-outpost="bio" data-scope="global" data-type="richtext">Default bio text.</div>

<!-- Link field — outputs the raw URL for use in href -->
<a data-outpost="twitter_url" data-scope="global" data-type="link" href="#">Twitter</a>

<!-- Conditional — only render if the global has a value -->
<outpost-if field="ga4_id" scope="global" exists>
  <script async src="https://www.googletagmanager.com/gtag/js"></script>
</outpost-if>

See Template Syntax → Global Fields for the complete syntax reference.

Auto-scaffolding

You do not register global fields in advance. When you add a new data-outpost="field_name" data-scope="global" attribute anywhere in your site — including partials and all page templates — that field appears automatically in Admin → Globals the next time any page is rendered or the admin is opened.

<!-- Add this to head.html — a "favicon" field appears in Admin → Globals immediately -->
<link rel="icon" href="default-favicon.ico">
<img data-outpost="favicon" data-scope="global" data-type="image" src="default-favicon.ico" style="display:none">
Tip — Define all your site identity, social, analytics, and contact fields in your partials (partials/head.html, partials/nav.html, partials/footer.html). They all scaffold into Globals automatically at first render.

Field types for globals

The admin input type is inferred from the data-type attribute in the template tag. Use the appropriate type to get the right editor widget:

Template tagAdmin field typeUse for
<span data-outpost="field" data-scope="global">Default</span> Text input Site name, tagline, phone number, GA4 ID, email address
<p data-outpost="field" data-scope="global" data-type="textarea">Default</p> Textarea Multi-line plain text — short bio, address block
<div data-outpost="field" data-scope="global" data-type="richtext">Default</div> Richtext (WYSIWYG) Footer text with links, bio with formatting
<img data-outpost="field" data-scope="global" data-type="image" src="default.jpg"> Image (media picker) Logo, favicon, profile photo, OG image
<a data-outpost="field" data-scope="global" data-type="link" href="#">Link</a> Link / URL input Social profile URLs, CTA links
<span data-outpost="field" data-scope="global" data-type="select">Default</span> Select dropdown Theme mode toggle, language selection

Managing globals in admin

Navigate to Admin → Globals. All global fields from all templates and partials at your site root appear on a single form, grouped and ordered by the sequence they appear in the template source. Each field renders the appropriate editor widget based on its type.

Press Cmd/Ctrl+S or click Save to save all globals in one request. Saving global fields clears the HTML page cache so visitors immediately see the updated values.

<!-- In partials/nav.html -->

  </outpost-if>
  <outpost-if field="logo" scope="global" not="">
  </outpost-if>
  <span data-outpost="site_name" data-scope="global">My Site</span>
</a>

    </outpost-if>

    </outpost-if>

    </outpost-if>
  </div>

Google Analytics 4

<!-- In head.html — only outputs GA script if ga4_id has been set in admin -->
<outpost-if field="ga4_id" scope="global" exists>
<script async src="https://www.googletagmanager.com/gtag/js"></script>
<script>
  window.dataLayer = window.dataLayer || [];
  function gtag(){dataLayer.push(arguments);}
  gtag('js', new Date());
  gtag('config', document.querySelector('[data-outpost="ga4_id"][data-scope="global"]').textContent);
</script>
</outpost-if>
<span data-outpost="ga4_id" data-scope="global" style="display:none"></span>

Footer with contact info

<footer>
  <div data-outpost="footer_text" data-scope="global" data-type="richtext">Footer content here.</div>

  <address>
    <outpost-if field="contact_email" scope="global" exists>
      <a data-outpost="contact_email" data-scope="global" data-type="link" href="mailto:hello@example.com">hello@example.com</a>
    </outpost-if>
    <outpost-if field="phone" scope="global" exists>
      <a href="tel:555-555-5555"><span data-outpost="phone" data-scope="global">555-555-5555</span></a>

      <span><p data-outpost="address" data-scope="global" data-type="textarea">123 Main St</p></span>
    </outpost-if>
  </address>
</footer>

Globals in conditionals

Use <outpost-if field="field_name" scope="global" exists> to render a block only when a global has content. This is the standard pattern for optional fields — analytics IDs, social links, and promotional banners that may not always be active:

<!-- Optional social icons — only show links that have been filled in -->
<div class="social">
  <outpost-if field="instagram_url" scope="global" exists>
    <a data-outpost="instagram_url" data-scope="global" data-type="link" href="#">Instagram</a>
  </outpost-if>
  <outpost-if field="twitter_url" scope="global" exists>
    <a data-outpost="twitter_url" data-scope="global" data-type="link" href="#">Twitter</a>
  </outpost-if>
  <outpost-if field="youtube_url" scope="global" exists>
    <a data-outpost="youtube_url" data-scope="global" data-type="link" href="#">YouTube</a>
  </outpost-if>
</div>

Storage and caching

Global field values are stored in the settings table in the SQLite database as key-value pairs. The key is the field name (e.g., site_name) and the value is the content string (or a JSON string for richtext and image fields).

When a global field is saved in Admin → Globals, the entire HTML page cache is invalidated so all pages are regenerated on next request with the updated values. This ensures that globals used in partials (like the site name in the page title) are consistent across the site immediately after saving.

Migrating a field from page-scoped to global

If you initially put a field in one page template (<span data-outpost="author_name">Default</span>) and later decide it should be available everywhere, add data-scope="global" to the attribute (<span data-outpost="author_name" data-scope="global">Default</span>). The new global field scaffolds automatically. Content from the old page-scoped field does not carry over — copy the value manually to Admin → Globals after making the change.

Note — Removing a tag from a template does not delete the stored value. The field simply stops appearing in the admin form. If you later re-add the tag, the previously stored value is still there.
