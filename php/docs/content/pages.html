Pages

Static CMS-managed pages that map to HTML template files at your site root — About, Contact, Home, Services, and any other page in your site.

What are pages?

Pages are the static, individually-editable content items in your site — the About page, Contact page, Home page, and so on. Unlike collection items, which repeat in a structured list, each page stands alone and maps directly to a single .html template file at your site root.

Pages are managed in Admin → Pages. You do not create pages manually in the admin — they are discovered automatically the first time the router receives a request for a URL that matches an existing HTML template file.

How pages are discovered

When a visitor (or the admin) first loads a URL, the router looks for a matching .html template at the site root. If one is found and no page record yet exists in the database, the CMS creates the page record automatically. From that point on, the page appears in Admin → Pages and its fields can be edited.

Note — The pages list in admin only shows paths that map to a template file that exists at your site root.

Page properties

Every page has the following core properties:

Property Description
TitleThe page's display title. Editable at the top of the page editor.
PathThe URL path (e.g. /about). Set by the router at discovery time; not editable.
Statuspublished or draft — controls public visibility.
Visibilitypublic, members, or paid — controls who can access the page.
Meta titleSEO title tag. 60 character limit.
Meta descriptionSEO meta description. 160 character limit.
FieldsContent fields scaffolded from the page's template (see below).

Page status

The status controls whether a page is publicly accessible.

Status Behavior
publishedAccessible to visitors (subject to visibility rules).
draftReturns a 404 to all visitors, even if they know the URL. The page is still editable in admin.
Warning — Draft status returns a true 404, not a login redirect. Use draft for pages you are building, and members or paid visibility to gate finished pages from the public.

Page visibility

Visibility controls who can view a published page. This is independent of status — a page must be published first, then visibility determines the audience.

Visibility Behavior
publicAnyone can access. Default for all new pages.
membersRequires a logged-in member account. Non-members are redirected to the login page.
paidRequires a paid member account. Free members are redirected to the upgrade page.

Visibility is enforced in both front-router.php (PHP dev server) and test-site/index.php (Apache/production) by calling cms_require_member() or cms_require_paid_member(). See Protecting Pages for the members system overview.

Page locking

Admins can lock individual pages to prevent non-admin roles from editing them. A locked page shows a lock icon in the pages list and a "This page is locked" banner in the editor for non-admin users.

  • super_admin and admin can lock/unlock pages and always edit locked pages
  • developer and editor cannot edit or delete locked pages — the API returns 403
  • The lock is set via the Lock page checkbox in the Page sidebar tab of the editor

See Admin Roles for full details on per-page locks and collection-scoped editors.

Page fields and auto-scaffolding

Page fields are the editable content slots that your template uses. When a template contains <span data-outpost="hero_title">Default</span>, the CMS creates a "hero_title" field for that page automatically. This is called auto-scaffolding.

The scaffolding process runs when the page is first discovered and whenever the page is loaded in the admin. This means adding a new data-outpost="field_name" attribute to a template makes that field appear in the admin immediately — no manual schema setup required.

The field type is inferred from the data-type attribute in the template tag:

Template tag Field type created
<span data-outpost="field_name">Default</span>text
<p data-outpost="field_name" data-type="textarea">Default</p>textarea
<div data-outpost="field_name" data-type="richtext">Default</div>richtext (WYSIWYG)
<img data-outpost="field_name" data-type="image" src="default.jpg">image (media picker)
<a data-outpost="field_name" data-type="link" href="#">Link</a>link
<span data-outpost="field_name" data-type="select">Default</span>select (dropdown)
Tip — If you remove a field from a template, the field and its stored value remain in the database. It simply stops appearing in the admin for that page. No data is deleted.

Where page fields are stored

Page fields are stored in the fields table, keyed by the page path. Field values persist as long as the template file exists.

Page fields vs. collection item fields

Page fields Collection item fields
Storagefields table, keyed by page pathJSON in collection_items.data
Schema sourceAuto-scaffolded from template analysisDefined explicitly in collection schema editor
Edit locationAdmin → Pages → edit pageAdmin → Collections → edit item
Multiple itemsNo — one set of fields per pageYes — each item has its own data

The page editor

Open a page in Admin → Pages to edit it. The editor uses a two-column layout:

  • Left column — the page title (large, editable inline) followed by all content fields scaffolded from the template. Fields appear in the order they are encountered in the template source.
  • Right sidebar — three tabs: Page (status and visibility toggles), SEO (meta title, meta description, and a live Google search preview), and History (browse and restore previous versions — see Revision History).

SEO fields

The SEO tab in the right sidebar provides:

  • Meta title — overrides the <outpost-seo /> title output in the template. 60 character limit with a live character counter.
  • Meta description — overrides the SEO meta description. 160 character limit.
  • Search preview — a simulated Google result card that updates live as you type, showing how the page will appear in search results.

Saving

Press Cmd/Ctrl+S or click the Save button in the header. All fields on the page save together in a single request.

Template tags for page fields

In your template, output page fields using data-outpost attributes. The field name must match exactly (case-sensitive, underscores, no spaces).

<!-- Plain text field -->
<h1 data-outpost="hero_title">Default Hero Title</h1>

<!-- Richtext field (renders HTML as-is) -->
<div class="body-content" data-outpost="page_body" data-type="richtext">Default body content.</div>

<!-- Image field (registers as image picker) -->
<img data-outpost="hero_image" data-type="image" src="default.jpg" alt="Hero image">

<!-- Textarea field (preserves line breaks) -->
<p data-outpost="intro_text" data-type="textarea">Default intro text.</p>

<!-- Link field -->
<a data-outpost="cta_link" data-type="link" href="#"><span data-outpost="cta_label">Call to Action</span></a>

<!-- SEO meta tags (in <head>) -->
<outpost-seo />
Note — All output is auto-escaped by default. Use data-type="richtext" only for fields that contain intentional HTML. See Output Tags & Filters for the full reference.

Adding a new page

To add a new page to your site:

  1. Create a new .html template file at your site root (e.g., team.html).
  2. Add data-outpost attributes for the fields you want (<span data-outpost="team_intro">Default</span>, <img data-outpost="team_photo" data-type="image" src="default.jpg">, etc.).
  3. Visit the URL in a browser or in the admin preview (/team).
  4. The CMS discovers the page, creates the page record, and scaffolds the fields.
  5. The page now appears in Admin → Pages ready to edit.
Tip — You do not need to restart anything or run a migration. The CMS discovers and scaffolds pages on the first request.

Deleting a page

Pages can be deleted from the page editor. Open a page in Admin → Pages, then click Delete page at the bottom of the Settings sidebar tab. A confirmation step asks you to confirm before the deletion is permanent.

Deleting a page removes:

  • The .html template file from the site root (e.g. about.html)
  • The page record and all of its fields from the database
  • The field registry entries for that page path
Warning — Page deletion is permanent and cannot be undone. The template file, page record, and all field content are removed. If you want to temporarily hide a page, set its status to draft instead.
Note — The homepage (/) cannot be deleted. The delete button does not appear for the homepage in the editor.
