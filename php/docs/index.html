Overview

Outpost is a developer-first, self-hosted CMS built on PHP 8.x and SQLite. Drop one directory onto any PHP server and you have a fully working content management system — no Composer, no npm on the server, no database server to configure.

What is Outpost?

Outpost is a self-contained CMS that lives entirely inside a single outpost/ directory. Copy it to your server, point your web server at it, and your first HTTP request automatically creates the SQLite database, runs all schema migrations, and seeds default data. There is nothing to install, no package manager to run server-side, and no external services to configure.

The admin panel is a Svelte 5 single-page application served from /outpost/. It talks to a REST API at outpost/api.php. The public-facing site is powered by a Liquid-inspired template engine that compiles .html theme files to PHP, caches them on disk, and serves them fast. Editors never touch PHP — they work entirely in the admin panel or in plain .html template files.

Note — Outpost is not a hosted service, not a SaaS platform, and not a page builder. It is a directory you copy onto a PHP server you already control. You own everything: the code, the database, the uploads, the templates.

Tech Stack

Outpost's entire server-side runtime is vanilla PHP with no third-party dependencies. The admin UI is pre-built — you never need Node.js on the server.

What Outpost Is Not

Understanding Outpost's boundaries helps you choose the right tool.

Not a page builder. There is no drag-and-drop layout editor. You write HTML templates as a developer. Editors manage content in the admin; developers control presentation in template files.

Not a framework. Outpost does not give you a PHP application routing layer, a dependency injection container, or an ORM. It is a CMS with a defined feature set, not a foundation for arbitrary application code.

Not a hosted service. There is no cloud dashboard, no monthly subscription, and no vendor lock-in. You host it on any server you control that runs PHP.

Not WordPress. There is no plugin marketplace, no PHP hooks and filters API, and no theme directory. Outpost is intentionally lean — the entire codebase is readable in an afternoon, and every behaviour is explicit.

Key Concepts

Six systems make up the core of Outpost. A high-level understanding of each will orient you before you dive into the detailed documentation.

Themes

A theme is a directory of .html files that use Outpost's Liquid-inspired template syntax. Each file maps directly to a URL path: index.html handles /, about.html handles /about, post.html handles /post/{slug}, and so on. Templates are compiled to PHP on first request and written to a cache directory — subsequent requests serve the cached version until Outpost detects the source file has been modified.

Themes are completely standalone. Outpost adds no classes, no JavaScript, and no CSS of its own to the public site output. You have full control over every byte of HTML the site serves.

Pages

Pages are CMS-managed content items that map to specific URLs you define in the admin. Unlike collection items, each page is unique. Pages support a rich-text body, custom meta fields, SEO metadata (title, description, Open Graph), and per-page visibility (public, members-only, paid).

Collections

Collections are repeatable content types. A blog is a collection of posts. A portfolio is a collection of projects. You define the fields for each collection in the admin — text, rich text, image, toggle, select, textarea, repeater, gallery, and more — and Outpost generates the editor UI automatically. In templates, collection items are queried with {% for %} loops and individual items with {% single %} blocks.

Global Fields

Global fields are site-wide variables available in every template. Use them for the site name, tagline, footer copy, social URLs, analytics IDs, or any value that appears on multiple pages. In templates, global fields are accessed with {{ @field_name }}.

Members

The member system provides front-end user accounts with email and password authentication. Members are distinct from admin users — they are your site's audience. Pages and collection items can be restricted to logged-in members or a paid-member tier. Built-in member pages handle login, registration, password reset, and profile management without any custom PHP required.

API

Outpost exposes three APIs. The Admin API (used by the admin SPA) requires a session cookie and CSRF token. The Public Content API returns collection and page data as JSON without authentication — use it to power a headless front-end or a mobile app. The Member API handles member authentication, registration, and profile management from your front-end templates.

How the Template Pipeline Works

When a visitor requests a URL, the following sequence occurs:

  1. Routing. The web server forwards all requests to index.php (Apache or Nginx) or front-router.php (PHP built-in dev server). The router maps the URL to a theme template file — for example, /blog maps to themes/active-theme/blog.html and /post/my-first-post maps to post.html with ?slug=my-first-post.
  2. Cache check. Outpost checks whether a compiled PHP file exists in outpost/cache/templates/ and whether it is newer than the source .html file. If the cache is fresh, it skips compilation.
  3. Compilation. If the cache is stale or missing, the template engine reads the .html source, walks through every Liquid tag, and outputs equivalent PHP code. The result is written to disk as a .php file in the template cache directory.
  4. Execution. The compiled PHP file is included. Template output tags resolve to CMS function calls — cms_text(), cms_collection_list(), cms_global() and so on — that query the SQLite database and return content.
  5. Output. The complete HTML response is sent to the browser. Optionally, Outpost can cache the full rendered HTML of each page so that repeat requests skip database queries entirely.

From a theme developer's perspective this is entirely transparent — you write {{ title }} and the page title appears. Compilation and caching happen automatically in the background.

Tip — Because templates are compiled to native PHP, every page load runs at full PHP speed. There is no interpreted template engine executing at request time — just a thin PHP file that calls database functions.

The Admin Panel

The admin panel lives at /outpost/. This path is reserved — no theme template can conflict with it. The admin is a Svelte 5 SPA that communicates exclusively through the REST API at outpost/api.php. It requires no separate process, no WebSocket server, and no build step on the server.

On first visit to /outpost/ when no database exists, Outpost bootstraps itself: it creates the SQLite file at outpost/data/cms.db, runs all schema migrations, creates a default admin account, and seeds a starter theme. You are taken directly to the login screen.

Warning — The default admin credentials are admin / admin. Change your password immediately after first login via Admin → Settings → Account.

No Server-Side Dependencies

A core design goal of Outpost is that deploying a site requires only copying files. There is no composer install to run, no npm install on the server, no build step, and no cron job required for basic operation. As long as the server runs PHP 8.x with the pdo_sqlite extension enabled, Outpost works.

The admin SPA is a pre-built artifact — a small set of static JS and CSS files in outpost/admin/. The Svelte source code lives in a separate cms/src/ directory and is only relevant when you are modifying the admin UI itself. Most theme developers and all content editors never need to touch it.
