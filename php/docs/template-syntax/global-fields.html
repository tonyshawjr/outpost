Global Fields

Global fields hold site-wide values — the same content available on every page and every template. Managed in Admin → Globals, referenced in templates with the @ prefix.

What Are Global Fields?

Most CMS fields are tied to a specific page or collection item. Global fields are different — they exist once and are available everywhere. A global field is the right choice for content that appears in multiple places:

  • The site name shown in the browser tab and nav bar
  • The logo image used in the header and footer
  • Social media URLs linked from multiple pages
  • The author bio shown on every blog post
  • A Google Analytics ID included on every page

If you find yourself copying the same value into multiple pages, that value belongs in a global field.

Syntax

Global fields use the @ prefix inside double curly braces:

{{ @field_name }}

The same filters available to page fields work with global fields:

{{ @site_name }}              <!-- escaped text -->
{{ @bio | textarea }}         <!-- multi-line text with nl2br -->
{{ @about_body | raw }}       <!-- richtext HTML -->
{{ @logo | image }}           <!-- image URL (media picker) -->
{{ @website_url | link }}     <!-- URL input -->
{{ @brand_color | color }}    <!-- hex color picker -->

Auto-scaffolding

The first time a template references a global field that does not yet exist, Outpost creates it automatically in Admin → Globals. There is no schema file to update and no migration to run. Visit any page that uses the new global and the field will appear in the admin ready to edit.

Tip — This means you can design your template first and fill in content later. Add {{ @site_name }} to your nav partial, deploy, and the field will appear in Globals automatically.

Auto-migration

If you move content from a page field to a global, rename it with the @ prefix in your template. Outpost detects the change and automatically migrates the existing value from the page field to the global field on the next page load. No data is lost.

<!-- Before: page-specific field -->
<title>{{ site_name }}</title>

<!-- After: promoted to global -->
<title>{{ @site_name }}</title>

Checking for a Value

Use {% if @global_name %} to conditionally render elements when a global field has been filled in. This is important for optional fields like social media links — you do not want to render an empty anchor tag if the URL has not been set.

{% if @profile_photo %}
  <img src="{{ @profile_photo | image }}" alt="{{ @author_name }}">
{% endif %}

{% if @twitter_url %}
  <a href="{{ @twitter_url | link }}" target="_blank" rel="noopener">Twitter</a>
{% endif %}

{% if @ga4_id %}
<script async src="https://www.googletagmanager.com/gtag/js?id={{ @ga4_id }}"></script>
<script>
  window.dataLayer = window.dataLayer || [];
  function gtag(){dataLayer.push(arguments);}
  gtag('js', new Date());
  gtag('config', '{{ @ga4_id }}');
</script>
{% endif %}

Common Global Field Patterns

Site Identity

FieldFilterUsage
@site_name(none)Browser tab title, nav brand text
@site_tagline(none)Subtitle under the site name, homepage hero
@logo| imageHeader and footer logo <img>
@favicon| image<link rel="icon" href="{{ @favicon | image }}">

Author / Profile

FieldFilterUsage
@author_name(none)Byline, about section heading
@bio_short| textareaShort bio paragraph in sidebar or footer

FieldUsage

Full Example: head.html Partial

This is a typical partials/head.html that uses global fields for site identity, the favicon, and GA4.

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>{{ meta.title "{{ @site_name }}" }}</title>
  <meta name="description" content="{{ meta.description "" }}">

  {% if @favicon %}
    <link rel="icon" href="{{ @favicon | image }}">
  {% endif %}

  <link rel="stylesheet" href="/themes/my-theme/assets/style.css">

  {% if @ga4_id %}
  <script async src="https://www.googletagmanager.com/gtag/js?id={{ @ga4_id }}"></script>
  <script>
    window.dataLayer = window.dataLayer || [];
    function gtag(){dataLayer.push(arguments);}
    gtag('js', new Date());
    gtag('config', '{{ @ga4_id }}');
  </script>
  {% endif %}
</head>
<body>

Full Example: nav.html Partial

<header class="site-header">
  <a href="/" class="brand">
    {% if @logo %}
      <img src="{{ @logo | image }}" alt="{{ @site_name }}">
    {% else %}
      <span>{{ @site_name }}</span>
    {% endif %}
  </a>
  <nav>
    {% for item in menu.main %}
      <a href="{{ item.url }}">{{ item.label }}</a>
    {% endfor %}
  </nav>
</header>

<footer class="site-footer">
  <div class="footer-inner">
    <div class="footer-about">

    </div>

Best Practices

  • Use global fields for anything that appears on more than one page — site name, logo, social links, footer copy, GA4 ID.
  • Use page fields for content specific to one page — page title, hero image, body text.
  • Always wrap optional globals like social links in {% if @field %} so empty fields do not render broken HTML.
  • Define all your global fields in your partials/head.html or partials/nav.html so they are scaffolded on the first page load — even if those partials are included on every page.
