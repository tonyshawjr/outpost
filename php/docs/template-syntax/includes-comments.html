Includes & Comments

Split templates into reusable partials with {% include %}, and leave developer notes with {# #} comments that are stripped at compile time and never reach the browser.

Includes

The {% include %} tag inserts another template file inline. It is the primary way to keep themes DRY — header, navigation, and footer markup is written once and shared across every page template.

{% include 'partial-name' %}

This loads partials/partial-name.html from the current theme directory. Do not include the file extension in the tag name.

Standard Partial Names

By convention, themes use these partial names. Sticking to these names makes themes easier to understand and share.

Tag File Contains
{% include 'head' %} partials/head.html DOCTYPE through opening <body> — charset, viewport, title, CSS, GA4
{% include 'nav' %} partials/nav.html Site header and navigation
{% include 'footer' %} partials/footer.html Footer content, closing </body></html>
{% include 'sidebar' %} partials/sidebar.html Sidebar content (optional)

Subdirectory Partials

Partial names can include path segments to organize components into subdirectories. Forward slashes map to subdirectories inside partials/.

{% include 'components/card' %}
{# loads partials/components/card.html #}

{% include 'components/hero' %}
{# loads partials/components/hero.html #}

{% include 'sections/pricing' %}
{# loads partials/sections/pricing.html #}

Context Inheritance

Partials have full access to all CMS functions and global variables. Loop variables from the parent template are also accessible inside an included partial. This enables component-style composition — write a card partial once and include it inside any loop.

{# blog.html #}
{% for post in collection.post limit:10 %}
  {% include 'components/post-card' %}
{% endfor %}
{# partials/components/post-card.html #}
<article class="post-card">
  <a href="{{ post.url }}">
    {% if post.featured_image %}
      <img src="{{ post.featured_image }}" alt="{{ post.title }}">
    {% endif %}
    <h3>{{ post.title }}</h3>
    <p>{{ post.excerpt | or_body }}</p>
    <span class="read-more">Read more &rarr;</span>
  </a>
</article>
Note — The loop variable name you choose (here post) is the same name you use inside the partial. If your loop uses {% for item in collection.post %}, the partial accesses fields as {{ item.title }}, not {{ post.title }}.

Compilation and Caching

Partials are compiled to PHP and cached independently of the templates that include them. When you edit a partial, clear the template cache to see the change take effect. The cache is stored in outpost/cache/templates/.

Typical Theme Page Structure

A well-structured theme page template is mostly includes, with only the page-specific content in the middle.

{# index.html — homepage #}
{% include 'head' %}
{% include 'nav' %}

<main>
  <section class="hero">
    <h1>{{ hero_heading }}Welcome.{{ /hero_heading }}</h1>
    <p>{{ hero_subheading }}</p>
    {% if cta_url %}
      <a href="{{ cta_url | link }}" class="btn">{{ cta_label }}Get started{{ /cta_label }}</a>
    {% endif %}
  </section>

  <section class="posts">
    <h2>Latest Posts</h2>
    {% for post in collection.post limit:3 %}
      {% include 'components/post-card' %}
    {% else %}
      <p>No posts yet.</p>
    {% endfor %}
  </section>
</main>

{% include 'footer' %}

Comments

Template comments use the {# #} syntax. They are stripped entirely during template compilation — they do not appear in the rendered HTML, and they do not exist in the compiled PHP cache. They are developer-only notes.

{# This is a template comment — never output to the browser #}

Multi-line Comments

{#
  TODO: Add folder filter bar here.
  Will need the ?category= query param.
  See loops/folder-loops for the syntax.
#}

Use Cases

Documenting sections:

{# ============================================
   HERO SECTION
   Fields: hero_heading, hero_body, cta_label, cta_url
   ============================================ #}
<section class="hero">
  <h1>{{ hero_heading }}Build something great.{{ /hero_heading }}</h1>
  <div>{{ hero_body | raw }}</div>
  <a href="{{ cta_url | link }}">{{ cta_label }}Learn more{{ /cta_label }}</a>
</section>

{# ============================================
   FEATURED POSTS — shows 3 most recent
   ============================================ #}
<section class="featured-posts">
  {% for post in collection.post limit:3 %}
    {% include 'components/post-card' %}
  {% endfor %}
</section>

Temporarily disabling template code:

{# Sidebar disabled until content is ready
<aside class="sidebar">
  {% include 'sidebar' %}
</aside>
#}

Leaving notes for other developers:

{# The post.url field is auto-built from the collection url_pattern.
   For this theme it outputs /post/{slug}.
   Do not hardcode /post/ — the pattern could change. #}
<a href="{{ post.url }}">{{ post.title }}</a>

Template Comments vs. HTML Comments

Syntax Stripped at compile? Visible in browser DevTools?
{# template comment #} Yes — removed before the PHP cache is written No
<!-- HTML comment --> No — compiled and output as-is Yes

Use {# #} for developer notes you never want to expose. Use <!-- --> only when you intentionally want the comment in the browser source, such as for IE conditional comments or structured HTML section markers.

Tip — Template comments have zero runtime cost. Because they are removed at compile time (not render time), they do not affect page performance regardless of how many you write.

Front Matter Convention

By convention, Outpost theme templates use a front matter block at the very top of each file — a structured comment that describes the template's purpose, the fields it uses, and which engine features it demonstrates. This is purely a developer documentation convention; the engine strips all {# #} comments at compile time.

The format mirrors YAML front matter from static site generators like Jekyll and Hugo, making it immediately familiar to most developers.

{#---
Template:   index.html
Route:      /
Title:      Homepage

Page fields:
  home_heading   text   — hero headline
    {{ home_heading }}Hello, I'm a Writer{{ /home_heading }}

Global fields:
  @profile_photo   image   — hero avatar
  @bio_short       text    — short bio beneath headline
  @author_name     text    — used in image alt text
  @twitter_url     link
  @github_url      link

Loops:
  collection.post    limit:3, orderby:published_at
  gallery.gallery    — page gallery field, outputs {{ img.url }}

Features demonstrated:
  Global field conditionals   {% if @global %}
  Global image                {{ @global | image }}
  Global link filter          {{ @global | link }}
  Collection loop             limit + orderby + for/else
  Gallery loop                {{ img.url }}
  or_body filter
  Page text field with default value
---#}

Why Use It

  • Instant orientation — open any template and immediately know what fields it needs, what CMS features it uses, and what route it serves.
  • Zero runtime cost — comments are stripped at compile time; front matter adds nothing to the rendered HTML.
  • Self-documenting themes — when sharing or selling a theme, developers understand the structure without reading the full template.
  • Field inventory — the front matter serves as a checklist of fields to configure in the admin before the page renders correctly.

Fields Reference in Front Matter

KeyWhat to list
Page fields:Fields using {{ field }} — editable per page in Admin → Pages. For fields with defaults, show the wrapping syntax on the line below: {{ field }}Default{{ /field }}
Global fields:Fields using {{ @global }} — set once in Admin → Globals
Loops:All {% for %} and {% single %} tags with their options
Features demonstrated:Engine features used — helpful for starter themes used as learning examples
Tip — The --- delimiters inside the comment block are cosmetic only. They are part of the comment text and have no special meaning to the engine. Use them to visually distinguish front matter from regular comments.
