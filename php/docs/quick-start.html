Quick Start

Get Outpost running locally in under five minutes. This guide walks you from an existing HTML site to a fully editable CMS with collections, navigation, and content management.

Requirements

Outpost's server requirements are minimal. You need a standard PHP environment with one commonly available extension.

Requirement Details
PHP version PHP 8.0 or higher
PHP extension pdo_sqlite — ships with most PHP builds, verify with php -m | grep pdo_sqlite
Optional extension gd — required only for automatic image thumbnail generation
Web server Apache 2.4+, Nginx, or PHP's built-in server (for local development)
Disk space Approximately 2 MB for the CMS files; database and uploads grow with your content
Note — Outpost does not require Composer, npm, Node.js, or any external package manager on the server. There is nothing to install beyond copying files.

Step 1: Get the Files

Upload install.php to the root of your website — the same directory where your HTML files live. For a local dev environment, your project directory might look like this:

my-site/
  index.html       ← your homepage
  about.html       ← your about page
  css/
    style.css
  images/
  install.php      ← the Outpost installer

Run the installer by visiting http://localhost:8080/install.php in your browser. It downloads and extracts the outpost/ directory, writes .htaccess rewrite rules, and creates the front controller. After installation your site looks like this:

my-site/
  index.html       ← your homepage (untouched)
  about.html       ← your about page (untouched)
  css/
  images/
  outpost/         ← CMS engine (created by installer)
  .htaccess        ← rewrite rules (created by installer)
  index.php        ← front controller (created by installer)
Tip — The installer only creates the outpost/ directory and routing files. It never modifies your existing HTML, CSS, or JavaScript files. Your site keeps working exactly as before.

Step 2: Start the Dev Server

PHP's built-in server is the fastest way to run Outpost locally. From your project directory, run:

php -S localhost:8080 -t . outpost/front-router.php

Breaking this command down:

  • -S localhost:8080 — starts the server on port 8080
  • -t . — sets the document root to the current directory (your project root)
  • outpost/front-router.php — the router script that handles all requests
Note — PHP's built-in server is single-threaded and suitable only for local development. For production, use Apache or Nginx. See the Deploy guide.

You can use any available port. Common alternatives:

php -S localhost:3000 -t . outpost/front-router.php
php -S 0.0.0.0:8080 -t . outpost/front-router.php  # accessible on your local network

Step 3: First Boot

Open your browser and navigate to:

http://localhost:8080/outpost/

On first visit, Outpost detects that no database exists and automatically:

  1. Creates the SQLite database at outpost/content/data/cms.db
  2. Runs all schema migrations (creates all tables)
  3. Creates a default admin account (admin / admin)
  4. Launches the setup wizard with Forge site analysis

This happens on a single request with no manual configuration. You will land on the login screen.

Warning — Change the default password immediately. The default credentials (admin / admin) are well known. Go to Admin → Settings → Account after your first login.

Step 4: Log In

Log in with:

  • Username: admin
  • Password: admin

You will land on the Dashboard. Take a moment to explore the sidebar navigation — everything in Outpost is accessible from here. Before you do anything else, go to Settings → Account and set a strong password.

Step 5: Forge Site Analysis

After logging in for the first time, the setup wizard runs Forge — Outpost's site analysis engine. Forge scans all .html files in your site root and automatically discovers:

  1. Pages — every .html file becomes an editable page in the admin
  2. Sections — <!-- outpost:sectionname --> comments define editable content blocks
  3. Navigation — <nav> elements are auto-detected for menu creation
  4. Fields — elements with data-outpost attributes are registered as editable fields

The wizard shows the site name (auto-detected from your HTML), the Forge analysis results, and lets you apply. Once applied, every discovered page and field is immediately editable in the admin panel.

Tip — Forge can be re-run at any time from the admin panel. When you add new HTML files or new data-outpost attributes, Forge will discover them automatically.

Step 6: Edit Your HTML Files

Your site's HTML files live in the site root — the same place they have always been. You can edit them two ways:

Option A: In-browser code editor

Go to Admin → Code Editor. The editor shows a file tree of your site on the left. Click any file to edit it. Changes are saved to disk immediately when you click Save (or press Cmd+S / Ctrl+S).

The code editor is convenient for quick edits and works well on any device that can access the admin panel.

Option B: Directly in your text editor

Your HTML files are in the site root — the parent directory of outpost/:

my-site/
  index.html
  about.html
  blog.html
  partials/
  css/
  images/

Open that directory in VS Code, Cursor, or any editor you prefer. Edit and save — because Outpost checks file modification times, the next page request will recompile and serve the updated template automatically. No restart required.

Tip — The template cache auto-invalidates on file save. You never need to manually clear the cache during development. Just edit the file and refresh the browser.

A simple index.html with Outpost data attributes might look like this:

<outpost-include partial="head"></outpost-include>
<outpost-include partial="nav"></outpost-include>

<!-- outpost:hero -->
<main>
  <h1 data-outpost="title">Welcome</h1>
  <div data-outpost="body" data-type="richtext">Your content here</div>
</main>
<!-- /outpost:hero -->

<outpost-include partial="footer"></outpost-include>

Step 7: Create Your First Collection

Collections are repeatable content types — the blog posts, projects, products, or whatever repeating content your site needs.

  1. Go to Admin → Collections
  2. Click New Collection
  3. Give it a name (e.g. "Blog Posts") and a slug (e.g. post)
  4. Set a URL pattern — for example /post/{slug}
  5. Add fields: click Add Field and choose a field type (Text, Rich Text, Image, Toggle, etc.)
  6. Save the collection

Once your collection exists, you can create entries under Admin → Blog Posts (or whatever you named it). The collection will appear in the sidebar automatically.

To display the collection in a template, add a loop to your HTML file:

<outpost-each collection="post" limit="10">
  <article>
    <h2><a data-bind="href:url" data-outpost="title">Post Title</a></h2>
    <p data-outpost="excerpt">Post excerpt</p>
  </article>
</outpost-each>

For individual post pages, create a post.html file in your site root and use <outpost-single>:

<outpost-single collection="post" var="post">
  <article>
    <h1 data-outpost="title">Post Title</h1>
    <div data-outpost="body" data-type="richtext">Post body</div>
  </article>
</outpost-single>

Step 8: Add Navigation

Go to Admin → Navigation. Outpost lets you create named navigation menus that you can reference in templates.

  1. Click New Menu and name it (e.g. "Main Navigation")
  2. Add items: pages, custom URLs, or collection archive links
  3. Drag items to reorder them
  4. Save

In your nav.html partial (or directly in your HTML), render the menu:

<outpost-menu name="main">
  <a data-bind="href:url" data-outpost="label">Link</a>
</outpost-menu>

Outpost also auto-detects <nav> elements in your HTML files during Forge analysis, so existing navigation structures can be converted to CMS-managed menus.

Step 9: Configure Global Fields

Global fields hold site-wide values that appear across templates. Go to Admin → Globals and add fields for your site name, tagline, footer text, social links, and any other values used in multiple places.

In templates, reference global fields with the data-scope="global" attribute:

<span data-outpost="site_name" data-scope="global">My Site</span>
<span data-outpost="footer_text" data-scope="global">Footer text</span>
<a data-outpost="twitter_url" data-scope="global" href="#">Twitter</a>

Because global fields are resolved at render time, changing a value in the admin updates every page that references it on the next request.

Development Workflow

Once you have the basics running, here is the day-to-day workflow for theme development:

  1. Start the dev server — php -S localhost:8080 -t . outpost/front-router.php
  2. Edit HTML files — open your site root in your editor
  3. Refresh the browser — Outpost detects file changes and recompiles automatically
  4. Manage content in the admin at http://localhost:8080/outpost/
  5. Commit changes — your HTML files are plain text and version-control friendly
Tip — Outpost has no watch mode or hot-reload — it simply checks file modification times on every request. This means you can edit a file, switch to the browser, and refresh to see your changes. No build step, no terminal watcher.

Environment Configuration

Outpost does not use a .env file. All configuration is stored in the SQLite database and managed through Admin → Settings. This includes:

  • Site name, URL, and description
  • SMTP mail settings (for contact forms and member password resets)
  • Member registration settings (open, invite-only, closed)
  • Cache settings
  • Analytics configuration

There is no configuration file to edit on the server. Everything is exposed through the admin UI.

Modifying the Admin SPA

Most developers never need to touch the admin source code. The admin SPA is a pre-built artifact in outpost/admin/. If you are building a custom Outpost-based product and need to modify the admin UI, the source is in cms/src/ and is built with Vite.

# One-time setup
cd cms/
npm install

# Build the admin SPA
npm run build
# Outputs to cms/php/admin/

# Deploy the built admin to your site
cp cms/php/admin/index.html your-site/outpost/admin/index.html
cp cms/php/admin/assets/app-*.js your-site/outpost/admin/assets/
cp cms/php/admin/assets/index-*.css your-site/outpost/admin/assets/
Warning — Always copy index.html when deploying a new build. The admin SPA uses hashed asset filenames (e.g. app-a1b2c3d4.js) that change on every build. If you copy the new JS file but leave the old index.html, the browser will load the old bundle forever.
