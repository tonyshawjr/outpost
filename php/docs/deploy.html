Deploy

Deploy Outpost to any PHP server — cPanel shared hosting, a VPS with Apache, or Nginx. No build step, no package manager — just upload the files, and you are live. An .htaccess and front controller are included.

PHP Requirements

Before deploying, confirm your server meets these requirements:

Requirement Details
PHP version PHP 8.0 or higher
pdo_sqlite Required — for the SQLite database. Included in most PHP builds. Verify: php -m | grep pdo_sqlite
gd Strongly recommended — used to generate image thumbnails in the Media Library. Without it, thumbnail generation is skipped.
mbstring Required — for multibyte string operations (used throughout the template engine and CMS functions)
json Required — included in all standard PHP builds; used for API responses and database JSON columns
file_uploads Must be On in php.ini to allow media uploads through the admin panel
upload_max_filesize Recommended: 10M or higher. Controls the maximum size of a single uploaded file.
post_max_size Recommended: 12M or higher. Must be larger than upload_max_filesize.

To check which extensions are loaded on your server, create a temporary file with <?php phpinfo(); and visit it in a browser, or run php -m from the command line.

cPanel / Shared Hosting

Outpost is designed primarily for cPanel shared hosting. Most cPanel servers come with Apache, mod_rewrite, and the required PHP extensions already enabled — so deployment is usually just uploading files.

Step 1 — Upload files

Using cPanel File Manager or an FTP client, upload the following to your public_html/ directory:

public_html/
  .htaccess         ← Apache rewrite rules (included with Outpost)
  index.php         ← front controller (included with Outpost)
  outpost/          ← the entire CMS directory

If you already have a .htaccess file in public_html/ (some hosts create one by default), add the Outpost rewrite rules to it rather than replacing it. The rules you need are:

RewriteEngine On
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d
RewriteRule ^ index.php [QSA,L]

Step 2 — Check PHP version

Outpost requires PHP 8.0+. In cPanel, go to MultiPHP Manager (or PHP Selector) and set your domain to PHP 8.0 or higher.

Step 3 — Check PHP extensions

Go to MultiPHP INI Editor (or Select PHP Version → Extensions) and confirm these are enabled:

  • pdo_sqlite — required (the database)
  • mbstring — required (text processing)
  • gd — recommended (image thumbnails)

Most cPanel installations have all three enabled by default.

Step 4 — Set permissions

In File Manager, right-click each of these directories and set permissions to 755:

  • outpost/content/ (and all subdirectories)
  • outpost/cache/
Tip — On most cPanel hosts, the default permissions are already correct. You only need to change them if the admin panel shows permission errors when saving content or uploading media.

Step 5 — Visit your site

Navigate to https://your-domain.com/outpost/. You should see the login screen. Log in with admin / admin and immediately change your password.

SSL on cPanel

Most cPanel hosts include free SSL via AutoSSL (Let's Encrypt). Check SSL/TLS Status in cPanel — if your domain shows a valid certificate, HTTPS is already working. After confirming, update your site URL in Admin → Settings → General to use https://.

Subdirectory installs on cPanel

To run Outpost in a subdirectory (e.g. example.com/blog/), create the folder in File Manager and upload all three items (.htaccess, index.php, outpost/) inside it. No extra configuration is needed — the .htaccess works the same way in a subdirectory.

Note — The sections below cover VPS deployments where you manage Apache or Nginx directly. If you are on cPanel shared hosting, you can skip to First Boot on Production.

Copying Files

Deployment is a file copy. Transfer the following to your server:

# What to copy to your web root
outpost/          ← the entire CMS directory
index.php         ← front controller for Apache/Nginx

The simplest method is rsync, which handles incremental updates efficiently on subsequent deploys:

# Initial deploy or full update
rsync -avz --exclude 'outpost/content/' \
           outpost/ user@your-server.com:/var/www/html/outpost/

rsync -avz index.php user@your-server.com:/var/www/html/index.php

Note the --exclude flag: the rsync command deliberately skips content/ to protect your live data (database, uploads, themes, and backups).

Warning — Never overwrite or delete outpost/content/ on your production server. This directory contains your SQLite database, uploaded media, themes, and backups — your entire site's data. It must be preserved on every re-deploy.

What to preserve on re-deploy

Path Action
outpost/content/ Preserve — never overwrite or delete
outpost/cache/ Safe to clear — Outpost rebuilds it automatically on next request
All other outpost/ files Replace with the new version
index.php Replace with the new version

Apache Configuration

Apache needs a rewrite rule to send all non-file, non-directory requests to index.php. Place this .htaccess file at your web root (the same directory as index.php):

# .htaccess at web root
RewriteEngine On
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d
RewriteRule ^ index.php [QSA,L]

The two RewriteCond lines ensure that requests for real files (CSS, JS, images) and real directories are served directly without going through PHP. Only requests for paths that do not match an actual file or directory are routed to index.php.

The mod_rewrite module must be enabled. On Debian/Ubuntu:

sudo a2enmod rewrite
sudo systemctl restart apache2

Your virtual host configuration must also allow .htaccess overrides. In your Apache virtual host block:

<VirtualHost *:80>
    ServerName example.com
    DocumentRoot /var/www/html

    <Directory /var/www/html>
        AllowOverride All
        Require all granted
    </Directory>
</VirtualHost>
Note — The AllowOverride All directive is what enables .htaccess files to function. Without it, the rewrite rules are silently ignored and all URLs except the homepage will return 404.

Nginx Configuration

With Nginx there is no .htaccess — all configuration goes in your server block. The key directive is try_files, which serves real files directly and falls back to index.php for everything else:

server {
    listen 80;
    server_name example.com;
    root /var/www/html;
    index index.php;

    location / {
        try_files $uri $uri/ /index.php?$query_string;
    }

    location ~ \.php$ {
        fastcgi_pass unix:/var/run/php/php8.2-fpm.sock;
        fastcgi_index index.php;
        include fastcgi_params;
        fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
    }

    # Deny direct access to sensitive Outpost directories
    location ~ ^/outpost/(content/data|content/backups|cache)/ {
        deny all;
        return 404;
    }
}

Adjust the fastcgi_pass socket path to match your PHP-FPM version and configuration. Common paths:

  • Ubuntu/Debian: unix:/var/run/php/php8.2-fpm.sock
  • CentOS/RHEL: unix:/run/php-fpm/www.sock
  • TCP (when PHP-FPM is on a separate host): fastcgi_pass 127.0.0.1:9000;
Tip — The location block that denies access to content/data/, content/backups/, and cache/ is a security best practice. While your SQLite database is not directly executable from the web, it is readable as a file. Blocking direct HTTP access prevents anyone from downloading your database or backup files.

File Permissions

Outpost needs write access to the content/ and cache/ directories. The web server process (typically www-data on Debian/Ubuntu, nginx or apache on other systems) must be able to write to these paths.

# Set directory permissions
chmod -R 755 outpost/content outpost/cache

# Set database file permissions (if it already exists)
chmod 644 outpost/content/data/cms.db

# If you need to set ownership (replace www-data with your web server user)
chown -R www-data:www-data outpost/content
chown -R www-data:www-data outpost/cache

If the directories do not exist, create them first:

mkdir -p outpost/content/data outpost/content/uploads outpost/content/themes outpost/content/backups outpost/cache/templates
chmod -R 755 outpost/content outpost/cache
Warning — Do not use chmod 777. World-writable directories are a security risk. Use 755 for directories and ensure the web server user is the owner, or use group ownership with 775 if your deployment user and web server user share a group.

First Boot on Production

If this is a fresh install (no database yet), the first HTTP request to your site automatically:

  1. Creates outpost/content/data/cms.db
  2. Runs all schema migrations to create every table
  3. Seeds a default admin account and sample data

Navigate to https://your-domain.com/outpost/ and you will see the login screen. Log in with admin / admin and immediately change your password.

If you are deploying to a server where a database already exists (i.e., you are updating an existing installation), the migration system runs automatically and applies only the migrations that have not been applied yet. No manual SQL is ever required.

Clearing the Cache After Deploy

After copying updated PHP files or theme files to the server, clear the template cache to ensure visitors get the latest compiled templates. The page cache should also be cleared to avoid serving stale HTML.

# Clear page cache
rm -f outpost/cache/*.html

# Clear compiled template cache
rm -f outpost/cache/templates/*.php

Alternatively, use the button in the admin panel: Admin → Settings → Clear Cache. This clears both cache layers from the browser.

Note — Clearing the cache is safe at any time. Outpost recompiles templates and rebuilds the page cache on the next request. There is no downtime and no data loss. The only effect is that the first request after clearing the cache may be slightly slower as it recompiles.

HTTPS / SSL

Outpost has no protocol-specific code — it works identically over HTTP and HTTPS. SSL configuration is handled entirely by your web server.

The recommended approach for a standard VPS is Certbot with Let's Encrypt, which provides free SSL certificates and automates renewal:

# Install Certbot (Debian/Ubuntu with Nginx)
sudo apt install certbot python3-certbot-nginx

# Obtain and install a certificate
sudo certbot --nginx -d example.com -d www.example.com

Certbot will modify your Nginx configuration to add the SSL block and set up automatic renewal via a cron job or systemd timer.

Tip — Once HTTPS is configured, update your site URL in Admin → Settings → General to use https://. This ensures sitemap.xml, canonical tags, and any absolute URLs generated by the CMS use the correct protocol.

Subdirectory Installs

Outpost does not need to run at a domain root. You can install it at any subdirectory, such as https://example.com/blog/.

For a subdirectory install, place index.php inside the subdirectory and update the .htaccess or Nginx location to match that path. For example, with Nginx at /blog/:

location /blog/ {
    try_files $uri $uri/ /blog/index.php?$query_string;
}

location ~ ^/blog/.*\.php$ {
    fastcgi_pass unix:/var/run/php/php8.2-fpm.sock;
    fastcgi_index index.php;
    include fastcgi_params;
    fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
}

Also update the site URL in Admin → Settings to reflect the subdirectory path so that generated URLs are correct.

Multiple Sites on One Server

Because Outpost is entirely self-contained in a single directory with its own SQLite database, running multiple sites on the same server is straightforward. Each site gets its own:

  • Virtual host or server block pointing to its own document root
  • outpost/ directory with its own database at outpost/data/cms.db
  • Independent uploads, cache, and themes

There is no shared database, no shared configuration, and no cross-contamination between sites. You can update each site's Outpost engine files independently without affecting others.

Deploying Admin SPA Updates

If you have modified the admin SPA source (in cms/src/), you need to build and deploy the updated SPA files. Normal content and theme changes do not require this step.

# Build the admin SPA
cd cms/
npm install    # only needed once
npm run build  # outputs to cms/php/admin/

# Deploy to server (MUST include index.html — filenames are hashed)
scp cms/php/admin/index.html user@server:/var/www/html/outpost/admin/index.html
scp cms/php/admin/assets/app-*.js user@server:/var/www/html/outpost/admin/assets/
scp cms/php/admin/assets/index-*.css user@server:/var/www/html/outpost/admin/assets/
Warning — Always copy index.html when deploying a new admin build. The SPA uses hashed asset filenames (e.g. app-3f7a2b1c.js) that change on every build. index.html contains the references to the current filenames. If you deploy only the new JS file but leave the old index.html, the browser will continue loading the old bundle and your changes will never appear.

Backup Strategy

An Outpost backup is simple because all persistent state lives in one directory: outpost/content/.

What to back up Why
outpost/content/ Database, uploads, themes, and backups — your entire site

PHP engine files, admin SPA, and docs can be restored from a release zip. Only content/ contains data that exists nowhere else.

A simple daily cron backup:

# Add to crontab with: crontab -e
# Daily database backup at 2am, kept for 30 days
0 2 * * * cp /var/www/html/outpost/content/data/cms.db /backups/cms-$(date +\%Y\%m\%d).db
0 2 * * * find /backups/ -name "cms-*.db" -mtime +30 -delete

For a full site backup (database + uploads + themes), use the built-in backup feature in Admin → Settings → Backups, or rsync the entire content/ directory to an off-site server.

Upgrading from Beta to Stable

If you are running any 1.0.0-beta.* version, upgrading to 1.0.0 stable is straightforward. No manual database migration is required — Outpost runs schema migrations automatically on the next request.

Option A: Built-in auto-updater (recommended)

If your site is running beta.11 or later, use the built-in updater:

  1. Go to Settings → Updates in the admin panel
  2. Click "Check for updates"
  3. Review the release notes, then click "Update now"
  4. The updater replaces core PHP files and the admin SPA. Your themes, database, uploads, and cache are never touched.

See Auto-Updater for full details on what gets replaced.

Option B: Manual file replacement

If your site is running beta.10 or earlier (before the auto-updater existed), or if you prefer manual control:

  1. Download the latest outpost-vX.X.X.zip from GitHub Releases
  2. Extract the zip
  3. Copy all .php files from the zip root into your site's outpost/ directory, overwriting existing files
  4. Copy the entire admin/ directory (replace the existing one)
  5. Copy the docs/, member-pages/, and tools/ directories
  6. Delete the contents of outpost/cache/templates/ to force template recompilation
Do NOT overwrite content/. This directory contains your database, uploads, themes, and backups. You may safely clear cache/templates/.

What changes between beta and stable

Area Change
Database schema New columns added automatically via migrations on first request. No manual SQL needed.
Session keys Member session keys were renamed in beta.14. Active member sessions will be logged out once — they simply log in again.
Taxonomy → Folder rename Database tables were auto-migrated in beta.7. Old API endpoints (taxonomies/*, terms/*) still work as aliases. Old template syntax ({% for term in taxonomy.slug %}) still works.
Template cache Cleared automatically after auto-update. If upgrading manually, delete cache/templates/*.php.
Security hardening Beta.14 added SSRF prevention, IP-based rate limiting, TOTP replay prevention, SVG DOM sanitization, and more. All automatic — no configuration needed.
Config constants OUTPOST_VERSION in config.php is updated automatically. No manual version bump needed.

Post-upgrade verification

  1. Log in to the admin panel — the version pill in the footer should show the new version
  2. Visit the frontend — pages should render correctly (template cache is rebuilt automatically)
  3. Upload an image in Media to verify file permissions
  4. If you use member login, verify a member can still log in
  5. If you use channels, verify the channel sync still works (Settings → Channels → Sync)
