API Overview & Auth

Outpost exposes three separate APIs for different use cases: a session-authenticated Admin API for CMS operations, a public Content API for headless and frontend use, and a key-authenticated Sync API for the Outpost Builder desktop app.

The three APIs at a glance

API Base URL Auth Use case
Admin API api.php?action=<endpoint> Session + CSRF or Bearer token CMS admin UI, CI/CD, headless integrations
Public Content API api.php?action=content/<endpoint> None (open CORS) Headless frontends, static site generators, JS apps
Member API member-api.php?action=<endpoint> Member session cookie + CSRF token Member login, registration, profile updates
Sync API sync-api.php?action=<endpoint> X-Outpost-Key header Outpost Builder desktop app, theme file sync

Base URL

All API files live inside the /outpost/ directory of your site. Assuming your site is at https://example.com:

https://example.com/outpost/api.php?action=<endpoint>
https://example.com/outpost/member-api.php?action=<endpoint>
https://example.com/outpost/sync-api.php?action=<endpoint>
Note All requests return Content-Type: application/json; charset=utf-8. Send JSON request bodies with Content-Type: application/json.

HTTP methods and routing

Each endpoint is addressed by combining the HTTP method with the action query parameter. The same action value may behave differently depending on the method:

GET  api.php?action=pages         → list all pages
GET  api.php?action=pages&id=5    → get a single page
PUT  api.php?action=pages&id=5    → update a page
POST api.php?action=collections   → create a collection
DELETE api.php?action=items&id=12 → delete an item

Admin API authentication

The Admin API supports two authentication methods: session-based (for the browser admin UI) and API key-based (for headless/external use). You can use either one — they access the same endpoints.

Option A — API key (headless)

For CI/CD pipelines, static site generators, and external tools, use a Bearer token. Create an API key in Settings → API Keys, then pass it in every request:

Authorization: Bearer op_a1b2c3d4e5f6...

No session cookie or CSRF token is needed — the Bearer token handles both authentication and authorization. The key inherits the role and permissions of the user it was created under.

Example — curl:

# Read pages
curl -H "Authorization: Bearer op_a1b2c3d4e5f6..." \
     "https://example.com/outpost/api.php?action=pages"

# Create a collection item (no CSRF token needed)
curl -X POST \
     -H "Authorization: Bearer op_a1b2c3d4e5f6..." \
     -H "Content-Type: application/json" \
     -d '{"collection": 1, "slug": "my-post", "data": {"title": "Hello"}}' \
     "https://example.com/outpost/api.php?action=items"
Security API keys are shown once on creation and stored as bcrypt hashes — they cannot be retrieved later. Treat them like passwords. Revoke compromised keys immediately in Settings.

Option B — Session + CSRF (browser)

The browser admin UI uses session cookies. You must log in to obtain a session cookie, then pass a CSRF token on all mutating requests (POST, PUT, DELETE).

Step 1 — Log in

POST api.php?action=auth/login

Authenticates an admin user and sets the outpost_session cookie. This endpoint is public — no prior auth required.

Request body:

{
  "username": "admin",
  "password": "your-password"
}

Success response (no 2FA):

{
  "success": true,
  "user": {
    "id": 1,
    "username": "admin",
    "role": "super_admin"
  },
  "csrf_token": "a1b2c3..."
}

If the user has two-factor authentication enabled, the response will instead contain requires_2fa and a totp_token — see the 2FA section below for details.

The session cookie outpost_session is HttpOnly and has a 24-hour lifetime. All subsequent admin API requests must carry this cookie.

Rate limiting The login endpoint is rate-limited to 5 failed attempts per 60 seconds per IP address. After 5 failures you will receive a 429 Too Many Requests response.

Step 2 — Get a CSRF token

GET api.php?action=auth/me

Returns the currently authenticated user and a fresh CSRF token. Use this to bootstrap your frontend after login.

Response:

{
  "user": {
    "id": 1,
    "username": "admin",
    "email": "admin@example.com",
    "display_name": "Admin",
    "avatar": "",
    "bio": "",
    "role": "super_admin",
    "totp_enabled": false
  },
  "csrf_token": "a1b2c3d4e5f6...",
  "version": "1.0.0"
}

Step 3 — Send CSRF token on mutations

All POST, PUT, and DELETE requests to the Admin API require a valid CSRF token. Send it as an HTTP header:

X-CSRF-Token: a1b2c3d4e5f6...

Or as a field in the JSON body:

{
  "_csrf": "a1b2c3d4e5f6...",
  "title": "My Page"
}
Note The media upload endpoint (media/upload) and the WordPress import endpoint (import/wordpress) are exempt from the standard JSON CSRF check because they use multipart form uploads. They perform their own CSRF validation via a form field named csrf_token.

Full login example — curl

# 1. Log in and save the session cookie
curl -c cookies.txt -X POST https://example.com/outpost/api.php?action=auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"secret"}'

# 2. Get CSRF token
curl -b cookies.txt https://example.com/outpost/api.php?action=auth/me

# 3. Use the token on a mutating request
curl -b cookies.txt -X PUT \
  "https://example.com/outpost/api.php?action=pages&id=1" \
  -H "Content-Type: application/json" \
  -H "X-CSRF-Token: a1b2c3d4e5f6..." \
  -d '{"title":"Home","status":"published"}'

Full login example — JavaScript fetch

// 1. Log in
const loginRes = await fetch('/outpost/api.php?action=auth/login', {
  method: 'POST',
  credentials: 'include',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ username: 'admin', password: 'secret' }),
});

// 2. Get CSRF token
const meRes = await fetch('/outpost/api.php?action=auth/me', {
  credentials: 'include',
});
const { csrf_token } = await meRes.json();

// 3. Mutating request
const updateRes = await fetch('/outpost/api.php?action=pages&id=1', {
  method: 'PUT',
  credentials: 'include',
  headers: {
    'Content-Type': 'application/json',
    'X-CSRF-Token': csrf_token,
  },
  body: JSON.stringify({ title: 'Home', status: 'published' }),
});
const result = await updateRes.json();

Log out

POST api.php?action=auth/logout

Destroys the admin session and clears the cookie. Requires an active session and a valid CSRF token.

Admin password reset

Admin users can reset their own password via a two-step flow. These endpoints are public — no session required.

POST api.php?action=auth/forgot

Sends a password reset email to the given address if it belongs to an admin account. Always returns {"success": true} regardless of whether the address exists, to prevent user enumeration.

Request body:

{ "email": "admin@example.com" }
POST api.php?action=auth/reset

Completes a password reset. The token is valid for 1 hour. Requires a password of at least 8 characters.

Request body:

{
  "token": "64-char-hex-token-from-email",
  "password": "new-password-min-8-chars"
}

Two-factor authentication (TOTP)

Outpost supports optional per-user TOTP two-factor authentication. When enabled, the login flow adds a second step — the user must enter a 6-digit code from their authenticator app (or a one-time backup code) after their password is verified.

API key auth is unaffected Two-factor authentication only applies to browser session login. API key (Bearer token) authentication bypasses 2FA entirely.

Login with 2FA enabled

When a user with 2FA enabled logs in via auth/login, the response changes:

{
  "success": true,
  "requires_2fa": true,
  "totp_token": "base64-signed-token..."
}

No session is created yet. The totp_token is a signed, time-limited token (5 minutes) that proves the password was verified. You must pass it to the verify endpoint along with the TOTP code.

Verify TOTP code

POST api.php?action=auth/totp/verify

Public endpoint. Validates a TOTP code (or backup code) against the signed token from login. On success, creates the session and returns the same response as a normal login.

Request body (authenticator code):

{
  "code": "123456",
  "totp_token": "base64-signed-token..."
}

Request body (backup code):

{
  "code": "abcd234567",
  "totp_token": "base64-signed-token...",
  "is_backup": true
}

Success response (same as normal login):

{
  "success": true,
  "user": { "id": 1, "username": "admin", "role": "super_admin" },
  "csrf_token": "a1b2c3..."
}

2FA management endpoints

These endpoints require an active admin session and a valid CSRF token.

GET api.php?action=auth/totp/status

Returns the current user's 2FA status and remaining backup code count.

Response:

{
  "enabled": true,
  "backup_codes_remaining": 7
}
POST api.php?action=auth/totp/setup

Generates a new TOTP secret and returns it along with the otpauth:// URI for QR code rendering. The secret is stored but 2FA is not yet enabled — you must call auth/totp/enable with a valid code to activate it.

Response:

{
  "secret": "JBSWY3DPEHPK3PXP...",
  "uri": "otpauth://totp/Outpost%20CMS:admin?secret=JBSWY3DPEHPK3PXP...&issuer=Outpost%20CMS&algorithm=SHA1&digits=6&period=30"
}
POST api.php?action=auth/totp/enable

Verifies a 6-digit code against the stored secret and enables 2FA. Returns 8 plaintext backup codes that the user must save — they are stored as bcrypt hashes and cannot be retrieved again.

Request body:

{ "code": "123456" }

Response:

{
  "success": true,
  "backup_codes": [
    "abcd234567", "efgh891234", "jkmn567890", "pqrs234567",
    "tuvw891234", "xyz2345678", "abcd567890", "efgh234567"
  ]
}
POST api.php?action=auth/totp/disable

Disables 2FA for the current user. Requires password confirmation. Clears the TOTP secret and all backup codes.

Request body:

{ "password": "your-password" }
POST api.php?action=auth/totp/backup-codes

Regenerates backup codes for the current user. Requires password confirmation. Invalidates all existing backup codes and returns 8 new ones.

Request body:

{ "password": "your-password" }

Response:

{
  "success": true,
  "backup_codes": [ "abcd234567", "efgh891234", ... ]
}

Response format

All Admin API responses follow one of two shapes:

Success — read operations

{
  "pages": [ ... ],
  "collection": { ... }
}

Success — write operations

{ "success": true }
// or with a created ID:
{ "success": true, "id": 42 }

Error

{ "error": "Human-readable error message" }

HTTP status codes

Code Meaning
200Success
201Created (resource was created)
204No content (OPTIONS preflight)
400Bad request — missing or invalid parameters
401Unauthorized — not logged in, or invalid credentials
403Forbidden — logged in but insufficient role, or invalid CSRF token
404Not found — resource does not exist, or unknown action
429Too many requests — rate limit exceeded
500Server error

CORS policy

The Admin API only allows CORS requests from localhost origins. This covers local development but prevents cross-origin requests from arbitrary production domains.

The Public Content API (content/ actions) sets Access-Control-Allow-Origin: * — any origin can call it, no credentials required.

Roles and capabilities

Admin API endpoints enforce role-based access. Certain resource groups require specific capabilities:

Content API authentication

The Public Content API requires no authentication. It is completely open — any client from any domain can call api.php?action=content/... endpoints. Only published content is returned.

See Public Content API for the full endpoint reference.

Member API authentication

The Member API uses a separate session system from the Admin API. Members authenticate via member-api.php. The session cookie is named outpost_member and lasts 30 days.

CSRF protection works the same way as the Admin API — get a token from member-api.php?action=me after login, then pass it as X-CSRF-Token on mutations.

See Member API for the full endpoint reference.

Sync API authentication

The Sync API uses a static 64-character hex key stored in the site's settings database. Pass it in every request as a header:

X-Outpost-Key: your-64-char-hex-key

Generate or view the sync key in the Outpost admin under Settings. The Sync API also enforces HTTPS on non-localhost requests and rate-limits failed authentication attempts to 5 per hour.

See Sync API for the full endpoint reference.
