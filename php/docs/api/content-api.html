Public Content API

A read-only, unauthenticated JSON API for headless frontends, static site generators, and JavaScript applications. No API key required. CORS is open to all origins. Only published content is returned.

Overview

All Content API endpoints are accessed through the main api.php file using action values that start with content/. All requests must use the GET method — write operations are not supported.

GET https://example.com/outpost/api.php?action=content/items&collection=blog

Rate limiting: 120 requests per 60 seconds per IP address. Exceeding the limit returns 429 Too Many Requests with a Retry-After header.

Tip Because CORS is fully open (Access-Control-Allow-Origin: *), you can call these endpoints directly from any browser-side JavaScript, mobile app, or native client without a proxy. The Authorization header is also allowed for future authenticated content access.

Quick start

The Content API requires no setup, API keys, or authentication. Any HTTP client can call it immediately.

  1. Base URL — https://yoursite.com/outpost/api.php?action=content/...
  2. Method — GET only (all other methods return 405)
  3. Headers — None required. No Authorization, no API key, no CSRF token.
  4. CORS — Open to all origins. Call from any browser, mobile app, or server.
# Test it right now — no setup needed
curl "https://yoursite.com/outpost/api.php?action=content/pages"

Pages vs. Items — choosing the right endpoint

This is the most common source of confusion. Outpost has two separate content types, and each has its own endpoint. Using the wrong one will return a 404 or an empty result.
Pages (content/pages)Items (content/items)
What they are Standalone pages — Home, About, Contact, etc. Each maps to a .html template file in the theme. Entries in a collection — blog posts, team members, projects, products. Like rows in a database table.
Identified by path — the URL path, e.g. /, /about, /contact slug within a collection — e.g. collection=blog&slug=my-first-post
Use when Fetching a specific page's content (hero text, page-level images, etc.) Listing or fetching entries from a collection (blog posts, products, etc.)
Supports pagination No — returns all pages (typically <20) Yes — limit, offset, orderby, order
Supports filtering No Yes — by folder labels (e.g. &category=tutorials)
Example ?action=content/pages&path=/about ?action=content/items&collection=blog&slug=my-post

Rule of thumb: If the content lives at a fixed URL like /about or /contact, use content/pages. If it's one of many entries in a list (blog posts, team members, products), use content/items.

Response envelope

All Content API responses wrap the payload in a data key. List endpoints also include a meta object with pagination information.

Single resource:

{
  "data": {
    "id": 1,
    "slug": "my-post",
    ...
  }
}

List of resources:

{
  "data": [ ... ],
  "meta": {
    "total": 42,
    "limit": 10,
    "offset": 0
  }
}

Error:

{ "error": "Collection not found" }

Endpoints

Schema

GET api.php?action=content/schema

Returns the complete content schema for the site: all collections with their field definitions, pages with their template-derived fields, global fields referenced across theme templates, and all folders. Page and global fields are extracted by parsing the active theme's templates, so only fields actually used in templates are returned.

Response:

        "name": "Blog Posts",
        "singular_name": "Blog Post",
        "url_pattern": "/blog/{slug}",
        "sort_field": "created_at",
        "sort_direction": "DESC",
        "items_per_page": 10,
        "fields": [
          { "name": "title", "type": "text", "label": "Title", "required": true },
          { "name": "body",  "type": "richtext", "label": "Body", "required": false }
        ],
        "folders": [
          { "slug": "category", "name": "Categories", "type": "flat" }
        ]
      }
    ],
    "pages": [
      {
        "path": "/",
        "title": "Home",
        "fields": [
          { "name": "hero_title", "type": "text" },

Pages

GET api.php?action=content/pages

Returns all public pages with their field values. Members-only and paid pages are excluded. Each page's fields are returned as a flat key-value map.

Response:

{
  "data": [
    {
      "id": 1,
      "path": "/",
      "title": "Home",
      "meta_title": "Welcome to My Site",
      "meta_description": "A personal blog about...",
      "fields": {
        "hero_title": "Hello World",
        "hero_image": "/outpost/uploads/hero.jpg"
      }
    }
  ],
  "meta": { "total": 5 }
}
GET api.php?action=content/pages&path=/about

Returns a single page by its URL path.

Query paramTypeDescription
pathstringThe page path, e.g. / or /about.

curl example:

curl "https://example.com/outpost/api.php?action=content/pages&path=/about"

JavaScript example:

const res = await fetch(
  '/outpost/api.php?action=content/pages&path=/about'
);
const { data: page } = await res.json();
console.log(page.fields.hero_title);

Collections

GET api.php?action=content/collections

Returns all collections with their field count and published item count.

GET api.php?action=content/collections&slug=blog

Returns a single collection by slug, including its full field schema and published item count.

Items

GET api.php?action=content/items&collection=blog

Returns published items from a collection. Supports pagination, sorting, and folder filtering. Only items with status = published and a published_at in the past are returned.

Query paramTypeDefaultDescription
collectionstring—Required. Collection slug.
limitinteger10Number of items to return. Max 100.
offsetinteger0Number of items to skip (for pagination).
orderbystringCollection defaultSort field. Allowed: created_at, updated_at, published_at, slug, sort_order. Also accepts orderBy (camelCase).
orderstringCollection defaultASC or DESC.
{folder-slug}string—Filter by a folder label slug. E.g. &category=news filters to items tagged with the news label in the category folder.

Response:

{
  "data": [
    {
      "id": 10,
      "slug": "my-first-post",
      "title": "My First Post",
      "url": "/blog/my-first-post",
      "status": "published",
      "created_at": "2024-02-01 08:00:00",
      "updated_at": "2024-02-01 08:00:00",
      "published_at": "2024-02-01 08:00:00",
      "fields": {
        "title": "My First Post",
        "excerpt": "A brief summary...",
        "featured_image": "/outpost/uploads/cover.jpg",
        "body": "<p>Full HTML content here...</p>"
      },
      "labels": {
        "category": [
          { "id": 2, "name": "News", "slug": "news", "parent_id": null }
        ]
      }
    }
  ],
  "meta": {
    "total": 24,
    "limit": 10,
    "offset": 0
  }
}

The fields object contains all field values flattened from the item's data. Rich-text (richtext) fields contain rendered HTML. If the item was created with block content, a body key is automatically assembled from all text/HTML blocks.

The labels object groups assigned folder labels by folder slug. Each label object has id, name, slug, and parent_id.

GET api.php?action=content/items&collection=blog&slug=my-first-post

Returns a single published item by its slug.

Query paramTypeDescription
collectionstringRequired. Collection slug.
slugstringRequired. Item slug.

curl example:

curl "https://example.com/outpost/api.php?action=content/items&collection=blog&slug=my-first-post"

JavaScript example — fetch a blog listing with pagination:

const page = 2;
const limit = 10;
const offset = (page - 1) * limit;

const res = await fetch(
  `/outpost/api.php?action=content/items&collection=blog&limit=${limit}&offset=${offset}&orderBy=published_at&order=DESC`
);
const { data: posts, meta } = await res.json();

console.log(`Showing ${posts.length} of ${meta.total} posts`);
posts.forEach(post => {
  console.log(post.fields.title, post.slug);
});

JavaScript example — filter by folder label:

// Get all posts tagged with the "tutorials" category
const res = await fetch(
  '/outpost/api.php?action=content/items&collection=blog&category=tutorials'
);
const { data: posts } = await res.json();

Pagination pattern

The items and media endpoints follow the same offset-based pagination pattern. Use limit and offset together with the meta.total from the response to build page navigation.

// Calculate total pages
const totalPages = Math.ceil(meta.total / meta.limit);

// Next page
const nextOffset = meta.offset + meta.limit;

// Previous page
const prevOffset = Math.max(0, meta.offset - meta.limit);

curl example — fetch page 3 with 5 items per page:

curl "https://example.com/outpost/api.php?action=content/items&collection=blog&limit=5&offset=10"

Folder filtering

To filter items by folder, add the folder slug as a query parameter with a label slug as its value. You can combine multiple folder filters — all must match (AND logic).

# Filter blog posts to the "tutorials" category
GET api.php?action=content/items&collection=blog&category=tutorials

# Filter blog posts to "tutorials" AND tagged with "beginner"
GET api.php?action=content/items&collection=blog&category=tutorials&tag=beginner

If a label slug doesn't exist in the folder, the endpoint returns an empty array rather than an error.

Full example — headless blog

This example fetches a paginated list of blog posts and renders them with vanilla JavaScript.

async function loadPosts(page = 1) {
  const limit = 10;
  const offset = (page - 1) * limit;

  const url = new URL('https://example.com/outpost/api.php');
  url.searchParams.set('action', 'content/items');
  url.searchParams.set('collection', 'blog');
  url.searchParams.set('limit', limit);
  url.searchParams.set('offset', offset);
  url.searchParams.set('orderBy', 'published_at');
  url.searchParams.set('order', 'DESC');

  const res = await fetch(url);
  if (!res.ok) throw new Error('Failed to load posts');

  const { data: posts, meta } = await res.json();

  const container = document.getElementById('posts');
  container.innerHTML = posts.map(post => `
    <article>
      <h2><a href="${post.fields.url || '/blog/' + post.slug}">${post.fields.title}</a></h2>
      <p>${post.fields.excerpt || ''}</p>
      <time>${new Date(post.published_at).toLocaleDateString()}</time>
    </article>
  `).join('');

  // Render pagination
  renderPagination(page, meta.total, limit);
}

loadPosts(1);

CORS and security

The Content API sets the following CORS headers on every response:

Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, OPTIONS
Access-Control-Allow-Headers: Content-Type

Only GET requests are accepted. Any non-GET request returns 405 Method Not Allowed.

The Content API never exposes:

  • Draft or scheduled content
  • Members-only or paid pages (filtered by visibility = 'public')
  • Admin users, passwords, or session data
  • The sync API key or any credentials
  • Private settings

IP-based rate limiting (120 requests per 60 seconds) prevents abuse. Exceeding the limit returns HTTP 429 with a Retry-After header.

Authentication

The Content API requires no authentication. No API key, no session cookie, no Authorization header, and no CSRF token. Every endpoint is fully public and read-only.

This means you can call the API from:

  • Browser-side JavaScript (no proxy needed)
  • Static site generators at build time
  • Mobile apps (iOS, Android, React Native, Flutter)
  • Server-side code (Node.js, Python, PHP, Ruby, etc.)
  • cURL, Postman, or any HTTP client

If you need to create, update, or delete content, use the Admin API instead, which requires session authentication or an API key.

Error handling

All errors return a JSON object with a single error key and an appropriate HTTP status code. Your code should always check res.ok before parsing the response.

HTTP status codes

StatusMeaningWhen it happens
200OKRequest succeeded. Payload is in data.
400Bad RequestMissing required parameter (e.g. collection for items endpoint).
404Not FoundThe requested resource doesn't exist — wrong path, slug, collection, folder, or menu slug.
405Method Not AllowedYou sent a POST, PUT, or DELETE. Only GET is supported.
429Too Many RequestsRate limit exceeded (120 requests/60s per IP). Includes a Retry-After header.

Error response shape

// HTTP 404
{ "error": "Page not found" }

// HTTP 400
{ "error": "Missing required parameter: collection" }

// HTTP 429
{ "error": "Too many requests. Please wait." }
// Header: Retry-After: 45

// HTTP 405
{ "error": "Method not allowed" }

JavaScript error handling pattern

async function fetchFromOutpost(action, params = {}) {
  const url = new URL('https://yoursite.com/outpost/api.php');
  url.searchParams.set('action', action);
  Object.entries(params).forEach(([k, v]) => url.searchParams.set(k, v));

  const res = await fetch(url.toString());

  if (!res.ok) {
    // Parse the error message from the JSON body
    const body = await res.json().catch(() => ({}));
    const message = body.error || `HTTP ${res.status}`;

    if (res.status === 429) {
      const retryAfter = res.headers.get('Retry-After');
      throw new Error(`Rate limited. Retry after ${retryAfter} seconds.`);
    }
    if (res.status === 404) {
      return null; // Resource doesn't exist
    }
    throw new Error(message);
  }

  return res.json();
}

// Usage
const result = await fetchFromOutpost('content/pages', { path: '/about' });
if (!result) {
  console.log('Page not found');
} else {
  console.log(result.data.fields.hero_title);
}

// Fetch items with error handling
try {
  const { data: posts, meta } = await fetchFromOutpost('content/items', {
    collection: 'blog',
    limit: '10',
    orderby: 'published_at',
    order: 'DESC',
  });
  console.log(`${meta.total} total posts`);
} catch (err) {
  console.error('API error:', err.message);
}

Edge cases

ScenarioBehavior
Collection exists but has no published items Returns { "data": [], "meta": { "total": 0, "limit": 10, "offset": 0 } } — not an error.
Collection slug doesn't exist Returns HTTP 404 with { "error": "Collection not found" }.
Page path doesn't exist Returns HTTP 404 with { "error": "Page not found" }.
Item slug doesn't exist in collection Returns HTTP 404 with { "error": "Item not found" }.
Folder filter label doesn't exist Returns an empty array { "data": [], "meta": { "total": 0 } } — not an error.
Members-only or paid pages Excluded from content/pages entirely — they don't appear in lists or single lookups.
Draft or scheduled items Excluded from content/items. Only status = published with published_at in the past are returned.
limit exceeds 100 Silently capped at 100.
offset exceeds total items Returns empty data: [] with the correct meta.total.
Special characters in slug/path URL-encode the value: ?path=%2Fabout. Slugs are always lowercase alphanumeric with hyphens.
Globals page (__global__) Never returned by content/pages. Use content/globals instead.
No menus defined content/menus returns { "data": [], "meta": { "total": 0 } }.
